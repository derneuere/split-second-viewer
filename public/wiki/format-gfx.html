Scaleform UI (.gfx)

.gfx files are Scaleform GFx assets — the Flash-based middleware Black Rock used for the entire 2-D interface: front-end menus, in-race HUD, the QA/debug menus, and the bitmap font library. A .gfx is structurally a SWF (Flash) movie with a GFx-specific signature, authored in Flash and exported through Scaleform's gfxexport tool. The engine drives them through the Scaleform runtime (the EBOOT logs via GFxLog_Error / GFxLog_Warning / GFxLog_Info).

Signature & versioning

All 82 .gfx files in this build begin with the same four bytes: 

47 46 58 08   =  'G' 'F' 'X'  0x08

The first three bytes are the GFx signature GFX (uncompressed). The fourth byte, 0x08, is the SWF version the asset targets — Flash 8. This mirrors the SWF header convention, where an uncompressed movie starts FWS <ver>; GFx swaps the FWS/CWS tag for GFX/CFX. No CFX (zlib-compressed) variants appear in this build — every file is the uncompressed GFX form.

Unlike most data in this PS3 title, .gfx is little-endian: GFx keeps the SWF byte order regardless of host platform, so multi-byte fields (file length, tag codes) are read LSB-first even on the big-endian PS3.

Header layout (SWF-compatible)

OffsetLengthTypeNameDescription
0x00x3char[3]signatureASCII GFX. CFX would mark a zlib-compressed body (not seen here).
0x30x1uint8_tversionSWF target version. 0x08 (Flash 8) for every file.
0x40x4uint32_t (LE)fileLengthTotal file size in bytes. Verified to equal the on-disk size on every sample.
0x8varRECTframeSizeStage bounds as a bit-packed SWF RECT (5-bit nbits prefix + 4 twips fields). Variable length — this is why bytes 0x8+ differ per file. Most assets (42 of 82) use a 1280×720 stage (0..25600, 0..14400 twips, nbits=16, 9 bytes); others range down to 320×540 etc.
after RECT0x2uint16_t (8.8 fixed)frameRatePlayback rate as SWF 8.8 fixed-point, stored frac-byte then int-byte. Verified across all 82 files: 46 @ 30 fps, 32 @ 60 fps, 4 @ 12 fps — the fractional byte is always 0. (For the 1280×720 assets this lands at file offset 0x11, e.g. Bootup 00 3C = 60 fps.)
+0x20x2uint16_t (LE)frameCountNumber of frames on the main timeline (commonly 1–2).
—…SWF tagstagStreamSequence of RECORDHEADER + body tags (DefineShape, DefineSprite, DefineFont, DoAction/DoABC, GFx-specific tags…). Opaque middleware payload.

Hex walk-through

UI/Bootup/Bootup.gfx (4,935 bytes = 0x1347):

000000  47 46 58 08  47 13 00 00  80 00 03 20  00 00 01 c2
        └ 'G''F''X'  └ fileLength └────────── RECT (stage) ────────
          v8           =0x00001347                              …
                       =4935 bytes
                       (matches file size)

fileLength reads 47 13 00 00 little-endian = 0x1347 = 4935, the exact byte count of the file — proving the LE field interpretation. The same check holds on the largest asset, UI/Fonts/gfxfontlib.gfx: 

000000  47 46 58 08  65 73 0a 00  …
                     └ 0x000A7365 LE = 684901 = file size ✓

Per-file tag streams (DefineShape geometry, embedded DjVu/JPEG images, ActionScript bytecode) are authored Flash content and are treated here as an Opaque middleware payload. They can be inspected with standard SWF/GFx tooling (gfxexport, JPEXS); this page documents only the container.

Open questions

  • Decode the GFx-specific export tags (e.g. ExporterInfo, DefineExternalImage, font/glyph cache tags) that Scaleform's gfxexport emits — the first tag after the header is consistently a GFx record, not a stock SWF tag.
  • Identify the remaining GFx-specific tag IDs (gradient/edge AA settings) versus stock SWF tags.
  • Map ActionScript callbacks to the C++ UI controllers (FE_* widget classes).
