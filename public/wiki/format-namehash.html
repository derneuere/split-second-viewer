Archive nameHash — CRC & RAM-Dump Attempt

Every entry in an .ark table of contents is keyed by a 32-bit nameHash of the resource's id string. Cracking it would let the viewer resolve every packed level member to its real filename instead of <hash8>.<ext>. This page records the full investigation: the hash function is fully reverse-engineered, but turning it into an offline name → hash oracle is blocked by a runtime-only data dependency, and the RPCS3 RAM-dump attempt to lift that data went nowhere.

The algorithm (cracked from the EBOOT)

Disassembling EBOOT.BIN (inner ELF at file 0x90; the resource-id hasher at VA 0x19900, inlined at dozens of call sites) recovered the routine exactly — a classic byte-at-a-time reflected table CRC with no final XOR: 

H = seed
for each byte c of the NUL-terminated resource-id string:
    H = (H >> 8) ^ T[(H ^ c) & 0xFF]
return H                       # no final xor

PPC evidence at the hash body: xor r9,H,c / srwi r10,H,8 / rlwinm r9,r9,2,0x16,0x1d (= (H^c)&0xFF ·4) / lwzx r0,T,r9 / xor H,r10,r0.

Resource-id string form

The hashed string is <tag>|<relpath>, where the tag is assembled at runtime via sprintf from a format-string global. Observed tags: CityStatic|<sector>_rig<n>, CityStream|…, Resident|UI/HUD/Textures/<name>, environment|Backdrop/<x>_backdrop.model. Some resources also carry a 1-byte trailing suffix.

Why offline resolution is blocked

The function is solved; the data it reads at runtime is not in the file. Two hard blockers:
  1. The literal 256-entry table T is built at runtime into BSS. The TOC slots that load it (-0x7bbc(r2), -0x62f8(r2), -0x71c8(r2)) point to data-segment addresses that are zero-filled in the file image. None of the 18 static CRC tables in .rodata (VAs 0xf3a360…0xf45bf4) is the right one (verified: no TOC pointer targets them; none reproduce the corpus deltas).
  2. The table is not recoverable from the on-disk corpus. The table is XOR-linear, so it is fully determined by 8 basis vectors T[2^k]. The low-nibble basis B0..B3 was pinned exactly (15/15 clean low-nibble single-char deltas), but the high-nibble basis B4..B7 (T[0x10..0xFF]) is rank-deficient: the 309 ground-truth pairs contain no high-nibble character variation. The one high-bit constraint that does exist (the L↔R substitution, index 0x1E, measured delta 0x3967B15B) contradicts the polynomial-predicted table — proving the runtime table is not the naive CRC table of the poly.

Solving T from the corpus by linear algebra is provably infeasible: the lookup index (H^c)&0xFF depends on the running hash, i.e. on the unknown table — the system is non-linear in the unknowns. Independent verification: 0 / 309 absolute matches across a sweep of 66 (table × seed × string-form) configurations, and 0/pack across all five TEXS packs (no overfitting). Closing it requires the literal T[256] from runtime memory (or a decrypted EBOOT in which the table is materialized).

What was proven (the salvage)

  • The algorithm shape, the within-byte poly 0xDB710641, and both seeds.
  • The low-nibble table basis B0..B3 (so same-length, low-nibble deltas are computable).
  • A Rosetta corpus of 309 ground-truth name → hash pairs, harvested from paired .textures (TEXS) tables + sibling .filenames. The viewer uses this map to give UI/texture members their real names today.
  • The level .ark TOCs hold thousands of unique hashes (≈3.8k–5.5k across the dumps); they are a disjoint set from the UI Rosetta pairs (different archives), so the corpus alone never names them.

The RPCS3 RAM-dump attempt — went nowhere

Because the table is present in memory as soon as the engine's archive system initialises — i.e. even at the title screen, no loaded level required — a RAM dump is the obvious way to lift it. The attempt:

StepPlanOutcome
BootLaunch the game under RPCS3 (CLI --no-gui, or GUI via computer-use) to reach engine init.Reached emulated execution.
CaptureRead rpcs3.exe process memory (it maps the 256 MB PS3 main RAM) or create an RPCS3 savestate (zstd-compressed memory).Got into emulated memory; no usable savestate existed.
Locate TSignature-scan for a 1 KB region that reproduces the 309 pairs, or resolve the TOC pointer slots to the table vaddr.Began the basis (T1) search but was stopped before validating any table.
Net result: no validated table recovered. Offline name resolution stays blocked; the viewer continues to use the 309-pair Rosetta map and falls back to <hash8>.<ext> (with a magic-sniffed type) for everything else. computeNameHash() in the app remains a documented stub carrying the verified algorithm shape, ready to be completed the moment T[256] is in hand.

What would actually close it

  1. A clean runtime dump of T[256]. Resolve the TOC base r2 in a running instance, read the three table-pointer slots, dump the 1 KB table, and validate it reproduces the 309 Rosetta pairs.
  2. Signature scan of a process-memory / savestate snapshot for the table (the first 1 KB region that, plugged into the routine above, reproduces a large majority of the 309 pairs).
  3. Then: hash CityStatic|<sector>_rig<n>, CityStream|… (sector ids from the .sectorInfo build paths) and match them to the dumped level TOC hashes → real filenames for every level member in the viewer.
