Resources & Streaming

Split/Second has to keep a dense, fully-destructible city resident in a 256 MB console while a race tears through it at speed. It does this with a two-tier resource model: a Resident set that is loaded up-front and pinned for the duration of a context, and a sector streaming system that continuously pages level geometry and textures in and out as the camera moves. Both are fed by background loader threads and (in the shipping configuration) packed into .ark archives.

This is the devkit/QA build, so on disk everything is loose under USRDIR/Deferred/ and the .ark archives are present but the game can be told to ignore them. The strings, debug paths and file names below are taken from eboot_strings.txt and from the data tree itself.

Two-tier residency: Resident vs. Streamed

Every load request is tagged with a residency policy, encoded right in the request string with a Policy|Path syntax. The dominant tag is Resident: load it now and keep it. A (noassert) qualifier means "it's fine if this is missing". Real examples from the EBOOT:

Resident|UI/HUD/Hud.gfx
Resident|Scoring/Scoring.params
Resident|Movies/FE_MM_Static.bik
Resident(noassert)|AreaOfEffects/EffectParams.params
Resident(noassert)|Environments/Levels/%s/Cameras/Cameras.xml
Resident(noassert)|Vehicles/Bodies/%s/%s.rData

Other policy prefixes appear for non-pinned content — environment|…, environment(noassert)|…, data|… and root|… — which scope a resource to a load context (the whole environment, the root data folder, etc.) rather than pinning it for the entire session. The format generally is:

TokenMeaning
ResidentLoad up-front, pin for the lifetime of the owning context
environmentScoped to the current level / environment load
data / rootScoped to the data-root / archive context
(noassert)Suffix: do not assert/fail if the file is absent
%sRuntime-substituted (level name, vehicle name, …)

The Resident set is broad: HUD and frontend Scaleform (.gfx), boot movies (.bik), the .params tuning files for scoring / camera / AreaOfEffects, the global shader DB, generic vehicle textures and shadows, and per-level camera/particle config. These never stream — they are the "always there" backbone the streaming system builds on.

Loader threads

Loading happens off the main thread. The QA build's profiler bucket names reveal a main thread / loader thread split for both the in-game and frontend contexts:

Thread / jobRole
Level (Loader Thread)Streams & loads level resources during a race
Front End (Loader Thread)Loads frontend/menu resources
Level (Main Thread) / Front End (Main Thread)Main-thread counterparts (commit/use)
LoaderThread / LoaderJobThe generic loader thread & its work items
Loading ThreadThe boot/install-time loader

Loads are queued as LoaderJobs and serviced by LoaderThread, so the main thread issues a request and later finds the resource ready. PlayStation Store install/copy uses its own threads — GameData_InstallThread and GameData_LoadThread — which copy the shipping .arks and report "Total KBytes copied %d/%d".

.ark archives

For shipping, related resources are concatenated into .ark archives with a hashed table-of-contents at the front. The byte layout is documented on the Archives (.ark) page; here we cover the role they play. The loader looks up logical paths and, if archives are enabled, satisfies them from the matching .ark rather than from a loose file. Top-level archives referenced by the EBOOT:

ArchiveContents
Resident.arkThe Resident set (always-loaded core)
Frontend.arkFrontend / menu resources
InGame1.ark, InGame2.ark, InGame3.arkIn-race resource pools
FrontendPowerplays.ark / %sPowerplays.arkPowerplay assets (per-level)
Environments/Levels/%s.arkPer-level bundle
Characters/RiderAnimations/animationNetwork/rider_%02d.arkPer-rider animation set

Per-level Static and Stream archives

The big payoff is the level sector data, split into two archives per level under Environments/Levels/<Level>/Sectors/: a .Static.ark (low-LOD, always-present geometry that defines the level skeleton) and a much larger .Stream.ark (the high-detail mid/detail-LOD geometry and textures that page in by sector). On disk, all ten levels ship both — 20 .ark files in total. The size split tells the whole story:

Downtown.Static.ark    91,566,080 bytes  (~87 MB, low-LOD core)
Downtown.Stream.ark   563,722,240 bytes (~538 MB, streamed detail)

The Stream archive is roughly 6× the size of the Static one — i.e. the overwhelming majority of a level's art is paged in on demand and never all resident at once. The loader names these resource groups internally as CityStatic / CityStaticOverlay and CityStream / CityStreamOverlay, keyed by subtrack + light-rig (e.g. CityStream|%s_rig%d).

Archives are optional in this build. The data-stream/file system honours a -noark command-line flag: with it set, "Archives disabled - remove -noark to enable them" and the loader reads loose files straight from USRDIR/Deferred/; without it, "Archives enabled - use -noark if you need to disable them" and it pulls from data|Resident.ark etc. This is why the disk has both the loose tree and the archives.

Sector streaming

A level is partitioned into sectors. The .sectorInfo file (see Sectors) describes the partition — for Downtown it is ~58 KB describing the grid — and a per-subtrack SectorVisibility.xml precomputes which sectors are visible from which. At runtime the Stream Manager (a "data stream manager" with a staging buffer and a memory pool) decides, per frame, which sectors should be resident and queues their geometry and textures.

Streaming isn't limited to levels: the frontend vehicle showroom streams car bodies too. Each of the 39 frontend bodies under Vehicles/Frontend/Bodies/<Car>/ ships a <Car>.model.stream + <Car>.streamtex pair (39 of each on disk), e.g. Musclecar_01.model.stream (~1.2 MB) and Musclecar_01.streamtex (~3.1 MB) — the high-res "hero" car detail is paged in only when you're looking at that car.

Size cap on streamed resources

There is a hard upper bound on a single streamable chunk. If a resource is larger than the stage buffer can handle in one shot, the loader refuses to stream it and logs:

Stream texture resource too big to stream: %s
Stream model resource too big to stream: %s

So oversized textures/models have to be authored to fit the streaming budget (or be made Resident instead). Theory the bound is the configured stage-buffer size (see debug menu below).

LOD & residency state

Sectors don't just stream in/out — they have a residency state reported by the QA build's sector HUD. The states track how far the sector's data has progressed:

StateMeaning
ONLY HIGHOnly the high (low-detail/static) LOD is present
Not ForcedNo forced-LOD override active
MappedResource mapped/available
ExpandedFully expanded / detail resident

The HUD shows per-sector Geom Residency, Tex Residency, Priority and Visibility columns, with queue-size buckets 128 (32k), 256 (128k), 512 (512k), 1024 (2048k). LOD selection by camera distance is driven by CityLOD.params per level. Default Downtown/Params/CityLOD.params:

/EnvironmentSS/LODs:
    'a; High LOD Draw Dist'      = 150.0  (0.0, 1000.0);
    'b; Mid LOD Draw Dist'       = 300.0  (0.0, 1000.0);
    'c; Low LOD Draw Dist'       = 1500.0 (0.0, 2000.0);
    'd; MaxNumSectorsRendered'   = 100    (0, 100);
    'e; High LOD Fade Range'     = 30.0   (0.1, 500.0);
    'f; Physics Objects Fade Range' = 20.0 (1.0, 100.0);

So in Downtown the high-detail mesh is drawn within 150 m, mid-LOD out to 300 m, low-LOD to 1500 m, capped at 100 simultaneously-rendered sectors. The generic Generic/Params/CityLOD.params ships with all draw distances at 10000 and the cap at 1000 — i.e. an "everything visible" debug default that per-level files override.

Stream-manager tuning (debug menu)

The streaming budget and look-ahead are tunable live in the QA build under Sectors/Streaming/… (these are debug registers, not shipped .params files):

Debug parameterControls
c; Stage Buffer MBSize of the streaming staging buffer
d; Memory Pool MBSize of the streamed-resource memory pool
e; Max Sector Stream DistHow far ahead sectors are kept resident
f; Geom Priority Offset / g; …Per ViewBias geometry vs. texture priority
h; Max queue size KBCap on outstanding stream requests
i; Look Ahead Dist / j; Look Ahead Damp / k; Look Ahead Reset VelPredictive pre-loading in the direction of travel
a; Disable geometry streaming / b; Disable texture streamingKill switches for debugging
Eject Stream ResourcesForce-flush the streamed set

The look-ahead registers are the key trick: the manager projects a stream position ahead of the camera along the velocity vector (damped, with a reset threshold) so that the geometry for where you're about to be is already paging in. Boot-time switches also exist — game_setup/disable_sector_streaming, low_res_sector_textures and load_Sector_overlays — and the stream manager reports its allocation at startup: "Creating data stream manager: %dk stage buffer, %dk mem pool".

"Stage Buffer MB" and "Memory Pool MB" are exposed as tunables in this QA build; the shipped default sizes live in the binary default.global_regs (see Global Registry) and the debug-menu defaults, not in any plaintext file, so exact MB figures are Partial here.

Resource accounting

The build ships with a full resource-stats system for tracking memory. It buckets every resource by Category / Super Category and can dump a CSV (%srs-%s.csv, written by ResourceStatsCsv) plus an on-screen ResourceStats.gfx overlay. Memory pools are labelled GPU Local, GPU Main and System — the RSX-visible vs. main-RAM split that the residency decisions ultimately have to respect.
