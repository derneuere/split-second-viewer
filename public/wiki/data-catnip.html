Catnip Entity System

Catnip is Black Rock Studio's in-house, data-driven entity / timeline / scripting layer. It is the glue that turns the game's loose data files — entities, powerplays, triggers, timelines and the interactive-object physics meshes — into the runtime behaviour you see on track: a trigger volume fires a powerplay, the powerplay plays a timeline, and the timeline sequences keyframed model animations, particle/light/screen FX, lens flares, area-of-effect explosions, audio events and AI route changes against a single clock. Almost every spectacular set-piece in Split/Second (the exploding jets, the collapsing crane, the falling silo) is a Catnip timeline.

The "Catnip" name only survives in the EBOOT as a set of serialiser namespaces — Catnip/Powerplay/Powerplay/, Catnip/Trigger/Trigger/, Catnip/Powerplay/Timeline/, Catnip/Audio/Timeline/AudioEvent/, Catnip/Physics/InteractiveObjects/, Catnip/LensFlare/, Catnip/Lighting/ and Catnip/PowerplayTimelines/. These strings are the per-class registration tags Catnip uses when it reads each block; there is no monolithic ".catnip" file. The data simply lives next to each level.

What Catnip is

Catnip is not a scripting language in the bytecode sense — there is no VM. It is a declarative event graph: data files describe entities (boxes, spheres, triggers, start positions, pickups), powerplays (named set-pieces with a power level and event list), triggers (spatial volumes that own powerplays), and timelines (an ordered list of timed effects). Everything is keyed by a stable id string of the form <authoring-session>.<ordinal> (e.g. 1223996260.2) and a 32-bit idCRC (e.g. 0x9C5AA2F4). The CRC is what the runtime actually resolves; the human-readable id survives only for tooling and debugging.

The data flow: trigger → powerplay → timeline

A single set-piece chains together three files, all stored under a level's subtrack folder (Environments/Levels/<level>/Subtracks/<A–D>/):

StepFileWhat it provides
1 — Spatial volume<level>.triggersA position+radius volume that owns one powerplay and lists the timelines to fire
2 — Set-piece definition<level>.powerplaysThe powerplay's name, power level, master material, world matrix and EventParams
3 — SequencePowerplays/Timelines/<level>/<name>_T.xmlThe ordered list of timed effects (anim, FX, audio, AOE, route changes)
4 — Geometry / physicsPowerplays/Animations/<level>/<pp>/*.mcl, *.modelInteractive-object collision (.mcl) and renderable assets the timeline shows/hides/animates

Concretely, in docks.triggers a trigger declares its volume and the powerplay it fires:

<Trigger nameCRC="0x86AE958D" id="1224175005.4" idCRC="0x4D76B619" triggerType="Standard"
         enabled="true" autoTrigger="false" enableForOwningPlayer="false"
         enableForOpponents="false" dontSetOwner="false" notActiveOnLastLap="false">
   <posAndScale X="-933.56476" Y="0.0000000" Z="222.84633" W="100.00000"/>
   <Powerplay id="1224174882.2" idCRC="0x9FC2B522">
      <Timeline id="1224174577.4"  idCRC="0xD099061B"/>
      <Timeline id="1256748727.1565529610" idCRC="0x1573DBA"/>
   </Powerplay>
</Trigger>

The matching powerplay in docks.powerplays is the PA01 ship-to-shore crane (idCRC="0x9FC2B522"), and the Timeline name="Powerplays\Timelines\docks\PA01_ship_to_shore_crane_T.xml" inside it points at the actual sequence. The posAndScale.W field is the trigger radius (here 100 m).

The id is a date stamp. The left half of every Catnip id (e.g. 1224174882) is a Unix-style authoring timestamp — these are 2008–2010 dates from when the level designer first placed the object. The right half is a per-session ordinal. The 32-bit idCRC is the only part the runtime hashes.

Powerplays (.powerplays, XML)

The .powerplays file is an XML list (<Root FileVersion="1.0000000">) of <Powerplay> elements. Each powerplay carries the gameplay metadata exposed in the powerplay system debug menu (Select PowerPlay). The attribute names match the EBOOT serialiser tags exactly (NiceName, PowerLevel, Repeatable, AICanTrigger, OpenShortcut, ForwardEffect, BackwardEffect, RouteChanging, TotallyUndriveable, MasterMaterial, contextual).

AttributeExampleMeaning
name..\Powerplays\Powerplays\RCA2_listing_ship_PAuthoring path / asset key
NiceNameRCA2_ListingShipDesigner-facing label (shown in debug menu)
PowerLevel3Power-bar cost: 1 = blue (Level 1), 3 = purple (Level 3 / route changer)
RepeatablefalseCan fire more than once per race
AICanTriggertrueAI cars may spend power on it
RouteChangingtrueOpens/closes an AI route (Level 3 shortcut)
OpenShortcutfalseReveals a new path rather than blocking the old one
TotallyUndriveablefalseMarks the result as a hard wall (instant wreck)
MasterMaterial7Index into cPowerplayMaterialManager for the glow/highlight shader
contextual0Contextual / "near miss" variant flag

The body of each powerplay holds a <worldmtx> (4×3 placement matrix), an <EventParams> list, one or more <Timeline> references each with its own transform, and an <Ambient> NODE_FACTORY block for any always-on instances. The EventParams wire one event to another by id and pass typed parameters:

<EventParams>
   <Event idString="1223996260.2,1223568975.4,1233056780.3" id="0xDB7A586E">
      <Param Name="Route" Type="5" Value="0xFC92D0F0"/>
   </Event>
</EventParams>

The idString is a comma-separated path of ids (powerplay → timeline → effect) describing where the event lives; id is its CRC. Each <Param> has a name, a Type tag and a CRC Value — the EBOOT debug formatters confirm the param model with the strings name %s : type %d : value %d, … value %f and name %s string crc : %x.

Param types

The Type field selects how Value is interpreted. The two seen throughout the level data are route and trigger references:

TypeParam name (observed)Value is
5RouteCRC of an AI route to switch on/off
6Trigger / Model0CRC / id of a trigger or model entity
Partial Only types 5 and 6 appear in the shipped level data; the int/float/string-CRC distinction from the EBOOT formatters implies the full enum is larger (at least int, float and string-CRC variants). The exact numeric assignment of the others is unconfirmed.

Triggers (.triggers, XML)

A .triggers file lists spatial <Trigger> volumes. Each is a sphere defined by posAndScale (X/Y/Z centre, W radius) plus a flag set that controls who and when can fire it. The attribute order is fixed by the serialiser (EBOOT: Trigger, triggerType, enabled, autoTrigger, enableForOwningPlayer, enableForOpponents, notActiveOnLastLap, dontSetOwner).

AttributeMeaning
triggerTypeStandard is the only value in shipped data
enabledActive at race start (route-changer triggers often ship false until enabled by a prior powerplay)
autoTriggerFires on overlap with no power spend (scripted, not player-driven)
enableForOwningPlayer / enableForOpponentsPer-side gating
notActiveOnLastLapDisable on the final lap
dontSetOwnerDon't credit the firing car as the powerplay's owner (for kill attribution)

Note that one trigger volume (idCRC 0x2EE8C43E in docks.triggers, which ships enabled="false") can be armed at runtime by another powerplay's timeline: the PA10_sideways_ship_launch powerplay in docks.powerplays carries an <Param Name="Trigger" Type="6" Value="0x2EE8C43E"/> that turns that disabled trigger on — that is how one powerplay opens up a normally-disabled shortcut trigger. The eboot debug menu exposes this directly under Powerplays/TriggerButtons/%s with sub-commands a; Trigger Index and b; Reset Trigger.

Timelines (_T.xml + .timelineInfo)

A timeline is the heart of Catnip. The authoring form is an XML file Powerplays/Timelines/<level>/<name>_T.xml (there are 317 of them on disk). A <TimeLine> has a total duration, a looping flag and a count, followed by that many <TimeLine_Effect> entries, each with a name, an Effect_Type, an event id, a 4×3 Effect_Transform and an <Effect_Timer> (start, end, stretch):

<TimeLine TimeLine_ID="" TimeLine_Total_Time="16.500000" TimeLine_Looping="false"
          TimeLine_Number_Effects="32">
   <TimeLine_Effect Effect_Name="...\PA04_ExplodingJets_A" Effect_Type="5"
                    Effect_EventIDString="1224256817.10" Effect_EventID="0xEF258EC">
      <ModelOffset X="0.0325935" Y="7.8628039" Z="1.8642635"/>
      <Effect_Transform> ...4×3 matrix... </Effect_Transform>
      <Effect_Timer Timer_Start="0.1643275" Timer_End="11.897661" Effect_Stretch="1.0000000"/>
   </TimeLine_Effect>
   ...
</TimeLine>

The element/attribute names line up exactly with the EBOOT timeline serialiser strings: TimeLine, TimeLine_ID, TimeLine_Total_Time, TimeLine_Number_Effects, TimeLine_Effect, Effect_Name, Effect_Type, Effect_Transform, Effect_EventID, Effect_EventIDString.

The remaining EBOOT labels — Timeline Camera Helper, Timeline PP Hit Box, Timeline Moving Assets, Timeline Danger Zone — likely occupy the unused values (6, 10, 12…), also unverified.

Entry events (Type 7) — the closest thing to scripting

Type-7 effects carry an inline <Event Type="…"> with parameters and are how a timeline mutates game state at a precise time. The full verb list is enumerated in the EBOOT (lines 21746–21765):

Event verbEffect
EnableTrigger / DisableTriggerArm/disarm a trigger volume by CRC
SwitchAIRouteOn / SwitchAIRouteOffOpen/close an AI route (the route-changer mechanic)
PauseTimeline / ResetTimelineControl nested timelines
ShowAnimation / HideAnimationToggle a keyframed model
SetAnimationClipSelect an animation clip
HidePreTimelineModels / ShowPostTimelineModelsSwap "before" geometry for "after" geometry
ShowStandardTimelineModels / HideStandardTimelineModelsToggle the mid-sequence model set
SwitchToDamageModelReplace a model with its damaged variant
SetPowerLevelRetag the powerplay's power-bar cost mid-sequence

The pre/standard/post model swap is exactly what you see in PA04_ExplodingJets_T.xml: two type-7 effects at Timer 0 call HidePreTimelineModels + ShowStandardTimelineModels (each with a <Param Name="Model0" Type="6"> naming the model id), then two more at Timer 16.5 call ShowPostTimelineModels + HideStandardTimelineModels — flipping the intact jets for the wreckage as the timeline ends. A fifth type-7 effect in the same file (Effect_Name="instance|box.cat") fires EnableTrigger at Timer 14.0, showing that entry events are scattered across the timeline, not just at its bounds.

Audio entries (Type 8) — Catnip/Audio/Timeline/AudioEvent

A type-8 effect embeds a complete <Audio> spec that feeds cAudioTimelineManager and the LLAudio2 engine. Each carries an FMOD sound path (sound_name="locale/GENERIC/BIG_EXPL"), inner/outer radii, omni & reverb amounts, a falloff response (Exponential / Linear), and a long list of focus / distance-filter / panning booleans that match the EBOOT audio-attribute strings (force_2d_panning, activate_only_if_listener_inside, force_head_shot, override_distance_filter, force_event_persistance, hyperreal_experience, …):

<Audio label="EXPL" sound_name="locale/GENERIC/BIG_EXPL" mixer_gain="1.0"
       inner_radius="20.000000" outer_radius="300.00000" inner_volume="1.0" outer_volume="0.0"
       falloff_volume_response="Exponential" falloff_reverb_response="Linear"
       is_looping_sound="false" hyperreal_experience="false" .../>

The same Catnip audio layer also owns the standalone audio entities — Catnip/Audio/Trigger, Catnip/Audio/Portal and Catnip/Audio/SwooshBox — authored per level in Audio/Locales/<level>/<level>_Audio_Swooshes.xml. A swoosh box is the whoosh you hear when you brush past overhead scenery:

<AudioSwoosh id="-47434963" label="gantry 2"
   fmod_event_name="Swooshes/Swooshes/OVERHEAD_GANTRY" asset_name="SwooshGeneric"
   inner_radius="0.0" outer_radius="10.000000" volume="1.0"
   trigger_fmod_event="true" track_filter="" transform="..."/>

Area-of-effect (Type 9) and Action Zones (Type 11)

A type-9 effect is the explosion's gameplay payload — it ties into the Area of Effects system. The shipped data exposes a rich tuning surface per AOE: separate radius/time proportions for camera, post-process, handling slip, fire, props, crash and track damage, plus flags such as allow_self_kills, disable_danger_zone and allow_player_kills:

<TimeLine_Effect Effect_Name="AreaOfEffect" Effect_Type="9" area_of_effect_type="Standard"
   camera_radius_proportion="1.0" camera_shake_scale="1.0" camera_shake_time="2.0"
   post_process_effect_time="2.0" handling_impulse_scale="1.0"
   crash_radius_proportion="0.25" track_damage_intensity="1.0"
   allow_self_kills="false" disable_danger_zone="false" allow_player_kills="false">

Type-11 Action Zones are oriented boxes (IsBox="true", ActionZoneIndex, Size) that the camera director and scorer watch — the EBOOT pairs them with ActionZones/Debug and the PowerplaySequenceDirector takedown logic.

Entities (.entities, binary ENTS)

Unlike powerplays/triggers (which ship as XML), the per-subtrack .entities file is a compact big-endian binary blob with magic ENTS. Every one in the build is exactly 808 bytes and holds the eight player start positions — the runtime list whose entity classes are enumerated in the EBOOT: cBoxEntity, cSphereEntity, cGenTrigger/cTriggerVol, cTriggerSphere, cStartPositionEntity, cInstantRespottingTriggerBox/Sphere, cFreestyleTimePickUp/cFreestyleScorePickUp, cPostProcessTrigger, cStartGridEntity and cEndPositionEntity (editor-side: cEditableBox, cEditableSphere):

The EBOOT spells out the three-tier version contract: a mismatched id or major version aborts the load ("Error: entities file id/major version mismatch"), while a minor mismatch only warns and "attempting to load". Full byte-level coverage of the per-record body lives on Entities (.entities).

Interactive objects & moving assets

Catnip/Physics/InteractiveObjects is the physics half of Catnip — the breakable / pushable props a timeline animates or knocks down. Their collision ships as mesh-collision .mcl files (the build contains 1,636 of them) alongside a .model, .fxc, .shaders and a .tex.crcs per object, organised under Powerplays/Animations/<level>/<powerplay>/. The AA/ subfolder holds the "ambient animated" assets (e.g. AA_jumbo_01.mcl, AA_skycrane_rotor_01.mcl, AA_Bell206B.mcl); the Pre/, Post/ and Generic/ folders hold the before/after geometry swapped by the type-7 model events above.

An .mcl opens with a chunked header — SDRI / INSS tags, an object name (unnamed), a CRC string and material parameter names (e.g. Diffuse_IPR_2_fresnel_power):

000000  53 44 52 49 00 00 00 08 49 4e 53 53 00 00 00 06  SDRI....INSS....
000010  cf 52 30 32 00 00 00 08 75 6e 6e 61 6d 65 64 00  .R02....unnamed.
000020  00 00 00 0b 30 78 65 30 62 64 38 36 61 64 00 00  ....0xe0bd86ad..

Keyframed scenery that is not a powerplay (rotating fans, idle cranes) is handled by cMovingAssetsManager from a level-wide moving_assets.xml, with the EBOOT exposing MovingAsssets/Enable animation [sic] and the ../Powerplays/Animations/ asset root.

Lens flares & lighting

Catnip/LensFlare registers the lens-flare entities. The per-level flare meshes live under Environments/Levels/<level>/LensFlare/ as FlareSun, FlareKicker and FlareEnvironment renderables. The flare data serialiser fields are visible in the EBOOT — NumFlares, diffuse_r/g/b, radius — and the loader resolves LensFlares/<name>/<name>.flares and a paired .params via the LensFlareNames.names table.

Catnip/Lighting is the timeline-spawnable dynamic light entity: the EBOOT lighting serialiser lists NumLights, diffuse_r/g/b, specular_r/g/b and falloff_scale — exactly the parameters a type-1 (Light Particle) timeline effect drives when, say, a fire lights up its surroundings.

Opaque The shipped build ships only the flare renderables under each level's LensFlare/ folder; the standalone .flares/.params data files the EBOOT references are not loose on disk in this devkit drop, so the flare data layout is documented from the EBOOT serialiser strings rather than a sample file.

Megabowles — the Catnip test harness

cMegabowlesController is Black Rock's automated powerplay-scenario profiler. It drives the Catnip event graph through scripted runs to catch broken timelines and measure per-set-piece cost — the same role EvolutionConfig plays for the AI. It pairs with the DebugTimelineLog debug page (Display Debug Timeline Log, Clear Log) and the ENABLE_TIMELINE_NIPPLECAT_LIVE_UPDATE live-reload path that lets designers hot-swap a _T.xml while the game runs.

Live update ("Nipplecat")

The EBOOT string ENABLE_TIMELINE_NIPPLECAT_LIVE_UPDATE is enabled reveals Catnip's editor-link feature: in the devkit build, the timeline system can receive updated timeline data from the authoring tool over the wire and reload it without restarting the level. This is why the data ships as readable XML — the same files the tool writes are what the game reads.

Why XML and binary?

Catnip uses two serialisations for two different access patterns:

  • XML for data the designer edits and the live-update path streams — powerplays, triggers, timelines (_T.xml), highlight tags, audio swooshes. Human-readable, diff-able, tool-writable.
  • Binary for data the runtime must load fast and validate hard — .entities (ENTS), the per-subtrack .timelineInfo index and Track.logicinfo (both big-endian id+CRC tables). These carry strict id/major/minor version checks because a bad load would corrupt the race.
