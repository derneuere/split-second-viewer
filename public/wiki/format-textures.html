Textures (.textures / .low.textures)

The .textures file is the Black Rock Crayon2 renderer's texture-set container. It bundles one or more GPU textures into a single TEXS archive: a fixed header, a table of per-texture descriptors (PS3 RSX/GCM pixel format, dimensions, mip count), the swizzled pixel payload, and a trailing C2NM name/CRC table that links each texture to a 32-bit CRC. The companion .low.textures is the same format holding only the smallest mip levels (a tiny always-resident fallback), while full-resolution payloads for streamed assets live in a sibling .streamtex blob.

Overview

Every car body, level sector, skydome, lens-flare and prop ships a .textures set. Three physical layouts of the same TEXS format occur:

FileContentsExample size
X.textures (object dir)Full mip chain, pixels stored inline after the descriptors481 KB (Musclecar_01 body)
X.low.texturesSame descriptors but only the lowest mip(s); always-resident LOD fallback4–5 KB
X.textures (Frontend dir)Descriptor-only stub; full pixels live in X.streamtex2.4 KB stub + 3.1 MB stream
The byte value 0x86 shows up in two unrelated places: as the GCM pixel-format code for DXT1 inside each descriptor, and (coincidentally) inside swizzled pixel payloads. It has nothing to do with the magic — "TEXS" is 54 45 58 53 ('T'=0x54).

File header

The header is a constant 24-byte (0x18) block. magic, version, flags and firstDescOff were verified constant across the 0-, 1-, 5-, 14- and 27-entry samples below; textureCount and payloadTableOff vary per file.

OffsetLengthTypeNameDescription
0x000x4char[4]magicAlways "TEXS" (54 45 58 53). Constant across every sample.
0x040x4uint32_tversionAlways 0x0000000C (12) — header size / format version.
0x080x4uint32_tflagsAlways 0x00000001 across all samples (likely "has C2NM names").
0x0C0x4uint32_tpayloadTableOffOffset of the post-descriptor placement/payload table (the region just before the pixel data and the C2NM trailer). Partial — value verified non-constant and file-dependent (0x68 in the 1-entry damageMap, 0x128 in the 5-entry .low, 0x310 in the 14-entry Frontend stub, 0x548 in the 27-entry _po). It is not the C2NM offset — the name trailer lives at end-of-file and is reached by scanning, not by this field.
0x100x4uint32_ttextureCountNumber of descriptor records. 0 (empty wheel), 1, 5, 14, 27 observed; matches the descriptor stride below.
0x140x4uint32_tfirstDescOffAlways 0x18. Points at a 20-byte sub-header, then the descriptor array at 0x2C.

At 0x18 sits a short 20-byte sub-header (e.g. 00 00 00 07 00 00 02 10 00 00 00 02 00 00 02 48 00 00 00 00 in the Frontend stub) before the descriptor array proper. Its dwords look like a small count plus two offsets that index into the placement table; the exact field meanings are Theory. The descriptor records themselves begin at 0x2C (firstDescOff = 0x18, and 0x2C − 0x18 = 0x14 = the 20-byte sub-header).

Texture descriptor

Each descriptor is a 36-byte (0x24) record. Stride was pinned by walking from one CRC to the next: descriptor 0's CRC sits at 0x2C, descriptor 1's CRC at 0x50 → 0x50−0x2C = 0x24. Field meanings were cross-checked against the embedded C2NM names and against on-disk byte sizes (DXT math, below).

OffsetLengthTypeNameDescription
0x000x4uint32_tcrcTexture name CRC-32. Re-appears in the C2NM trailer paired with the ASCII name, and is the key used by .crcs reference tables.
0x040x2uint16_tmarkerAlways 0xFFFF in every descriptor — start-of-record sentinel.
0x060x2uint16_tpad0Always 0x0000.
0x080x1uint8_tgcmFormatRSX/GCM pixel format (see enum). 0x86=DXT1, 0x88=DXT5, 0xA5=A8R8G8B8.
0x090x1uint8_tmipCountNumber of stored mip levels. For full-chain art it equals log2(max(W,H)) + 1 — verified 512×512→0x0A (10), 1024×2048→0x0C (12), 16×16→0x05 (5). Some assets store a single mip instead: the 512×512 A8R8G8B8 skydome and 1024×32 ColorCubes both carry mipCount = 0x01, so this is the actual stored count, not a derived constant.

Worked example — Frontend Musclecar_01.textures (14 descriptors)

This frontend stub lists the same seven textures twice: a small inline thumbnail set (descriptors 0–6, mostly 16×16) and the full-resolution set (descriptors 7–13) whose pixels live in Musclecar_01.streamtex. The repeated CRCs are the proof (the seven names from the C2NM trailer are paraboloid_back, paraboloid_front, Musclecar_01, Musclecar_01_DamageMap, DirtMap, ScratchMap, MetalMap):

#CRCFormatSizeMipsName (from C2NM)
086640e7eDXT116×165paraboloid_back (thumb)
786640e7eDXT1512×51210paraboloid_back (full)
9b7b2a83bDXT11024×204812Musclecar_01 (full)
112502de16DXT1256×2569DirtMap (full)
Summing the full-resolution DXT1 mip chains for descriptors 7–13 (2×512² + 2×(1024×2048) + 3×256², all DXT1 with mips) gives 3,276,904 bytes; the actual Musclecar_01.streamtex is 3,277,312 bytes — a 408-byte difference attributable to per-texture block alignment. That near-exact match is what confirms both the pixel format and the stub→stream relationship.

Pixel formats (gcmFormat enum)

The high byte is the CELL GCM texture format. Each was confirmed by reconstructing the expected on-disk byte size and comparing it to the real file (single-texture files make this unambiguous).

NameValueNotes
DXT1 (BC1)0x864 bpp block compression. damageMap 1024×2048: DXT1 full mip chain = 1,398,120 B; the file is 1,398,298 B with its C2NM trailer at byte 1,398,264, so pixels+header consume 1,398,264 B (≈144 B header+tables ahead of pixels). Remap 0xAAE4.
DXT5 (BC3)0x888 bpp block compression (alpha). The code is present and in use — many descriptors in Downtown_po.textures carry 88 <mip> 02 00 — and the eboot ships UI assets named …_DXT5. Remap 0xAAE4. Partial — code observed in real descriptors but not isolated to a single-texture file for exact byte-size verification here.
A8R8G8B80xA532-bpp uncompressed. Downtown/Skydome/Skydome_Midday 512×512 (single mip): 512·512·4 = 1,048,576 B vs file 1,048,718 B (C2NM at 1,048,696). ColorCubes holds three 1024×32 textures: 3·1024·32·4 = 393,216 B vs file 393,498 B. Remap 0xAA1B.
Other high-byte values surely exist (cube maps, R5G6B5, the FMT_A8R8G8B8 / R5G6B5 names appear in the eboot's image library). Only the three above have been size-verified here; treat unseen codes as Theory until checked.

Placement table & pixel payload

The dword at header 0x0C (payloadTableOff) points past the descriptor array to a secondary table of dword pairs — an ascending list of placement offsets the loader uses to map each mip block into RSX local memory (in the Frontend stub the second column climbs 0x210, 0x218, 0x220, 0x228 … in step with the sub-header's 0x210 value). The pixel payload follows, then the C2NM trailer at end-of-file. The exact pair semantics are Partial. Compressed pixels are stored in RSX swizzled order (the repeated 55 55 55 55 / ef 19 patterns are the swizzle of flat colour fields), not linear, which is why a raw .streamtex is not a plain DDS.

C2NM name trailer

Every TEXS file ends with a C2NM ("Crayon2 Name") table mapping each CRC back to a human-readable texture name. Records are tightly packed: CRC(4) · length(4) · ASCII name(length), big-endian.

000920  43 32 4e 4d 86 64 0e 7e 00 00 00 0f 70 61 72 61   C2NM.d.~....para
000930  62 6f 6c 6f 69 64 5f 62 61 63 6b 88 da ce c4 00   boloid_back.....
000940  00 00 10 70 61 72 61 62 6f 6c 6f 69 64 5f 66 72   ...paraboloid_fr
000950  6f 6e 74 b7 b2 a8 3b 00 00 00 0c 4d 75 73 63 6c   ont...;....Muscl
000960  65 63 61 72 5f 30 31 …                            ecar_01…

Annotation: 43 32 4e 4d = "C2NM". Then 86 64 0e 7e = CRC, 00 00 00 0f = length 15, "paraboloid_back". Next 88 da ce c4 / len 16 / "paraboloid_front", then b7 b2 a8 3b / len 12 / "Musclecar_01". Those CRCs are exactly the descriptor CRC fields above — the same 32-bit keys collected in .crcs reference tables.

How we know

  • Magic / header constants: identical first 24 bytes across 0/1/5/14/27-entry files (vehicle, frontend, environment).
  • Descriptor stride (0x24): distance between consecutive CRC fields; CRCs round-trip to the C2NM names.
  • mipCount = log2(maxdim)+1: holds for every observed dimension (16,32,256,512,1024,2048).
  • Pixel formats: single-texture files reproduce DXT1 and A8R8G8B8 byte sizes to within the header+name overhead (damageMap 1024×2048 DXT1, Skydome 512×512 ARGB8888); DXT5 (0x88) is seen in real descriptors but not yet isolated for a clean size check.
  • Stub→stream link: Frontend stub's full-res descriptor sizes sum to the matching .streamtex size.
  • GCM constants: 0x0200 dimension, 0xAAE4/0xAA1B remap words are standard CELL GCM values; the eboot references FMT_A8R8G8B8, R5G6B5, RSX shader patching in libs/Render/Support/Crayon2.
