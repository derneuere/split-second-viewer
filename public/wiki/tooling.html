Round-trip-capable formats (byte-exact write-back). The Split/Second Viewer ships parsers that re-emit the original bytes exactly (writeRaw(parseRaw(bytes)) === bytes, contract-tested against the devkit corpus) for 23 formats: .crcs · .splitlength · .linkorigins · .sideways · .checkpoints · .track · .nis · .gbx · .entities · .timelineinfo · .logicinfo · .sectorinfo · .names · .filenames · .parts · .dct · .global_regs · .params · .xml · .powerplays · .triggers · .deform. The graphics/geometry formats below (.model, .textures, .shaders, .fxc, .skel, .mcl, Havok) are read/decode-only so far — they parse and visualize but do not yet round-trip.

High-value custom work remaining (ranked):

  1. .model (+.model.stream) — the big prize; build a Noesis .py plugin (medium/high). Unlocks all geometry; no off-the-shelf tool exists.
  2. .textures / .streamtex — PS3 swizzle → PNG; the probe already IDs format/dims, extend to a full decoder.
  3. .skel, .deform (DFM), .mcl/.shaders/.shaderinst (SDRI/SHDR) — low-effort bespoke parsers; layouts already in the wiki.
  4. .phys / .mainColl — no tool reads Havok 5.5 packfiles; write a custom reader using the .hkPPsXml/.hkCollXml twins as schema, or evaluate leaked Havok 5.5 Content Tools (caveat).
  5. Solved the .ark nameHash — it is a reflected table-driven CRC (seed 0xDEAFD00D, poly 0xDB710641, step H=(H>>8)^T[(H^c)&0xFF], no final XOR; see format-ark → Naming). The remaining work is pure enumeration: build the resource-ID strings from .sectorInfo + the <archiveTag>|<relpath> templates and hash them to restore filenames. (No zlib inflate is needed — members are stored raw; only the 12-byte Stream sub-resource frame is stripped.)
Existing 3rd-party project to evaluate: Delutto's ARK Tool — the only Split/Second-specific tool found in the wild (auto-converts .dct→txt).

.ark Works

Recommendation: Use the bespoke C:\Users\Niaz\splitsecond\_tools\ark_extract.py - the 16-byte header + 16-byte TOC container is fully decoded and verified across all 12 archives. It dumps every member by deriving on-disk length from the next-higher offset. To go further than container-level extraction (i.e. decode payloads), two follow-ups remain: (1) locate the zlib deflate start inside each member and inflate to the TOC 'size'; (2) recover the nameHash function (needs real member-name strings, likely from .sectorInfo) to restore filenames. No need for QuickBMS/vgmstream/noesis for the container layer.

Custom effort: low. Big-endian. Header(16B): u32 version(0), u32 dataStart(=0x10+count*0x10), u32 count, u32 entrySize(0x10). TOC: count x 16B entries sorted ascending by nameHash, each = u32 size(decompressed/in-memory hint, NOT on-disk length), u32 nameHash(stable across levels), u32 reserved(0), u32 offset(absolute, into THIS file). Data region begins at dataStart (first ~8 bytes are a small leading record/pad before the first blob). On-disk stored length is NOT in the TOC - derive it as distance to the next-higher offset (clamp last to EOF). Static members are serialized objects framed by 02 00 00 08 + u32 itemType + length word; Stream members are deflate-packed model/texture resources behind an inner 00000000|innerSize|... table.
