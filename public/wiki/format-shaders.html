Shaders (.shaders / .shaderinst)

A renderable in Split/Second is described by a .model plus a tightly-coupled .fxc (compiled RSX microcode) and .shaders "shader set" — the latter two are 1:1 with each other (2357 each on disk). Vehicle bodies/wheels and a broad set of level geometry additionally carry a .shaderinst. The .shaders file holds the engine-side register block sets — the named constant- and sampler-binding tables that the Crayon2 renderer uses to patch the clean microcode at load time. The .shaderinst binds a concrete mesh instance to a shader set and overrides specific material constants (paint colour, emissive index, etc.).

Role in the render pipeline

The EBOOT spells the design out. libs/Render/Support/Crayon2/Crayon2/Shader/PS3/ShaderPatching.cpp runs a SPURS job whose three DMA inputs are the clean shader, a patch list, and a patch register buffer:

cellSpursJobGetInputList( …, alignedShaderSize, (s32)info.mCleanShader )
cellSpursJobGetInputList( …, alignedPatchDataSize, (s32)&info.mPatchList->GetPatchList()[0] )
cellSpursJobGetInputList( …, alignedPatchRegSize, (s32)&info.mPatchRegBuffer[0] )
— eboot_strings.txt

The "clean shader" is the RSX microcode stored in the .fxc; the patch list / register data are exactly the named register-block sets serialized in .shaders. The load message "Loading register block sets from %s." fires when the .shaders file is read, and the format is owned by cMaterialComboManagers / cMaterialComboFactory (both present as RTTI strings in the EBOOT). Each shader record is identified by an 8-hex-digit material-combo CRC; a missing one trips "Can't find material combo with CRC 0x%08x.. replacing with error combo!".

A .shaders set is a collection of material combos, not a single program. The übershader database global_shader_combos/global_shader_combos.shaders is the same format with 263 combos in it; the per-car files are subsets the build tool emitted for that one body.

.shaders — SHDR container

A flat little-overhead container: a fixed header, then count back-to-back shader records. There is no offset table — records are walked sequentially. All multi-byte integers are big-endian.

Header (0x14 bytes + name)

OffsetLengthTypeNameDescription
0x00x4char[4]magic"SHDR" (53 48 44 52). Constant across all 2357 samples.
0x40x4uint32_tversion6 in every sample checked (vehicle bodies, wheels, global DB). Format/version tag.
0x80x4uint32_tcombo_countNumber of shader/material-combo records that follow. Verified: equals the number of HDRB markers in the file and the count field of the sibling .fxc (Musclecar_01 = 23, _WheelBack = 9, global DB = 263).
0xC0x4uint32_tname_lenLength of the set name including its terminator (0x0B = 11 here).
0x10name_lenchar[]set_nameThe set-level CRC string, e.g. "0xe4249692". Identical to the name in the paired .fxc — this is how the two files are matched.

Shader record

Each record is an HDRB…HDRE header block, followed by two SDRS…SDRE slots (the vertex and fragment microcode slots — empty here, because the actual microcode lives in the .fxc), then the record's own CRC name. The HDRB block carries the named constant and sampler binding table.

OffsetLengthTypeNameDescription
+0x00x4char[4]hdr_begin"HDRB" — start of the register/binding header.
+0x40x8uint32_t[2]reservedTwo zero words in every observed sample (flags/reserved).
+0xC0x4uint32_tcount_aFirst of two consecutive count words. Both equal 2 in the Musclecar/Wheel samples examined; their distinct roles are not byte-proven (see caveat). Theory
+0x100x4uint32_tcombiner_countNumber of named "combiner" entries that follow (e.g. 2). Each combiner entry is {u32 tag; u32 = 1; u32 strlen; char[] name} where tag increments per entry (0x2f, 0x30, …); names such as VehicleGenericNoNormalMap_2_fadeout, ambient_occlusion_simple_3_fadeout.
…varstruct[]const_tableA list of named shader constants/samplers. Each record carries a small leading metadata word group, a {u32 strlen; char[] name} pair, and a register-slot index. Names are the engine uniform symbols — WorldCameraPos, sslighting_ambient, sslighting_sky, sslighting_diffuse, gbuffer_MotionID, ss_deferred_gbuffer_globalFace, etc. The slot index rises monotonically 0,1,2,… The exact field order within each record (kind / count / reg_index relative to the name) is only partially decoded. …0x4char[4]hdr_end"HDRE".
…0xC—vp_slot"SDRS" + u32 len(=0) + "SDRE" (12 bytes) — vertex-shader microcode slot, length 0 (microcode is in the .fxc).
…0xC—fp_slot"SDRS" + u32 len(=0) + "SDRE" (12 bytes) — fragment-shader microcode slot, length 0.

Both per-car and global files show SDRS 00 00 00 00 SDRE (zero-length microcode). That is the strongest evidence that .shaders is metadata only and the compiled programs are entirely delegated to the .fxc twin.

.shaderinst — SDRI / INSS container

The shader instance file binds mesh instances to material combos and supplies instance-specific constant overrides (paint colour, emissive light index, decals). It is not a single binding: after the header it carries a list of N instance records, each naming a node, the combo CRC it binds, and that node's overrides. .shaderinst is carried by vehicle bodies/wheels and by a wide range of level geometry (backdrops, lens-flare, light rigs, physics debris, sectors, skydomes, particle screens) — 710 files in total against 2357 .shaders; static geometry that needs no per-instance override omits it.

Floats and texture CRCs are legible, but the small index tables that follow each override (00 00 00 02 …) map names to RSX register/sampler slots in a way that is not yet fully confirmed — hence the Partial badge.

Material & texture binding scheme

The three render files are joined by CRC strings, not file offsets, which makes the binding recoverable without resolving the node tree:

  • Set-level join key. A .shaders and its .fxc twin carry a byte-identical set_name CRC string (e.g. "0xe4249692") and an identical combo_count. Matching those two fields pairs the register-block table to its compiled microcode.
  • Per-combo CRC. Each HDRB…HDRE record ends in a length-prefixed combo CRC (e.g. "0x618d6dad"); these are the keys behind "Can't find material combo with CRC 0x%08x". A .model mesh references a combo by this CRC, and .shaderinst binds a concrete node to it.
  • Constant/sampler symbols. The HDRB const-table names (WorldCameraPos, sslighting_*, gbuffer_MotionID, …) are the same engine-uniform symbols the .fxc symbol table maps onto hardware registers (C19, C20, …) and samplers (TEXUNIT0…). The two files describe one program from two angles: .fxc = the program + which register each symbol occupies; .shaders = the patch table that supplies each symbol's value.
  • Texture binding. A texture is bound by its 32-bit CRC — the same key the .textures descriptor and its C2NM trailer use. Instance overrides in .shaderinst (e.g. VehiclePaintFast_1_diffuseMap) carry a texture CRC inline; resolving that CRC against the renderable's .textures C2NM table (and the .crcs reference lists) gives the human texture name the sampler reads. Sampler slots themselves surface as TEXUNIT0… in the .fxc microcode.
The exact register-slot index encoding inside each const-table record, and the small 00 00 00 02 … index tables that follow each .shaderinst override, are only partially decoded — the legible part is the symbol/combo/texture CRCs and names, which are enough to reconstruct the binding graph; the per-record slot widths stay Partial.

How the three files fit together

  • .model — geometry + a per-mesh reference to a material combo.
  • .shaders — the material-combo register block sets (constant/sampler symbol → slot tables).
  • .fxc — the compiled RSX vertex + fragment microcode for those combos (the "clean shader").
  • .shaderinst — binds a concrete mesh instance to a combo CRC and overrides constants (paint, emissive, decals).

At load, Crayon2 reads the clean microcode (.fxc), applies the register-block patches (.shaders) plus instance overrides (.shaderinst) on a SPURS job, and uploads the patched program to RSX. The debug tweakvars renderer/filter_dup_shaderinst and renderer/filter_dup_shader deduplicate identical combos at runtime.
