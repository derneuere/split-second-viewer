Logic Info (.logicinfo)

Track.logicinfo is a small, fixed-size (288-byte) big-endian blob that travels with the per-subtrack track-logic data. It is loaded by the same race-logic loader that reads Track.checkpoints / Track.linkorigins and holds the metadata that logic needs before the route is live — most visibly the start-grid transform. Almost the whole record is constant across every track; only a handful of float fields change, and those decode to sensible world-space coordinates.

Where it lives

One Track.logicinfo exists per subtrack under each level's racing event folder:

The path tokens environment(noassert)|Event/%s/TrackLogic/%s and %s/Track.logicinfo appear in eboot_strings.txt directly alongside the other per-subtrack route files (%s/Track.checkpoints, %s/Track.linkorigins, %s/Track.lip) and the TrackLogic_Error / _Warning / _Info log channels. The nearby asserts — More than 1 StartGridRef added…, No StartGRidRef loaded, Found Start Grid @ (.2f, .2f, .2f), cStartGridEntity — identify the start-grid as a primary payload of this loader.

File layout

The size is rock-solid: every one of the 18 files is exactly 288 bytes. A 12-byte header (version + id + length) is followed by a 0x54-byte body (0x0C…0x5F) of interleaved constant CRCs and per-track floats, then a 188-byte zero-fill region and a 4-byte sentinel footer. The length word at 0x8 is 0x00000114 = 276, which is precisely 288 − 12 — i.e. the byte count after the 12-byte header. That is the strongest evidence for the header boundary.

How the constant/variable split was proven

Four files from different levels (Downtown A/B, docks A, Graveyard A, nem_storm A) were diffed byte-for-byte:

  • All 18 files were diffed word-by-word. Exactly five 4-byte words differ between files: 0x34, 0x3C, 0x44, 0x4C and 0x54. Every other word (0x00–0x33, the interleaved CRCs at 0x38/0x40/0x48/0x50/0x58, 0x5C, and 0x60–0x11B) is byte-identical in every file ⇒ header, field-name CRCs, padding and footer.
  • The five differing words decode (big-endian) to clean world-scale floats; the constant 4-byte fields immediately before each one decode to absurd floats (e.g. C9F4444C ≈ −2 000 033) — the classic signature of a CRC/hash word, not a coordinate. So the body is a tidy (field-name-hash, value) interleave of five named float fields, with a trailing hash at 0x58.
  • Note the duplicate: val0 (0x34) and val3 (0x4C) are byte-for-byte equal in all 18 files, so only four distinct float values are actually stored per track.
The 3-major / 0-minor versioning and the id mismatch / major version mismatch / minor version mismatch — attempting to load assert family that the entities loader uses suggest .logicinfo is produced by the same Catnip serializer that emits .entities. Treat the field names here as inferred roles, not ground-truth labels — hence the Partial badge.
Quick check for a fresh dump: read 0x08 as a u32; it must equal file_size − 12. If the file is not 288 bytes or the footer is not BA DC AD DE, the track has additional populated logic entities and the body layout below 0x60 will differ.

Open questions

  • Exact semantics of the five per-track floats (only four distinct values, since val0=val3). A start-grid origin (x,y,z) plus a heading/extent is the leading theory; the duplicated value at 0x4C may simply be the exporter re-emitting Y, or a paired grid endpoint. The mapping of val0/val1/val2 to X/Y/Z is inferred, not proven.
  • The constant float 471770.3 at 0x10 is shared by every track; it may be a default bounding/limit value or an un-overridden tuning default.
  • Whether the 188-byte zero region (0x60…0x11B) is fixed reserved space or a variable-length list that simply happens to be empty in all shipped tracks (all 18 are 288 bytes, so empty-by-default at minimum).
  • The six constant 4-byte words interleaved with the floats (0x30/0x38/0x40/0x48/0x50/0x58) are assumed to be field-name CRCs/hashes; the hash algorithm and the field names they encode are unverified.
