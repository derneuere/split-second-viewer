Lights (.lights)

A .lights file is the list of placed local lights (point and spot lights) for one LightRig — a named time-of-day lighting setup for a level. It is a flat, fixed-stride array of light instances: colour, world transform, range and cone, fed straight into the engine's deferred SPULight pipeline. The rig's global environment (sun, ambient, sky, scattering, reflection map) lives in the sibling .params text file, not here.

Where they live

Each level has a LightRigs/ folder with one sub-folder per time-of-day variant, and each variant ships a complete rig set:

Environments/Levels/docks/LightRigs/
  midday/     midday.lights  midday.params  midday.fxc  midday.shaders  midday.textures  midday.gbx
  sunrise/    sunrise.*
  sunset/     sunset.*
  twilight/   twilight.*
  nighttime/  nighttime.*

The runtime builds these paths from format strings in the EBOOT: %sLightRigs/%s/%s.lights, with siblings %sLightRigs/%s/%s.params, %sLightRigs/LightRigNames.names and a defaults file %sParams/LightRigDefaults.params. Rig selection per subtrack is listed in Environments\SubTrackLightRigs.txt ("Env '%s' : SubTrack '%s' : LightRig '%s'").

The four-byte .gbx beside each rig is a stub (the .gbx sequence/graph format is used elsewhere; here it is empty). The 8 KB .params twin is the authored environment config — see the breakdown below — while the binary .lights file holds the discrete light instances placed in the world.

File format

An 8-byte header followed by a tight array of 176-byte light records. No string table, no offset table — record i lives at 8 + i*0xB0.

Header

OffsetLengthTypeNameDescription
0x00x4uint32_tversionAlways 0x00000002 across all non-stub samples.
0x40x4uint32_tlightCountNumber of light records. Verified by arithmetic: docks sunset.lights is 2120 bytes with count 0x0C = 12, and 8 + 12×176 = 2120 exactly.

Light record (0xB0 = 176 bytes)

Offsets are relative to the start of the record. Field meaning is read off the values: two RGB colour triplets, a position, a 2-row orientation basis, then range/cone scalars and a type flag, an FF FF FF FF sentinel at +0x94, and trailing zero padding to the 176-byte stride. The .params slider ranges (e.g. diffuse 0–500, colours 0–1) corroborate the colour-vs-position split.

OffsetLengthTypeNameDescription
+0x000xCfloat32_t[3]diffuseColorRGB. e.g. docks sunset light 0 = (0.098, 0.0585, 0.0155) — a dim warm tint. HDR-range (the .params caps diffuse at 500).
+0x0C0x4float32_tpad0Zero (colour alpha / 16-byte align).
+0x100xCfloat32_t[3]specColorSecond colour triplet, often (1,1,1). Specular tint / secondary colour.
+0x1C0x4float32_tpad1Zero.
+0x200xCfloat32_t[3]positionWorld XYZ. e.g. (204.7, 1.44, -411.1) for docks sunset light 0 — plausible track coordinates.
+0x2C0x4float32_tpad2Zero.
+0x300x20float32_t[2][4]orientationDirection basis, two rows of 4 floats (xyz + 0). In both samples row0 = (0,-1,0,0) and row1 = (1,0,0,0) — axis-aligned. The third basis row is implied (cross product) or unused; only 0x20 bytes are present before range.
+0x500x4float32_trangeFalloff radius. docks sunset light 0 = 100.0 (42 C8 00 00); nem_downtown light 0 = 30.0.
+0x540x4float32_tparam130.0 (41 F0 00 00) in samples — inner cone / intensity.
+0x580x4float32_tparam220.0 (41 A0 00 00) — outer cone / second falloff.
+0x5C0x4uint32_tpad3Zero.
+0x600x4uint32_tlightType / flags0x00000001 in samples — point vs spot selector (the engine distinguishes PointLight/SpotLight, each with a _Low_LOD variant).
+0x640x30float32_t[…]extraMostly zero with two 1.0s (3F 80 00 00 at +0x74 and +0x88, 0x14 apart) — looks like the diagonal of a small gobo / shadow transform (the deferred path uses deferred_lighting_gobo_view_proj).
+0x940x4uint32_tsentinel0xFFFFFFFF. Confirmed at this relative offset in every record (file offsets 0x9C, 0x14C, 0x1FC, … step 0xB0).

The docks sunset.lights (12 lights) shows the array nature plainly: records 1–3 share identical colour/orientation and differ only in position (43 CC FF B4 vs 43 D2 AF 25 vs 43 D2 1D 15 in the X slot) — a row of street/dock lights placed along the track.

Light types & LOD

The renderer recognises four local-light shader permutations, all present as strings in the EBOOT:

NameValueNotes
PointLighttype 1 (observed)Omnidirectional; uses range.
PointLight_Low_LOD—Cheaper distant variant.
SpotLight—Cone light; uses orientation + cone params; can sample a gobo (deferred_lighting_gobo_view_proj).
SpotLight_Low_LOD—Cheaper distant variant.
VehicleLight / SunLight—Vehicle headlights and the rig's directional sun are handled separately, not via .lights.

Local lights are batched and LOD-sorted for the SPU lighting jobs. The debug log "Light LODs viewport %d, num lights %d, combined %d, full %d, qtr %d, spu %d" shows lights being split into full-res, quarter-res and SPU-processed buckets, and the shader DB has CombinedLight_%dSpot_%dPoint permutations and 16 JobPointLight1..16 SPU jobs. See Rendering Pipeline.

The .params environment twin

The text %s.params beside each rig is not a per-light list — it is the global environment for the rig, parsed under the /Lighting tree (a standard .params file with slider ranges). It defines the directional sun, ambience, sky, scattering and reflection map; the discrete placed lights stay in the binary .lights. Major sections:

SectionContents
/Lighting/ambienceambient/sky/ground RGB hemispheric terms, occlusion factors. A separate ambience_vehicle block tints cars.
/Lighting/sun + /sun/diffuseSun specular RGB, elevation, rotation, spec offsets, and 6 indexed diffuse colours (0_r…5_b).
/Lighting/sun shadowCascade shadow ranges, split weight, filter scale, and per-platform (xenon/ps3) slope/bias for single-player and split-screen.
/Lighting/LS, /ATV_LSLight-scattering coefficients (Mie/Rayleigh, inscattering rate/intensity/focus, extinction tone) — drives lightscattering_* shaders.
/Lighting/ReflectionMapCube reflection contributions and external file names (Env, Skydome, SunLight, PointLight, SpotLight).
/Lighting/ParticleOverridesPer-rig tints handed to the particle system: SunColor, AmbientColor, Unlit/Emissive/Light/Corona tint, EnvShadow strength.
/Lighting/Sky, /Params, /Emissive, /Radiosity, /Kicker, /UnlitSkydome file/rotation/height, post-process file binding, emissive scales, radiosity tweak colour, a fill "kicker" light, unlit sigmoid offset.

Stub rigs

Several Nemesis levels ship an 8-byte stub (00 00 00 02 00 00 00 00, version 2, zero lights) — those rigs rely entirely on the sun/ambience in .params with no placed local lights. The smallest real file is nem_downtown midday_v01.lights at 184 bytes (1 light); the largest is Powerplant twilight.lights at 73,400 bytes (417 lights), with docks twilight.lights close behind at 70,056 bytes (exactly 398 lights). Across all 19 files the identity 8 + lightCount×176 = fileSize holds exactly.

Inside the record, the colour / position / orientation / range / type field positions are Confirmed by value inspection across two files (nem_downtown, docks sunset) and by the constant 0xFFFFFFFF sentinel at +0x94. The orientation is only 0x20 bytes (two basis rows, not three — an earlier draft over-stated it as a full 3×4 matrix). The names/semantics — the split of the +0x54/+0x58 scalars into "inner/outer cone" vs "intensity", and the meaning of the +0x64 extra block — are Theory; there is no XML twin to cross-check them.
