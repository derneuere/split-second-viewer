Model (.model)

The .model file is Split/Second's renderable geometry container, produced by the Black Rock Crayon2 renderer's cRenderableFactory. Every drawable thing — car bodies, level backdrops, skydomes, reflection-map props, lens-flare billboards — is one .model. It is not a Havok packfile; it is a Crayon2 serialized typed-node tree whose leaf payloads are Havok hkxMesh vertex and index buffers cooked for the PS3 RSX.

Identification

The first four bytes are 02 00 00 08 and are constant across every sample inspected — vehicle bodies, level backdrops, and tiny reflection-map props alike. There is no ASCII signature and no Havok packfile magic (57 E0 E0 57), so the file is parsed structurally rather than by tag. Treat 02 00 00 08 as a format/version word: the high 02 is consistent with a Crayon2 "version 2" container and the low 08 appears to be a flags / alignment field.

The engine never opens .model by reading a string magic — the EBOOT loads geometry through cRenderableFactory / cRenderableGeom (libs/Render/Support/Crayon2). The on-disk class set is Havok's graphics export schema: the EBOOT string table lists hkxMesh, hkxMeshSection, hkxVertexBuffer, hkxVertexFormat, hkxIndexBuffer, hkxMaterial and the concrete vertex layouts hkxVertexP4N4C1T2 … hkxVertexP4N4T4B4W4I4Q4. Crayon2 cooks those into the layout described below.

Top-level layout

After the 4-byte magic the file is a header section (a small typed-node tree of (offset, type) pairs), followed by per-section float metadata (transforms and bounding boxes), then two explicit section tables — a vertex-buffer table and an index/draw-call table — then the bulk vertex buffers and the 16-bit index buffers they point at. The header grows with mesh complexity: the PointLight.model probe (1 054 bytes, one section) has a ~0x90-byte header; Musclecar_01.model (1.30 MB) and Downtown_backdrop.model (1.48 MB) carry long header trees describing dozens of sections. The two tables are what the in-app decoder reads to recover renderable submeshes (Section tables), superseding the earlier strip-scanning heuristic.

The node-descriptor tree

The header is built from repeating 8-byte records: a small relative offset followed by a small type tag. The tags observed are a tight set — 0x04, 0x05, 0x06, 0x0E, 0x0F — and the offsets are step values (0x08, 0x10, 0x18, 0x20, 0x28, 0x2C) that walk a node/child layout. This is the classic Crayon2 generic-serializer shape: a tree of typed properties whose leaves point at the float and buffer payloads. The exact tag→class mapping (which tag is mesh vs material ref vs vertex buffer) is not yet fully resolved, hence the Partial badge. Theory

Section tables: vertex buffers & draw calls

The boundary between the vertex region and the index region is not guessed by scanning for strip data — the node tree carries two explicit, regular section tables that the in-app decoder reads directly. (The older "scan for the first FFFF" approach is kept only as a fallback for stub .model files whose geometry has moved to a .model.stream twin.)

Vertex-buffer (VB) table — 0x24 stride

A run of 36-byte (0x24) records, one per vertex buffer:

OffsetLengthTypeNameDescription
+0x000x4uint32_tsizeByte length of this vertex buffer, padded up to a 16-byte multiple — i.e. size ≈ stride · vcount rounded up. This relationship is the table's recognition signature (0 ≤ size − stride·vcount < 64).
+0x040x4uint32_tstridePer-vertex byte stride (12–64 in samples).
+0x080x4uint32_tvcountVertex count for this buffer.
+0x0C0x18—(trailing)Remaining 24 bytes of the record (format/flags + the buffer's own offset hint, not yet fully mapped).

The bulk vertex data follows the VB table; buffers are stored back-to-back in table order, so buffer k begins where buffer k−1 ends (the decoder absorbs a small alignment gap after the table before buffer 0). The actual data start is found by probing the first record's decode just past the table.

Draw-call / index table — 32-byte stride

A run of 32-byte records, one per draw call:

+0x00  relOffset : u32   ; index-region-relative byte offset of this strip
+0x04  00 00 00 00       ; three zero words
+0x08  00 00 00 00
+0x0C  00 00 00 00
+0x10  idxCount  : u32   ; number of 16-bit indices in the strip
+0x14  idxCount·2: u32   ; byte length of the strip (== idxCount × 2) — the self-check
+0x18  00 00 00 00
+0x1C  …                 ; (32-byte stride)

The idxCount·2 == byteLength invariant is what pins this table down. The relOffset values are relative to the start of the index region; the absolute file position is recovered by solving a single correction constant — the last draw call must end at (or just before, modulo ≤16 bytes of pad) EOF, which fixes correction = fileSize − pad − (lastRelOffset + lastIdxCount·2); the first draw call is then validated to land on a strip-like run of small ascending u16s.

Draw-call → vertex-buffer binding

The explicit per-draw-call → vertex-buffer link in the node tree is not yet mapped, so a decoded strip is routed to the smallest vertex buffer whose vcount exceeds the strip's maximum index. That yields correct, renderable per-buffer submeshes for the samples checked, but a draw call that legitimately indexes a larger buffer than necessary could be mis-routed — hence the Partial on the binding (see Open questions).

Vertex-format auto-detect: half-float vs float32

Two storage formats occur, and the decoder distinguishes them by sampling a buffer rather than reading a format flag:

  • half — four big-endian IEEE-754 half-floats per vertex (P4: x, y, z, w ≈ 1.0), the cooked layout for car/level meshes. Detected by scanning the stride for a 4-half group whose 4th component reads ≈ 1.0 across ≥85 % of sampled vertices (the hkxVertexP4… position-with-w convention).
  • float — full float32 P3 + UV: position is the first float3 of the stride, a float2 UV follows at byte 12 when the stride is ≥ 20. Used by simple props (e.g. PointLight.model). Detected when every sampled position is finite and within a sane range.
The auto-detect matters because reading a float32 position buffer as half-floats (or vice versa) yields tens-of-thousands-of-units garbage. For the float path the decoder additionally scores each candidate data-start by how tightly the decoded AABB sits inside the header AABB, which disambiguates the 4-byte alignment slip where a stray table u32 happens to read as a finite denormal.

Section metadata: transforms and bounding boxes

In the single-section PointLight.model the header is short enough to read the float payload directly. Interpreting the region from ~0x90 as big-endian floats yields a transform plus a pair of AABB corners that repeat once per mesh section:

000090   0          0          0          0.70710677   ← quat .w = √2/2
0000a0  -0.5       -0.49999994 -0.000241898 1          ← AABB min  (x,y,z, 1)
0000b0   0.5        0.49999994  0.000241898 1          ← AABB max  (x,y,z, 1)
…
000110  -0.5       -0.49999994 -0.000241898 1          ← AABB min  (repeat / world)
000120   0.5        0.49999994  0.000241898 1          ← AABB max
000130   0          0          0          0.70710677   ← quat .w

The 0.70710677 (= √2/2) is a unit-quaternion component; the symmetric ±0.5 corner pairs are an axis-aligned bounding box used by sector culling. The EBOOT exposes Sectors/Culling/Cull Renderables and Cull Shadow Caster Renderables debug toggles that consume these bounds.

Vertex buffers

The bulk of a large .model is interleaved vertex data laid out for the RSX. Vertex layout is selected per section from Havok's hkxVertexFormat family; the concrete formats present in the EBOOT name table are:

Vertex formatDecodedUse
hkxVertexP4N4C1T2Pos4 · Normal4 · Colour1 · UV2Static / unskinned mesh, vertex colour
hkxVertexP4N4T4B4C1T2Pos4 · N4 · Tangent4 · Bitangent4 · Col1 · UV2Normal-mapped static mesh (TBN basis)
hkxVertexP4N4W4I4C1Q2Pos4 · N4 · Weights4 · BoneIdx4 · Col1 · UV2Skinned (no tangents)
hkxVertexP4N4T4B4W4I4C1Q2+ Tangent/Bitangent + Weights/IndicesSkinned + normal-mapped
hkxVertexP4N4T4B4W4I4Q4Full TBN + 4 weights/indicesHeaviest skinned format

In Havok's naming, P=position, N=normal, T=tangent, B=bitangent (binormal), W=blend weights, I=bone indices, C=colour, Q/T=texcoord set; the digit is the component count. The presence of the W4I4 (skinned) variants is consistent with the .skel skeletons used by animated powerplay props.

Per-vertex UV (texcoord) decode Partial

Because the per-buffer hkxVertexFormat declaration is not yet decoded (the node-tree tag→class mapping above is unresolved), the viewer recovers the texcoord statistically from a float32 vertex buffer. It samples every non-position column and keeps the two NON-CONSTANT columns whose values cluster in the [0,1] tiling band (a texcoord sits near mean 0.5; a unit normal is symmetric about 0, ≈ 50 % negative; a handedness lane is constant). The pair is emitted as [U, V] = [higher byte offset, lower byte offset] — the order verified against rendered output (the Helicopter_Bell206B_01 body livery, skycrane_military camo, and the float props all map correctly; the naive lower = U order rendered them rotated). The helicopter body in particular SPLITS its UV across the leading lane (U @0) and the trailing lane (V @28) around the unit normal at bytes 16–27, so the older "UV == posOffset+12" rule read the NORMAL as the UV.

Open — multi-UV-set disambiguation. Meshes that bake MORE than one texcoord set (a diffuse UV0 plus a lightmap/secondary set) expose three or more candidate columns, and the statistical pick can grab the wrong one. Confirmed on Generic/Models/NemTruckBarrels_*: the barrel's chevron band smears because the true "around-the-cylinder" diffuse coordinate (@40, the column that varies with circumferential angle) loses to the uniform lightmap-style pair (@28/@32). The pure parser cannot see the texture, so it has no way to score which set aligns with the diffuse atlas (band→cylinder side, caps→circles). Two resolution paths: (a) decode the hkxVertexFormat declaration so UV0 is identified explicitly (the robust fix — blocked on the tag→class mapping); or (b) texture-aware UV selection at the material/render layer (choose the set whose side/cap regions best match the diffuse's band/cap layout).

Verification harness: split-second-viewer/scripts/render-model.ts — a headless software rasteriser that renders a .model with its decoded diffuse to a multi-angle PNG contact sheet (modes texture / uv / checker, and a --uvcols U,V override to A/B raw texcoord columns without touching the decoder).

Deferred. Nailing the diffuse-vs-lightmap pick is a visual-iteration task — it hinges on judging rendered output, which the current pass could not resolve reliably enough. Parked until a future pass with stronger render-reading (vision) capability (i.e. revisit on Fable 5).

Index buffers

Every .model ends in one or more 16-bit triangle-strip index buffers. The sentinel 0xFFFF is the RSX primitive-restart marker that breaks one strip and starts the next. This is unambiguous in the tails of all three probes:

            Musclecar_01.model — tail (16-bit strip indices)
13e1a0  3f 55 3f 5b 3f 44 3f 5c  3f 44 3f 5d 3f 44 3f 5e
13e1b0  3f 45 ff ff 3f 5f 3f 60  3f 61 3f 62 3f 61 3f 63   ← ffff = restart
13e1c0  3f 61 3f 64 3f 65 3f 66  3f 65 3f 67 3f 65 3f 68

Indices ascend monotonically (0x3F55, 0x3F5B, 0x3F44…) and are punctuated by FFFF. The Downtown backdrop runs strips up to index 0xEDB2 (≈60 850 verts), confirming a single large mesh split into many strips. Strip topology (rather than triangle lists) keeps the index buffer small for RSX bandwidth.

A quick "is this geometry present?" check: hexdump the last few KB and look for a dense run of ascending 16-bit values broken by FF FF. That is always the index buffer.

The .model.stream twin

The EBOOT format strings %s(noassert)|%s/%s/%s.model.stream and %sMD.model.stream show that high-detail geometry can live in a sibling .model.stream file loaded on demand by the streaming system. This mirrors the .low.textures / .textures split used for textures. See Streaming (.stream/.streamtex) for the streaming envelope.

Streamed samples are present on disk — 46 .model.stream files ship under Vehicles/Frontend/Bodies/. In that layout the resident .model is a tiny stub (e.g. Frontend/Bodies/Musclecar_01/Musclecar_01.model is only 6 312 bytes but still carries the 02 00 00 08 magic and a full node tree) and the bulk vertex/index payload lives in the paired Musclecar_01.model.stream (1.22 MB). Note this is the Frontend (car-select) copy; the in-race Vehicles/Bodies/Musclecar_01/ folder instead ships the whole 1.30 MB .model with no stream.

The .model.stream file does not repeat the 02 00 00 08 magic. It begins with a 12-byte header — dword0 = 0, dword1 = (file size − 12) (i.e. the byte length of the payload that follows), dword2 = 0 — then from offset 0x0C the raw interleaved vertex data, ending in the same 0xFFFF-restart 16-bit triangle strips as a base .model. The filesize − 12 relationship was confirmed across all five Musclecar streams checked. Partial

Skinned / animated variant (magic 02 01 00 08)

The animated Powerplay props — helicopters, ferries, cranes, the Bell 206B and friends — use a skinned Crayon2 variant whose magic is 02 01 00 08 (the 01 flag byte distinguishes it from the static 02 00 00 08). There are 1 636 of these, mostly under Powerplays/Animations. The container differs from the standard variant in three ways:

  • Paired sub-record VB table (0x48 stride). Instead of 0x24 single records, the skinned table is a run of 72-byte (0x48) records, each holding two 0x24 sub-records that share a vertex count: sub-record A is a stride-12 position stream (3 big-endian float32 x,y,z — not half-floats), and sub-record B is a stride-8 aux stream (UV float2 / packed normal). Confirmed on AA_Bell206B.model: records at 0x560 (vc = 16) and 0x5A8 (vc = 688), exactly 0x48 apart.
  • Position buffers are located by scanning, not by the record's offset field. The embedded offset fields in the skinned records are not reliable file pointers, so each position buffer is found by scanning forward from the table end (16-byte-aligned first, then 4-byte) for the first contiguous float32-P3 block of the right vertex count whose AABB is finite and bounded, advancing the cursor past each block so buffers are taken in node order. Validated 100 % across a spread of cars, ferries and cranes (strides 12 and 16).
  • No 16-bit strips, no draw-call table — topology is in the Havok skinning section. Every skinned sample checked has zero 0xFFFF restarts and no dense u16<vcount run after the vertex data: the triangle topology lives in the compressed Havok skinning section, which is not yet cracked. The decoder therefore emits positions only (point meshes, empty indices) and flags the result partial. The combined decoded AABB is cross-checked against the header float AABB for sanity (extents match; the axes are permuted by the engine's coordinate convention).

Relationship to other vehicle files

A car body folder pairs the .model (visual mesh) with a parallel set of files: .phys (Havok rigid body), .mainColl (collision), .deform (damage mesh), .shaders/.shaderinst (materials), .fxc (compiled effects) and .textures. The .model references materials and vertex formats but contains only geometry — shading lives in the shader files.

Open questions. The vertex/index section boundary is now resolved — it is read from the explicit VB and draw-call tables, not by strip-scanning (which survives only as the stub fallback). What remains: the node-tree type tags (0x04/05/06/0E/0F) are still not mapped to specific Havok classes (mesh vs material-ref vs vertex-buffer); the per-draw-call → vertex-buffer binding is heuristic (smallest buffer that contains the strip's indices) rather than read from the tree; the skinned variant's triangle topology lives in an un-cracked compressed Havok skinning section, so those meshes are positions-only; and the 12-byte .model.stream header is decoded only as far as the filesize − 12 payload-length field (how a stub .model points the loader at its stream — filename convention vs embedded offset — is not yet pinned down).
