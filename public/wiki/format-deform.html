Deformation (.deform)

The .deform file is Split/Second's vehicle damage / deformation model: a per-car cage of deformable vertices linked by mass-spring "edges" and partitioned into named rigid part-groups. When a car crashes, the physics solver pushes these vertices and the result is streamed to the render mesh through the DeformVertexPosition vertex-shader attribute. Unlike its siblings in this section, it is not a Havok packfile — it is a small, custom Black Rock binary with its own magic, and it has no XML twin.

Identity & loading

Every file begins with the four bytes 44 46 4D 02 — ASCII "DFM" followed by a version byte 0x02 (so the 32-bit magic is 0x44464D02). The EBOOT loads it via the template %s/%s/%s.deform and validates the signature with a hard check:

Can't find deformation file %s
Bad magic in deformation file %s (expected %08x, got %08x)

The same code region references Deformation, MaxVertDeformations, PartGroups, Sections and the four DeformerPlane* tuning values — confirming the data is a vertex-deformation cage organised into part-groups and sections, and tuned per car alongside SurfaceDamage.params.

This format is custom and undocumented; field names below are inferred from the EBOOT strings and the data, not from a header. Confidence is honest: the magic, the vertex array and the named part-groups are Confirmed; several header counts and the spring/edge semantics are Theory.

File header

A 32-byte header: the magic followed by seven big-endian 32-bit values. Comparing many samples (small chassis files ≈6 KB up to Musclecar_01 at 31 KB) lets the constant from the per-file fields be separated. Field A is provably the vertex count (it predicts where the vertex array ends in every file); the rest are sized lists whose role is read from how they index each other.

Section 1 — deformable vertex cage

Immediately after the header (0x20) is an array of A vertices, each a homogeneous vec4 (x, y, z, w=1.0) in big-endian float32. These are the points the solver is allowed to move. In Coupe.deform the array is 61 vertices = 976 bytes and ends exactly at 0x3F0, where the index data begins — the clean prediction is how vertexCount was confirmed.

Section 2 — index & spring data

Following the vertices are two interleaved kinds of data:

  • uint16 index lists — triangle triplets and per-group vertex strips (e.g. 00 00 00 02 00 03 …), used to bind the cage to the render mesh and to grow part-groups.
  • 10-byte spring/edge records — a float32 rest-length, a 2-byte flag pair (01 01), then two uint16 vertex indices. This is the mass-spring graph that resists and propagates deformation.
0004de  3e c1 5c 8c 01 01 00 06 00 07   restLen=0.378  flag=0101  a=6  b=7
0004e8  3e c1 5c 8c 01 01 00 00 00 01   restLen=0.378  flag=0101  a=0  b=1
0004f2  3e c1 5c 8e 01 01 00 04 00 05   restLen=0.378  flag=0101  a=4  b=5

Spring records from Coupe.deform. Each is 10 bytes: float32_t restLength · uint8 flag · uint8 flag · uint16 vertA · uint16 vertB. Theory on the exact flag meaning; the <float> 01 01 <u16> <u16> stride is consistent across the whole section.

Section 3 — named rigid part-groups

The file ends with a block of fixed 100-byte records, one per partGroupCount (header field D) — on chassis files this block runs to EOF; on body files a short chassis-link footer follows it. Each record names a rigid sub-assembly of the car — wheels, discs, wishbones, chassis sections, lights, radiator — that deforms as a unit (or stays rigid). The 100-byte stride is Confirmed by walking Coupe.deform: consecutive names (CHASSIS_FRONT @ 0x1134, HUB_FL @ 0x1198, …, DISC_FL @ 0x1774) sit exactly 0x64 bytes apart, 17 records ending at EOF. The names are human-readable and match the EBOOT's PartGroups menu:

001648  48 55 42 5f 46 52 00 00 …   "HUB_FR"   (front-right wheel hub)
0016ac  48 55 42 5f 42 52 00 00 …   "HUB_BR"   (back-right wheel hub)
001710  57 49 53 48 42 4f 4e 45 5f 42 52 00 … "WISHBONE_BR"
001774  44 49 53 43 5f 46 4c 00 …   "DISC_FL"  (front-left brake disc)

Footer — chassis link

Body files (under Vehicles/Bodies/) close with a short footer: a null-terminated chassis-name string followed by three 1.0 floats interleaved with zero words (reading as a scale triple, see below). This binds the body's deformation cage to the shared chassis it sits on (Supercar_01.deform → "Supercar_Low", Truck_01.deform → "Truck") — mirroring the vehicle Body/Chassis split. Chassis files (Vehicles/Chassis/) have no such footer (their countF/countG header words are both 0). Confirmed: Supercar_01.deform ends with "Supercar_Low\0" at 0x6cfc.

The three footer floats read as little-endian 1.0 (00 00 80 3f) even though the body of the file is big-endian (3f 80 00 00). This is almost certainly an exporter quirk; treat the footer scale as Theory until cross-checked at runtime.

How it fits the engine

The cage feeds GPU deformation: the model's vertex shader samples the moved positions via the DeformVertexPosition_DeformTexture attribute seen in the EBOOT shader-context strings, so the visible body crumples without re-skinning. Tuning lives in the per-car SurfaceDamage.params and Vehicles/Generic/SurfaceDamageCommon.params (DeformerPlaneAngle/Height/Dist, scrape planes, MaxVertDeformations). Compare this to Burnout Paradise's crash deformation idea — the concept (a deformable vehicle cage) is similar, but Split/Second's DFM2 container is its own Black Rock format with no Havok and no shared layout.

Confidence & open questions

Overall Partial. Solid: the DFM2 magic and its EBOOT check; the 16-byte big-endian vertex array sized by header field A; the 100-byte named part-group records sized by header field D; the chassis-link footer. Open: the precise role of header counts B/C/E/F/G; the two flag bytes in the spring records; the exact field layout inside each 100-byte part-group record (name length vs. padding boundary); and the footer-float endianness.
