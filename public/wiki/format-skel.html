Skeletons (.skel)

The .skel file is the bone hierarchy (rig) for an animated mesh. In Split/Second it is used almost exclusively by powerplay set-pieces — collapsing buildings, exploding tanks, helicopters, rolling vans — where each destructible chunk is a joint driven by baked animation. It is a Black Rock container with the ASCII magic ftsc, and it maps onto Havok's hkaSkeleton (Havok Animation) at load time.

Identification

The first four bytes are 66 74 73 63 = ASCII ftsc, constant across every sample. (Read little-endian it spells cstf — plausibly a Crayon-family "structure file" tag.) It is followed by two 16-bit fields that are also constant: 00 64 (100) and 00 10 (16), most likely a version and an alignment/stride. The format is not a Havok packfile — there is no 57 E0 E0 57 magic and no embedded class strings — so the data is parsed by Black Rock's own reader and handed to Havok.

The EBOOT confirms the Havok-Animation backing: the string table contains hkaSkeleton, hkaSkeletonUtils, hkaSkeletonMapper, hkaSkeletonMapperData… and the assert "None of the bones in skeleton have translations locked — did you forget to lock translations?" from Rig/hkaSkeletonUtils.cpp. The ftsc file is the cooked, big-endian rig that populates an hkaSkeleton (parent map + reference pose). See Havok Packfiles for the wider Havok 5.5 stack.

Header & offset table

After the 12-byte preamble comes a 16-bit bone count, then an 8-entry offset table of 32-bit absolute file offsets. Each table entry points at one parallel array (all indexed by bone), followed by a final entry pointing at the name block. The layout was verified against three samples with bone counts 2, 24 and 58 — every offset lands exactly where the next array begins, and the array lengths scale linearly with the bone count.

Offset-table entries

The eight offsets describe the per-bone arrays. Lengths below are per bone × bone_count, padded to 16 bytes. (Values shown are the 2-bone PG02_Helicopter_A.skel probe.)

EntryPoints atElementNameDescription
[0]0x40uint16_tparent_index[]Parent bone index per joint; 0xFFFF marks the root. This is the hierarchy.
[1]0x50uint16_tbone_order[]A second index array — bone ordering / remap (sequential 0,1,2,… in flat rigs).
[2]0x60float32[16]matrix_A[]4×4 reference-pose matrix per bone (identity in flat rigs — diagonal 3F800000=1.0).
[3]0xE0float32[16]matrix_B[]Second 4×4 per bone — likely the world / inverse-bind matrix (carries the bone translation).
[4]0x160float32[4]vec_C[]16-byte vector per bone (quaternion or translation+pad). Theory
[5]0x180float32[4]vec_D[]16-byte vector per bone.
[6]0x1A0float32[4]vec_E[]16-byte vector per bone.

File-size sanity check (58-bone PG05_MultiBus_1_A.skel): the name block starts at 0x2920 and there are 58 × 0x40 = 0xE80 bytes of names → end 0x37A0 = 14 240 bytes = the exact file size. Every array offset in the table was confirmed the same way.

Annotated: ftsc magic; version 0x64; stride 0x10; bone_count=2; the 8-entry offset table; parent_index[0]=0xFFFF (root) and [1]=0 (child of bone 0); identity matrix_A; then the two 64-byte joint names root_group1_export and joint_Explosive1.

Bone naming conventions

Joint names are authored in the modelling tool and exported verbatim. Two patterns dominate:

PatternExampleMeaning
root_…_export / root_…_rigroot_PA03_Car1_A_rigThe root bone; embeds the animation/asset name.
joint_Explosive<n>joint_Explosive1A chunk that flies apart in an explosion.
joint_nxRigidBody<n>joint_nxRigidBody48A bone bound to a physics rigid body (the nx prefix points at the DCC physics-rig exporter).
Game_joint<n>Game_joint36Generic numbered joint (bulk destructible pieces).

The presence of dozens of independent joint_nxRigidBody* / Game_joint* bones — rather than a humanoid skeleton — confirms these rigs animate shattering scenery: each bone is one rigid fragment whose motion is baked into the paired animation. This matches the folder location (Powerplays/Animations/<Level>/…) and the powerplay set-piece system.

Where they live

.skel files sit under Deferred/Powerplays/Animations/<Level>/, grouped by event — e.g. Downtown/PA01_collapsing_building/, Downtown/PA03_Heli_tunnel/, Downtown/Generic/PG05_MultiBus/. Each rig is named with the _A suffix (animation take) and is accompanied by the corresponding animated .model mesh and baked animation data that the Catnip powerplay timelines trigger.

Open questions: the role split between matrix_A and matrix_B (local bind vs world/inverse-bind) is inferred from the identity-vs-translation contents, not proven; and the three 16-byte vectors (vec_C/D/E) are most likely a decomposed translation / rotation-quaternion / scale triple but their exact assignment is unconfirmed. The constant stride=0x10 in the header may be the element size of those vectors. Confirming this against a Havok hkaSkeleton tagfile would settle it.
