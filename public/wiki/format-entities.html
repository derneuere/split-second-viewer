Entities (.entities)

A small binary table of placed entity instances for one subtrack of a level. It is part of the Catnip gameplay data set that sits next to a level's .powerplays and .triggers files. In every shipped subtrack the file holds exactly one entity class — the eight Entity_StartPosition_PlayerN race-grid slots — so the format below is fully decoded for that case and the header is generic enough to describe other entity types should they appear.

At a glance

All 19 .entities files on disk are byte-for-byte identical in their header and are all exactly 808 bytes. The layout is trivially regular: a 32-byte header followed by count fixed-size 97-byte records, with no offset table, no string pool and no trailing data. The math closes exactly: 0x20 + 8 × 97 = 808, i.e. the file ends on the last record byte.

Unlike its two siblings, .entities is a real binary blob. The level's .powerplays and .triggers files are plain XML. Don't assume the Catnip family shares an encoding.

File header

The 32-byte header is constant across every sample (same magic, same reserved zeros, same version 3, same count 8). Only the magic, version and count carry meaning; the rest is reserved padding.

OffsetLengthTypeNameDescription
0x000x4char[4]magicAlways ENTS (45 4E 54 53).
0x040x4uint32_treservedAlways 0 in all samples. The EBOOT version-checks an id, a major and a minor version (see note below), so this word is plausibly the minor version or a file id — but since it never varies in shipped data the meaning is unconfirmed.
0x080x4uint32_tversionFile major version. 3 in this build. The EBOOT contains "entities file id mismatch", "entities file major version mismatch" and "entities file minor version mismatch - attempting to load", so the loader id- and version-checks the header (a minor mismatch is a soft warning, a major mismatch is hard).
0x0C0x4uint32_tcountNumber of entity records that follow. Always 8 here (the eight player start positions). Equals exactly (filesize − 32) / 97.
0x100x10byte[16]reservedAll zero in every sample. Padding to a 32-byte boundary for the first record.

Entity record (97 bytes)

Each record is a fixed-width, null-padded name followed by a 4×4-style transform stored as 16 big-endian float32s. The record stride was confirmed by locating consecutive Entity_StartPosition name markers: every gap is exactly 97 bytes across all eight records in every file.

OffsetLengthTypeNameDescription
0x000x21char[33]nameNUL-terminated, NUL-padded entity name, e.g. Entity_StartPosition_Player1 (28 chars, so the terminating NUL falls at +0x1C and bytes +0x1C…+0x20 are all NUL). The field width is 33 bytes (0x21), which is why position.x only decodes correctly when the float block is read from record +0x21 (verified: reading from +0x20 yields garbage 6.14, from +0x21 yields the correct −1512.30).
0x210xCfloat32_t[3]positionWorld-space translation X,Y,Z. Verified against the world: Downtown P1 = (−1512.30, 9.31, 321.57).
0x2D0xCfloat32_t[3]scalePer-axis scale; (1.0, 1.0, 1.0) for every start position.
0x390x24float32_t[9]rotationRow-major 3×3 rotation/basis matrix. Identical for all eight grid slots in a subtrack (the cars all face the same way).
0x5D0x4float32_tindexSpawn index, stored as a float. Matches the PlayerN suffix exactly: 1.0…8.0. This is how the engine maps a record to grid position N regardless of file order.
The transform decomposes as 3 position + 3 scale + 9 rotation (15 floats) with a 16th "index" float. Reading it as a flat float32[16] and slicing 3/3/9/1 reproduces sane world coordinates and an orthonormal basis; reading it as a single 4×4 matrix does not, because the index occupies what would be the matrix's last element.

Decoded example — Downtown / Subtrack A

The 8 start positions form a staggered 4-row × 2-column race grid: X steps ~14 units between columns, Z steps ~7.7 units between rows, all sharing one rotation. The index values 1–8 are not in file order (records appear 1,4,3,2,5,6,7,8), confirming the engine keys off the index field, not the slot order.

Open questions

  • The reserved word at 0x04 is always 0. The EBOOT performs three separate header checks (id, major version, minor version) but the header only exposes one obvious version field at 0x08; the word at 0x04 is the natural candidate for the minor version or a file id. Unconfirmed — no shipped sample varies it.
  • Only one entity class (cStartPositionEntity) is ever instanced in shipped data, so it is unknown whether other entity types would carry extra per-record payload after the transform, or whether the 97-byte record is a universal entity stride. The fixed record size and lack of a per-record type tag suggest the latter for this version.
