Route Data (.checkpoints / .sideways / .splitlength / .linkorigins)

Every race route in Split/Second is described by a small bundle of binary files under Environments/Levels/<Level>/Event/RACING/TrackLogic/<Route>/. Together they form the TrackLogic graph: a set of Nodes (waypoints), the directed Links between them (the racing line and its branches), lateral "sideways" connections used by the AI to change lanes, the per-section split distances, and the lap checkpoints. The same graph drives the SSAI FollowRacingLine / FollowCloseRacingLine behaviour and lap timing.

A route folder is named per sub-track (A, B, C, D) — the alternate routes a level exposes (the EBOOT log line SubTrack 'A', LightRig 'default' uses the same A…D naming). On this build Downtown carries routes A/B/C, docks A/B/C/D, etc.; a level may carry up to four route folders. The EBOOT loads them via the path templates environment(noassert)|Event/%s/TrackLogic/%s then %s/Track.linkorigins, %s/Track.checkpoints, %s/Track.logicinfo, %s/Track.nis, %s/Track.lip.

The route graph (Nodes & Links)

The shipping build keeps plaintext debug dumps next to the binaries, which document the model exactly. Track.debug.txt lists every Node with its world position and per-node flags; it then lists every Link as an ordered Node A → Node B pair, grouped into numbered SECTION blocks with a Helicopter Only flag. Track.sideways.txt states the link count and the sideways adjacency of each link.

For the Downtown route A the dumps report 276 Nodes, 283 Links and 19 Sections — numbers that the binary files reproduce exactly (see below), which is how each file's count field was identified. A "Node" is a point on the route; a "Link" is a directed spline segment between two nodes; a "Section" is a contiguous run of links forming one route branch (and is the unit used by lap timing and the respot system).

Track.linkorigins — per-link arc-length origins

Confirmed A 4-byte big-endian link count followed by one float32 per link. The count equals the "Total Links" in Track.sideways.txt for every sample checked, and the file size is exactly 4 + count×4. The floats rise roughly monotonically along the route, so they are the arc-length distance at which each link begins — i.e. where each link "originates" on the spline (hence the name). The AI's RouteTaking / TrackNavigator skills convert a distance-down-track into a link via this table.

OffsetLengthTypeNameDescription
0x00x4uint32_tlinkCountNumber of links. = "Total Links" in the txt dump.
0x40x4 × Nfloat32_t[]origin[N]Arc-length distance (metres) of each link's start along the route spline.
$ od -A d -t x1z Track.linkorigins | head -2     # Downtown route A
0000000 00 00 01 1b 43 3d 6c ed 43 5d 30 42 43 6f 2c f5  >....C=l.C]0BCo,.<
        └──count──┘ └origin[0]┘ └origin[1]┘ └origin[2]┘
        0x11b = 283 links      189.43      221.19      239.18   (big-endian float32)
Verification: Downtown A 1136 bytes → (1136−4)/4 = 283 links ✓; docks A header 00 00 01 21 = 289 = its "Total Links" ✓; Graveyard A header 00 00 01 50 = 336 ✓.

Track.sideways — lateral adjacency table

Confirmed A 4-byte big-endian link count (identical to .linkorigins) followed by one variable-length record per link: a single uint8 count N, then N uint16 link indices. Each record lists the links that run alongside the current one — the lanes the AI may slide into when overtaking, blocking or dodging a powerplay. This decodes Track.sideways.txt byte-for-byte.

OffsetLengthTypeNameDescription
0x00x4uint32_tlinkCountNumber of links (matches .linkorigins).
0x4—record[]sideways[N]One record per link, in link order. See record layout.
OffsetLengthTypeNameDescription
+0x00x1uint8_tcountNumber of sideways links for this link (0 = none).
+0x10x2 × countuint16_t[]linkIndex[]Indices of the laterally-adjacent links.
$ od -A d -t x1z Track.sideways | head -4         # Downtown route A
0000000 00 00 01 1b 00 00 00 00 00 00 02 00 f4 00 e7 02  >................<
        └──count──┘ │  │  │  │  │  │  └245┘ └231┘
        283 links   l0 l1 l2 l3 l4 l5  └─ link[6]: count=2 → (244,231) ─┘
                    (all 0 = empty)        ↳ matches "link [6] sideways links to ([244][231])"
Sideways targets are link indices, not node indices — the txt dump's ([244][231]) notation lists link IDs. A value pair like (244,231) means link 6 can change lane onto either link 244 or link 231.

Track.splitlength — per-section split lengths

Confirmed (layout) Partial (units) A 4-byte big-endian section count followed by one float32 per section. The count equals the number of SECTION n BEGIN blocks in Track.debug.txt for every sample (Downtown A 19, Graveyard A 18, Powerplant A 31, docks A 41 — each matching its SECTION block count), and the size is exactly 4 + count×4. The values cluster around 1.0 (3f 80 00 00) with small deviations (1.087, 1.069…), so they read as per-section length multipliers / split weights rather than raw metres — most likely scaling how each section contributes to the lap "split" used by the HUD split timer and the AI distUpSection calculations (cf. EBOOT distUpSection greater than section length).

OffsetLengthTypeNameDescription
0x00x4uint32_tsectionCountNumber of sections (= SECTION blocks in the debug dump).
0x40x4 × Nfloat32_t[]splitLength[N]Per-section length weight (≈ 1.0). Exact unit unconfirmed.
$ od -A d -t x1z Track.splitlength               # Downtown route A (80 bytes)
0000000 00 00 00 13 3f 80 00 00 3f 8b 1a a1 3f 89 80 a8  >....?...?...?...<
        └──count──┘ └─ 1.0 ─┘ └ 1.0874 ┘ └ 1.0742 ┘
        0x13 = 19 sections   (80 = 4 + 19×4)  ✓ matches 19 SECTION blocks

Track.checkpoints — lap checkpoint table

Partial Unlike the three files above, Track.checkpoints is a fixed-size serialized record — exactly 348 bytes on every route in the game — and uses Black Rock's tagged serialization framing rather than a plain count+array. The 12-byte header is constant (00 03 00 00 cd ab 0d f0 00 00 01 50) and the body always ends with the sentinel ba dc ad de. The sibling Track.logicinfo shares the identical framing (and an identical opening payload word), confirming both come from the same serializer.

OffsetLengthTypeNameDescription
0x00x4uint32_tversion0x00030000 = version 3.0. Constant across all routes.
0x40x4uint32_tbeginTag0xCDAB0DF0 — serializer "begin object" sentinel (shared with .logicinfo).
0x80x4uint32_tbodySize0x150 = 336 = file size − 12. Byte length of the payload that follows (it includes the body's own closing sentinel).
0xC0x150—bodyCheckpoint sub-objects (see below). A run of nested cd ab 0d f0 … ba dc ad de tagged blocks. The body's final 4 bytes (offset 0x158) are the closing 0xBADCADDE sentinel, so the file ends exactly at 0x15C = 348.
There is no separate trailing endTag field after the body: bodySize (0x150) already accounts for the closing ba dc ad de, and 0xC + 0x150 = 0x15C = 348 lands exactly on EOF. The last 12 bytes of every file are in fact three stacked ba dc ad de sentinels, closing the nested sub-objects.
$ od -A d -t x1z Track.checkpoints | head -2     # any route — always 348 bytes
0000000 00 03 00 00 cd ab 0d f0 00 00 01 50 49 39 a9 9a  >...........PI9..<
        └ver 3.0 ┘ └begin tag┘ └ 0x150  ┘ └ payload starts (49 39 a9 9a = 760473.6f) ┘
                                = 336 = 348-12

The EBOOT exposes the checkpoint record field names directly, which lets us name the payload contents even though the exact byte offsets inside the 336-byte body are not yet fully pinned:

The 336-byte body holds several nested cd ab 0d f0-tagged sub-objects, each closed by its own ba dc ad de sentinel (so the framing is recursive, not a single begin/end pair), interleaved with float triples — e.g. 48 4d c9 b4 = 210726.8 recurs as a constant across the sub-objects. These read as checkpoint placement transforms (position + plane normal) but their precise order is Theory pending a cross-reference against a level whose checkpoints are visible in-game.

Relationship to the rest of the level

  • Track.logicinfo — same serializer and framing (00 03 00 00 cd ab 0d f0 …), and like .checkpoints it is a fixed-size record: 288 bytes on every route, with a constant body-size field 0x114=276 (= 288−12). Caveat: 276 is not a per-route node count — it is identical on all routes, whereas node counts vary (Downtown 276, docks 274, Graveyard 330, Powerplant 265); the Downtown coincidence aside, the body is mostly zero-padded and holds only a short header of floats plus a couple of nested sub-objects. Its exact contents are Theory.
  • Track.nis — maps named non-interactive sequences (e.g. PA03-1, PP08-2) to link indices, triggering takedown/powerplay cameras as the pack passes.
  • Track.lip — the largest file in the folder; voice lip-sync for the commentary tied to the route.
  • .sectorInfo — streaming partition the route runs through.
