Engine Architecture

Split/Second (2010) runs on Black Rock Studio's in-house engine — the same technology lineage as the studio's Pure ATV racer, not a licensed or shared engine. This page is the hub: it maps the engine's subsystems, the third-party middleware it wraps, the internal codenames, and the studio's pervasive data-driven design where almost everything that matters lives outside the executable as plaintext .params and XML files.

Where this knowledge comes from

Everything below is verified against an actual devkit / QA debug build (title id NPXX00575) whose data ships loose on disk under USRDIR/Deferred/, plus the clear-text strings extracted from EBOOT.BIN. The executable's symbolized source paths follow a single tree — c:/Projects/proj_1/ws_0/Code/{libs,games,contrib}/… — which is effectively a table of contents for the engine. The module map below is reconstructed from those paths and from class names registered in the binary. See Build & File Layout for the build itself and Glossary & Codenames for the vocabulary.

The Code/ tree splits three ways: libs/ (reusable Black Rock engine libraries: Render, Audio, System…), games/SplitSecond/ (the title-specific layer), and contrib/ (third-party middleware: Havok, FMOD, Quazal, Sony Edge, oscpack).

Subsystem map

The engine is a fairly conventional console architecture for its era — a deferred renderer fed by an SPU job system, with middleware for physics, audio, networking, video and UI — wrapped in two distinctively Black Rock layers: Crayon2 (render support) and Catnip (data-driven gameplay/FX). The rendering core: Crayon2

Crayon2 is Black Rock's render-support library (libs/Render/Support/Crayon2). It drives a deferred renderer built around a multiple-render-target G-Buffer. The debug menu confirms exactly five G-Buffer channels — each has its own "Disable … writes" toggle under Lighting/Debug/GBuffer/:

Toggle (debug menu)G-Buffer channel
a;Disable albedo writesAlbedo / base colour
b;Disable unlit writes"Unlit" (emissive / pre-lit)
c;Disable normal writesSurface normals
d;Disable motion blur writesPer-pixel velocity (motion blur)
e;Disable specular writesSpecular

The frame pipeline shows up as named GPU timing stages in the EBOOT: Deferred - GBuffer - Opaque → GBuffer End MRT → GBuffer SSAO → GBuffer Blur → lighting → post (SSMotionBlur, bloom, edge). Two distinctive features: lighting runs on SPUs via "SPULight" jobs (SPULight Build Jobs, SPULight Copy Depth, SPULight Stall Global Light, SPULight Window of opportunity), and the renderer uses RSX ZCull (GBuffer ZCull Restore). Shaders are not statically linked: Crayon2/Shader/PS3/ShaderPatching.cpp patches shaders at runtime, fed by a 2.27 MB übershader database global_shader_combos.fxc (exact size 2,382,848 bytes on disk). Full detail on Rendering Pipeline.

The execution core: JobStream over SPURS

Work is dispatched through JobStream (libs/System/JobStream), Black Rock's wrapper over Sony's SPURS SPU runtime. The EBOOT contains the literal job-chain setup call:

cellSpursJobChainAttributeInitialize( &attr, mCommandList, mJobSize,
    mMaxGrab, mPriorities, mNumSPU, true, 0, 1, false, 256, 1 )

Above the SPU layer the game runs a small set of named CPU threads, visible as profiler labels: Front End (Main Thread) / Level (Main Thread) and dedicated Front End (Loader Thread) / Level (Loader Thread) plus a generic LoaderThread for asynchronous resource loading. Big consumers of SPU time are SPULight (lighting), EdgeGeom (geometry decompression), and Havok. See Threading & JobStream.

Data-driven design

The defining trait of this engine is how little is hard-coded. Tuning, content layout, AI behaviour and FX choreography all live in editable data files that the EBOOT reads at runtime:

  • .params — plaintext key/value tuning with slider ranges, e.g. an SSAI driving table reads '002;Multiplier to throttle when accelerating' = 1.0 (0.05, 1.0); where (0.05, 1.0) is the in-tool min/max.
  • XML — structured databases such as Vehicles/VehicleDatabase.xml (the full car roster), Season/Season.xml and Season/UnlockCriteria.xml.
  • Catnip — the entity/timeline system that choreographs Powerplays, interactive objects, audio events and lens flares (Catnip/PowerplayTimelines/, Catnip/Physics/InteractiveObjects/, Catnip/Audio/Timeline/AudioEvent/, Catnip/LensFlare/).
  • Havok tagfiles — many physics packfiles ship with an XML twin that documents the same data; see Havok Packfiles.

Because this is a QA build, two test/profiling harnesses are also data-driven: SSAI's EvolutionConfig.params (automated AI fitness scenarios) and Megabowles (cMegabowlesController), an automated Powerplay-scenario test runner with filters like MegabowlesScenarios/Filter/powerplay_name and MegabowlesScenarios/Filter/sector_name.

The gameplay signature: Powerplay

Split/Second's hook — blowing up the track around your rivals — is the Powerplay system, one of the most heavily referenced gameplay systems in the binary (Powerplay appears in ≈226 EBOOT strings). It is a constellation of classes: cPowerplayManager, cPowerplayController, cPowerplayMaterialManager, and PowerplaySequenceDirector, driven by Catnip timelines and a dynamic "danger zone" trigger system (GameplayParameters/Powerplays/Dynamic Trigger). It is tightly coupled to the camera director (CameraDirector/PowerplaySequenceDirector/Debug Takedown Sequences), AI, and the respot/teleport machinery that keeps the pack cinematic. Documented on Powerplay System.
