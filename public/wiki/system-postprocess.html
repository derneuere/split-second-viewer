Post-Processing Database

The PostProcessDatabase is Split/Second's library of full-screen "looks". Each named entry bundles a complete set of HDR post-process settings — tone mapping, bloom, colour-cube grading, motion blur, depth-of-field, lens/particle flare, gradient and vignette — that the engine blends in for a given game context (a crash cam, a Powerplay explosion, the pause menu, high-speed driving, the results screen, and so on). It is the data that gives each moment its distinct cinematic colour.

How it fits the HDR pipeline

Split/Second's deferred renderer resolves the lit scene into an HDR buffer (HDRPostProcessingInput in the eboot's render-target list). The post-process chain then runs as a sequence of named effects — the eboot exposes them as BloomEffect, SSMotionBlur, DepthOfFieldEffect, LensFlareEffect, ParticleFlareEffect and ColorCubeEffect, all under a Render_PostProcessing pass that ends at PostProcessFinish. The cPostProcessingController picks which database entry is active each frame and cross-fades its parameters in/out. Tone mapping and colour grading are the last creative stage before the image is shown.

The colour-cube grading shares its art with the camera director: the eboot keys Cameras/Data/<cam>/ColorCubeEnabled, …/ColorCubeBlending and …/ColorCubeInputBlending mirror the database's ColorCube block, and cColorCubeBehaviorData loads per-camera grade behaviour.

Entry structure

Every entry is a directory /PostProcessDatabase/<Context> with one section per effect plus a root section of overwrite flags. The root section names the entry, sets its Fade In Time / Fade Out Time (seconds), and — crucially — declares which effect configurations this context is allowed to override on top of the level's base look. For example the Area Of Effect entry overwrites tone mapping, bloom, colour-cube, motion-blur and vignette, but leaves DoF, gradient and the flares to the level:

/PostProcessDatabase/Area Of Effect:
	'a; Name' = 'Area Of Effect';
	'b; Fade In Time' = 0.5 (0.0, 30.0);
	'c; Fade Out Time' = 0.5 (0.0, 30.0);
	'd; Overwrite Tone Mapping Configuration' = true;
	'd; Overwrite Tone Mapping Saturation Configuration' = false;
	'e; Overwrite Bloom Configuration' = true;
	'e; Overwrite Color Cube Configuration' = true;
	'f; Overwrite DOF Configuration' = false;
	'g; Overwrite Motion Blur Configuration' = true;
	'h; Overwrite Gradient Configuration' = false;
	'i; Overwrite Vignette Configuration' = true;
	'j; Overwrite Lens-Flare Configuration' = false;
	'k; Overwrite Particle-Flare Configuration' = false;

The single-letter prefixes (a;, b; …) and @ rows are the .params editor's ordering/colour-swatch hints — e.g. '@f; Tone' = ':f; Tone:color:'f; Tone_r' 'f; Tone_g' 'f; Tone_b' ' tells the debug UI to draw the three RGB floats as one colour picker.

The 18 contexts

The dynamic database (PostProcessDatabase.params) contains eighteen entries. These map 1:1 to the gameplay/UI moments the engine can force — the eboot's debug menu even lists matching overrides (/a; ForceCrashEffect, /j; ForceHighSpeedEffect, /k; ForceJumpEffect, /n1; ForcePowerPlayTakedown, etc.).

Colour cubes (3D LUT grading)

The ColorCube block names two LUT textures and blends between them (Color Cube Blending) and against the ungraded image (Color Cube Input Blending). The cubes live as a .textures pack at UI/Frontend/ColorCubes/ColorCubes.textures (loaded by the Resident(noassert)|UI/Frontend/ColorCubes/%s path), and the database references them by name. The ColorCubes.textures pack uses the engine's standard TEXS texture container (magic 54 45 58 53, "TEXS"); the in-game frontend "Decals/ColorCubes" viewer reads the same data. A debug override exists at PostProcessDatabase/Debug/ColorCubeOverride (Enabled + Texture0/Texture1) plus a Reload Color Cubes command, which is how artists hot-swap grades in the QA build.

Worked example: tone mapping & bloom

The two most visually decisive blocks are the tone mapper and bloom. Comparing the gentle Area Of Effect look with the punchy Crash look shows how the same fields are re-tuned per context:

The crash look raises the white point (less clipping of the fireball), lifts the bloom threshold (only the brightest highlights bloom) and warms the bloom tone, while keeping the same blur cost. Auto-exposure runs over a 4-pass-blurred average luminance clamped to [0.2, 5.0] in both.

Filmic toe/shoulder are present but off. Every entry carries Filmic Enabled = false with a Reinhard-style toe/shoulder ready (Filmic Toe = 1.5, Filmic Shoulder = 0.3). So the shipped game uses the simple exposure tone curve, not the filmic path — treat the filmic branch as Partial (data exists, unused here).

Triggered by gameplay

The controller exposes a handful of timing knobs in the eboot, e.g. PostProcessDatabase/High Speed Post Process Effect: Velocity vs Intensity (a Velocity→Intensity curve), and the landing window PostProcessDatabase/Min / Max Time For Landing to apply High Air Post Process Effect that gates the Jump look on touch-down. A global PostProcessDatabase/Disable Game Event Post Processes switch turns the whole event-driven layer off for debugging. Powerplay-driven looks are fired by the Powerplay system via cPostProcessTrigger.
