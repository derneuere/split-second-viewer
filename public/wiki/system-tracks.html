Tracks & Levels

A Split/Second "level" is a small TV-studio set dressed up as a city, harbour or quarry. Each level lives under Deferred/Environments/Levels/<Level>/ and is assembled at runtime from a fixed set of sub-systems: Subtracks (the alternate routes A/B/C), Sectors (the streamable chunks of world geometry), LightRigs + ColorCubes + Skydome (the time-of-day look), a Backdrop and ReflectionMap set, and the per-route route data the race logic and AI run on. This page walks the roster and the pipeline, quoting the real on-disk layout (Downtown is used as the worked example).

The track roster

Deferred/directory.txt is the master asset manifest; its :TRACKS section lists every level, and a <Level>_SUBTRACK section lists each level's route letters. The Subtracks per level (verbatim from directory.txt):

LevelSubtracksType
DowntownA, B, CRace

nem_downtownANemesis

The retail game shipped many more named tracks (Airport, Construction, Suburbs, River, Aquarium…), but they are all subtracks / route layouts cut from this handful of physical environments. The same Downtown geometry is re-routed and re-lit to produce a whole set of distinct-feeling races. scale_track_01 and mlr_fastload appear in SubTrackLightRigs.txt as internal test/load-benchmark levels.

Subtracks: route variants

A subtrack is one drivable route through the level. The engine loads it from Environments/Levels/%s/Subtracks/%s (confirmed in the eboot, alongside SetSubTrack and the log line mSubtrackName: %s). Each subtrack folder (Downtown/Subtracks/A/) carries the gameplay layer for that route:

FilePurpose
Downtown.entitiesCatnip entity instances placed on this route — see Entities
Downtown.powerplaysPowerplay definitions active on this route — see Powerplays
Downtown.triggersTrigger volumes — see Triggers
Downtown.hkPPs / .hkPPsXmlPhysics packs (+ XML twin) — see Physics Packs
Downtown.hkRBs / .hkRBsXmlRigid bodies (+ XML twin) — see Rigid Bodies
tl.hkColl / tl.hkCollXml / tl.xmlThe route's (track-logic) collision mesh — see Collision
SectorVisibility.xmlWhich sectors are visible/streamed along this route (eboot path Subtracks/%s/SectorVisibility.xml)
Particles/TimelineParticles/Per-route particle sets — Geometry / Light / Screen / Standard, each with .emitterControllers, .emitterSurfaces, .timelineInfo (see Emitters)

Route data (race logic)

The racing line, checkpoints and respot/drop metadata live separately under <Level>/Event/RACING/TrackLogic/<Subtrack>/. This is the data the race director, lap timing and AI follow. Each subtrack directory contains the route binaries plus human-readable debug twins:

FileRole
Track.checkpointsBinary checkpoint/node graph (see below) — detailed on Route Data
Track.linkoriginsLink origin table for the node graph
Track.sideways + .sideways.txtLateral links between nodes (the width of the drivable corridor)
Track.splitlengthPer-split lengths (used for progress / position)
Track.logicinfoTrack-logic metadata — see Logic Info
Track.nis / Track.lipNon-interactive sequence + lip-sync (intro flythrough, radio chatter)
Track.debug.txtHuman-readable node dump (debug build only)

The debug dump shows each node's world position and flags. Note the respot / drop-zone / crash-zone flags per node — this is how the game knows where it is allowed to re-place a wrecked car:

Node [0]
Pos (-1342, 8, 254)
Crash zone [ ]
Can respot [x]
DropZone False
TriggerZone False

The sideways table records the lateral connectivity (Downtown route A: Total Links = 283):

link [6]  sideways links to ([244][231])
link [21] sideways links to ([252][259])

Track.checkpoints itself is a big-endian binary. The header begins 00 03 00 00 (a count/version), then a repeating record marker CD AB 0D F0 precedes each entry — a recognisable sentinel that makes the record stream easy to walk. Full layout on Route Data.

00000010: 48e6 5b4b 3f80 0000 cdab 0df0 0000 0138  H.[K?..........8
00000020: 0e19 a032 cdab 0df0 0000 00e0 5817 1462  ...2........X..b

Sectors & streaming

The bulk of the world — buildings, road, props — is partitioned into sectors and streamed. Sector data sits in <Level>/Sectors/:

FileRole
Downtown.sectorInfoSector table: bounds, ids, and which assets each sector owns — see Sectors
Downtown.Static.arkAlways-resident packed geometry/textures for the level
Downtown.Stream.arkStreamed-on-demand packed assets — see Archives
Downtown.fxc / .shaders / .shaderinst / .texturesSector render set (materials, shaders, textures)

The .sectorInfo opens with float bounds and is tagged with the Crayon source path (UTF-16) …\proj_1\ws_0\Data\Split… — the same build-tool signature seen across the renderer's authored data:

00000000: 3ff5 c28f 4396 0000 0000 0080 7130 3030  ?...C.......q000
00000010: 0000 5c00 7000 7200 6f00 6a00 5f00 3100  ..\.p.r.o.j._.1.
00000020: 5c00 7700 7300 5f00 3000 5c00 4400 6100  \.w.s._.0.\.D.a.

Per-subtrack streaming residency is selected by that route's SectorVisibility.xml, so route A and route C of the same level stream different chunks. See Resources & Streaming for the residency model.

Backdrop & Skydome

Far scenery is split into a low-detail Backdrop (the distant city silhouette) and a Skydome. The backdrop is loaded from environment|Backdrop/%s_backdrop.model (with a matching .texture.crcs), confirmed by the eboot's "Backdrop for '%s' not found!" assert. Downtown ships Downtown_backdrop.model + shaders/textures.

The Skydome is per-time-of-day. <Level>/Skydome/ contains a base Skydome plus Skydome_Midday, Skydome_Sunrise, Skydome_sunset variants (each a full model/shader/texture set). The active skydome is named in the LightRig's /Lighting/Sky/Filename field. Render order is visible in the renderer's pass list (Opaque Skydome Render, Opaque Env Backdrop Render) — see Rendering Pipeline.

Time-of-day: LightRigs, ColorCubes & ReflectionMaps

The dramatic "same track at sunrise vs sunset" looks are achieved by swapping a LightRig. Environments/SubTrackLightRigs.txt maps each level:subtrack:lightrig — i.e. which lighting a route ships with:

downtown:A:midday
downtown:B:sunset
downtown:C:sunrise
docks:A:midday   docks:B:nighttime   docks:C:sunset
Graveyard:A:midday   Graveyard:B:sunset
Powerplant:A:twilight   Powerplant:B:midday

The eboot confirms the loader: "Env '%s' : SubTrack '%s' : LightRig '%s'" and a fallback "Adding default combo of : SubTrack 'A', LightRig 'default'".

LightRig (LightRigs/<name>/<name>.params)

A LightRig is the master lighting descriptor for one time-of-day. Beside the .params it carries baked .lights, a .gbx, and its own shader/texture set. The .params is a single big /Lighting tree. Real Downtown midday values:

SectionWhat it sets
/Lighting/sunsun specular colour, elevation 0.4148, rotation -0.3565
/Lighting/sun/diffusea 6-step diffuse ramp (0→5) for sun intensity over the day
/Lighting/sun shadowsingle-player cascade range 300, fadeout 140, split weight 0.8, with separate per-platform xenon and ps3 per-cascade bias/slope blocks, plus a tighter split screen rig (range 150)
/Lighting/ambience & ambience_vehiclesky/ground hemisphere ambient (separate tint for cars)
/Lighting/kickera fill/rim "kicker" light (elevation 0.6525, rotation 0.2515)
/Lighting/RadiosityGI tweak (1.12) + tint colour
/Lighting/LS & ATV_LSatmospheric/light scattering (Mie/Rayleigh coeffs, extinction & inscattering tones)
/Lighting/ReflectionMap/Filesnames the reflection set: Env = environment_A, Skydome = Skydome_Midday, SunLight, PointLight, SpotLight
/Lighting/ParamsPost Process = PostProcess_Downtown_Midday.params (links to Post-Processing DB)
/Lighting/ColorCubeSet Name = Afternoon — selects the grading LUT
/Lighting/SkyFilename = Skydome_Midday, Rotation 241°
/Lighting/ParticleOverridesper-rig tint for particles (sun/ambient/emissive/light/corona colours)

LightRigs/default/default.params is the fallback rig; LightRigNames.names is the string table of rig names for the level. Downtown ships rigs midday, sunrise, sunset (matching its three subtracks).

ColorCubes (colour-grade LUTs)

A ColorCube is a 3D-LUT colour grade applied in post. They live in <Level>/ColorCubes/<Set>/default.textures, indexed by a dirlist. Downtown ships three sets:

ColorCubes/Afternoon/default.textures
ColorCubes/Sunrise/default.textures
ColorCubes/Sunset/default.textures
dirlist:  Afternoon\default  Sunrise\default  Sunset\default

The active cube is named by the LightRig (/Lighting/ColorCube/Set Name), and the camera system can blend cubes per shot — the ColorCubeBehavior exposes ColorCubeEnabled, ColorCubeBlending, ColorCubeInputBlending (see Cameras & Camera Director). The render pass is the eboot's "Post Processing ColorCube Render".

ReflectionMap set

The cubemap/reflection inputs sit under <Level>/ReflectionMap/, grouped by what they represent — each a small model+shader+texture bundle the engine composites into the environment reflection used by car paint and wet surfaces:

SubfolderContents
Environments/environment + per-subtrack environment_A / _B / _C
Skydomes/Skydome_Midday, Skydome_Sunset, Skydome_Surise (sic) variants for reflections
Lights/PointLight, SpotLight reflection proxies
SunLights/SunLight reflection proxy

The LightRig's /Lighting/ReflectionMap/Emissive block weights how much each contributes (e.g. Downtown midday: emissive scale 31.46, environment 1.0, skydome 0.75, sun 0.76).

Putting it together: loading a race

When the game loads "Downtown, route C", it: selects subtrack C; from SubTrackLightRigs.txt picks LightRig sunrise; the rig in turn names the Skydome (Skydome_Sunrise), the ReflectionMap env (environment_C), the ColorCube set and the PostProcess params; it loads the Static.ark always-resident set and streams sectors per Subtracks/C/SectorVisibility.xml; it loads route logic from Event/RACING/TrackLogic/C/Track.checkpoints; and it loads the route's entities, powerplays, triggers and particle timelines. The result is the same physical city, re-routed and re-lit into a fresh-feeling event.

Everything in this pipeline is plaintext config (.txt/.xml/.params) except the packed/streamed geometry (.ark, .sectorInfo, route binaries). The structure and quoted values are Confirmed from the Downtown tree + eboot strings; exact binary field layouts for .sectorInfo and Track.checkpoints are Partial and covered on their format pages.
