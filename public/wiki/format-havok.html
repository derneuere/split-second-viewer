Havok Packfiles

Every physics asset in Split/Second is a Havok 5.5.0 binary packfile — the serialized, platform-native form of a Havok object graph. The container is identical regardless of what it carries (a vehicle rigid body, a level collision mesh, a deformable body), so this page is the reference that the other Havok-format pages link back to. The on-disk magic is 57 E0 E0 57 and the embedded version banner is Havok-5.5.0-r1, matching the contrib/Havok/Havok550 middleware noted in the Middleware Stack.

The format is not Black-Rock-specific: it is the stock Havok packfile serialization (hkPackfileWriter). What makes it self-documenting is that every packfile embeds its own class reflection — the names and member layouts of the Havok classes it contains — so the file can be parsed without the matching Havok SDK headers. Split/Second ships these files loose on disk in the QA build, which is what makes them readable here.

All multi-byte integers and floats are big-endian. Offsets in the tables below are byte offsets from the start of the file unless stated otherwise. Verified against Vehicles/Bodies/Musclecar_01/Musclecar_01.phys (62 208 bytes) and Musclecar_01.mainColl (4 992 bytes); the fixed packfile header is byte-for-byte identical between the two except for userTag, contentsClassNameSectionOffset (the root class name sits at a different position), and the per-file section sizes/fixup offsets.

File magic

A packfile opens with two 4-byte signatures. The first is the Havok packfile magic; the second is a secondary/check word. Both are constant across every Split/Second sample examined.

OffsetLengthTypeNameDescription
0x00x4uint32_tmagic00x57E0E057 — Havok packfile signature.
0x40x4uint32_tmagic10x10C0C010 — secondary signature / sanity word.

Packfile header (hkPackfileHeader)

The fixed header follows the magic. Every field name below comes directly from the hkPackfileHeader reflection still present in the EBOOT string table (eboot_strings.txt), which lists, in order: magic / userTag / fileVersion / layoutRules / numSections / contentsSectionIndex / contentsSectionOffset / contentsClassNameSectionIndex / contentsClassNameSectionOffset / contentsVersion. The byte values below are from Musclecar_01.phys.

OffsetLengthTypeNameDescription
0x00x8uint32_t[2]magicThe two signatures above.
0x80x4int32_tuserTagApplication tag. 5 in .phys, 1 in .mainColl — the only header field that varies by asset type.
0xC0x4int32_tfileVersionPackfile format version. 0x00000005 in every sample (Havok 5.x serialization).
0x100x4uint8_t[4]layoutRules04 00 01 01 = {bytesInPointer=4, littleEndian=0, reusePaddingOptimization=1, emptyBaseClassOptimization=1}. The 00 confirms big-endian; the 04 confirms 32-bit pointers.
0x140x4int32_tnumSectionsNumber of section headers that follow. 3 here — exactly matches the three section tags found at 0x40 / 0x70 / 0xA0.
0x180x4int32_tcontentsSectionIndexIndex of the section holding the top-level object. 2 = __data__.
0x1C0x4int32_tcontentsSectionOffsetByte offset within that section of the root object (0 = first object).
0x200x4int32_tcontentsClassNameSectionIndexIndex of the section holding the root object's class name. 0 = __classnames__.
0x240x4int32_tcontentsClassNameSectionOffsetOffset within __classnames__ of the root class name string. See verification below.
0x28varchar[]contentsVersionNUL-terminated version banner Havok-5.5.0-r1, padded with 0xFF to the next 16-byte boundary, after which the section headers begin at 0x40.
How the root class is found. In Musclecar_01.phys, contentsClassNameSectionOffset = 0xAC. The __classnames__ data starts at file offset 0xD0, so the root class name sits at 0xD0 + 0xAC = 0x17C — which is exactly the string hkpPhysicsSystem. In Musclecar_01.mainColl the same field is 0x4B and resolves to hkRootLevelContainer. So the header tells you the top object's type purely by following two offsets.

Section headers (hkPackfileSectionHeader)

Immediately after the header come numSections section descriptors, each 0x30 (48) bytes: a 16-byte ASCII tag (NUL-padded), then a table of eight big-endian uint32 fields describing where this section's data lives and where its relocation ("fixup") tables are. Field names come from the EBOOT (hkPackfileSectionHeader: sectionTag / nullByte / absoluteDataStart / localFixupsOffset / globalFixupsOffset / virtualFixupsOffset / exportsOffset / importsOffset / endOffset). The 0x30 stride is confirmed directly by the file: the three tags sit at 0x40, 0x70 and 0xA0 — exactly 0x30 apart.

OffsetLengthTypeNameDescription
+0x000x10char[16]sectionTagSection name, NUL-padded to 16 bytes: __classnames__, __types__, or __data__.
+0x100x4uint32_tnullBytePadding/sentinel word. Reads 0x000000FF in every Split/Second sample (not 0x00000000); its low byte is the "null terminator" the field name refers to.
+0x140x4uint32_tabsoluteDataStartAbsolute file offset where this section's payload begins.
+0x180x4uint32_tlocalFixupsOffsetOffset (relative to absoluteDataStart) of the local pointer-relocation table.
+0x1C0x4uint32_tglobalFixupsOffsetOffset of the global (cross-section) fixup table.
+0x200x4uint32_tvirtualFixupsOffsetOffset of the virtual fixup table (binds objects to class-name entries — this is how each object knows its type).
+0x240x4uint32_texportsOffsetOffset of the named-export table (unused → equals endOffset).
+0x280x4uint32_timportsOffsetOffset of the import table (unused → equals endOffset).
+0x2C0x4uint32_tendOffsetTotal section length (relative to absoluteDataStart); marks the end of the section.
The offsets chain — and they check out. In Musclecar_01.phys the __classnames__ section starts at 0xD0 with endOffset = 0x3B0, so it ends at 0xD0 + 0x3B0 = 0x480 — which is exactly the absoluteDataStart of __types__. And the __data__ section has absoluteDataStart = 0x51A0 with endOffset = 0xA160; their sum is 0xF300 = 62 208, the exact file size. The header math is internally consistent, which is how each field above was confirmed rather than guessed.

The three sections

__classnames__ — the type registry

A flat list of every Havok class used in the file. Each entry is a 4-byte type signature/hash followed by a single 0x09 separator byte and a NUL-terminated class name. The first names are always the reflection meta-classes (hkClass, hkClassMember, hkClassEnum, hkClassEnumItem), then the content classes. Musclecar_01.phys registers 41 classes; Musclecar_01.mainColl registers a much smaller set (just shapes + hkRootLevelContainer).

Reading the first entry: 38 77 1F 8E (signature) · 09 (separator) · "hkClass" · 00. The hkpPhysicsSystem name at 0x17C is the one the header's contentsClassNameSectionOffset points at.

__types__ — class member layouts

The full reflected layout of each registered class: for every class, its members' names, types, offsets and sizes, plus enum definitions. This is what lets the runtime (and a reverse engineer) walk the binary objects field-by-field. It is where strings such as friction, restitution, inertiaAndMassInv, MOTION_BOX_INERTIA and the CONSTRAINT_TYPE_* enum live. Because the layout is embedded, the same parser handles any Havok packfile in the game.

__data__ — the object graph

The actual instance data: the constructed C++ objects (rigid bodies, shapes, motions, constraints) laid out in memory order, with pointers stored as file offsets and resolved at load time via the three fixup tables. This section is the per-asset payload and the only one whose size varies meaningfully between cars. For how it is interpreted as a vehicle, see Vehicle Physics (.phys).

Fixups (pointer relocation)

Because objects reference each other by pointer, the packfile stores three relocation tables per section, located by the *FixupsOffset fields:

TableResolvesNotes
localpointer → another object in the same section(srcOffset, dstOffset) pairs.
globalpointer → an object in another section(srcOffset, sectionIndex, dstOffset).
virtualobject → its class-name entry in __classnames__This is the "what type am I" binding; one per polymorphic object.

At load Havok rewrites the stored offsets into real PS3 addresses, so the on-disk graph is position-independent.

XML twins

Some Havok assets ship a companion XML serialization of the same object graph (the Havok tagfile/XML form), with extensions like .hkCollXml, .hkPPsXml, .hkRBsXml. Where present, the XML is effectively human-readable documentation of the binary class layout — <hkobject class="hkpRigidBody">, <hkparam name="..."> and so on — and is the recommended way to learn what a binary field means.

Not every packfile has a twin. The per-vehicle .phys files do not ship an .physXml — the directory next to Musclecar_01.phys contains no XML. For those, the embedded __types__ registry is the only schema. Twins are more common for the level-side collision/physics packs (.hkPPs, .hkRBs, .hkColl).

Confidence

Confirmed for the container itself: magic, the hkPackfileHeader and hkPackfileSectionHeader field layouts (cross-checked against EBOOT reflection strings and arithmetic that resolves to exact file sizes and section boundaries), the classname-entry encoding, and the section roles. Partial for the fine structure of the three fixup tables — their presence and offsets are confirmed, but exact per-entry tuple widths were not byte-traced here.
