Name Tables (.names / .filenames / dirlist)

Three tiny, near-plaintext formats provide the engine with name → index lookups. They are not really "track" formats, but they sit beside level data and are grouped here. All three are just lists of strings; the only difference is how the count is encoded and how the entries are separated.

Name tables
CategoryString / lookup tables
Members.names, .filenames, dirlist
EncodingASCII / 8-bit, NUL- or space-separated
Endiannessn/a (text)
StatusConfirmed

.names — counted NUL-terminated list

Confirmed A single leading ASCII count digit, a 0x00 separator, then that many NUL-terminated strings. Used for short fixed enumerations — most commonly LightRigs/LightRigNames.names, which names a level's time-of-day light rigs in the order the light-rig array indexes them.

OffsetLengthTypeNameDescription
0x00x1charcountDigitASCII digit giving the entry count (e.g. '3' = 3, '5' = 5).
0x10x1charsep0x00 separator after the count.
0x2varcstring[N]namesN NUL-terminated ASCII strings, in index order.
$ od -A d -c Downtown/LightRigs/LightRigNames.names    # 24 bytes
0000000   3  \0   m  i  d  d  a  y \0  s  u  n  r  i  s  e \0  s  u  n  s  e  t \0
          │  │    └─ [0] ─────┘    └─ [1] ──────┘    └─ [2] ───┘
        '3' sep    3 light-rig names, NUL-terminated

$ od -A d -c docks/LightRigs/LightRigNames.names       # 43 bytes
0000000   5  \0   t  w  i  l  i  g  h  t \0  s  u  n  s  e  t \0  m  i  d  d  a  y \0 …
        '5' sep   → 5 names: twilight, sunset, midday, nighttime, sunrise
The count is a literal text digit, not a binary byte — Downtown stores 0x33 ('3') and docks stores 0x35 ('5'), each matching the number of strings that follow. The single-rig frontend model stores 0x31 ('1').
Because the count is one ASCII digit, this scheme as observed only encodes counts 0–9. No sampled .names exceeds nine entries, so whether ≥10 uses a multi-digit count or a different field is unconfirmed.

.filenames — texture-atlas name list

Confirmed A bare run of NUL-terminated ASCII strings with no count header. These live next to a UI .textures atlas (UI/HUD/Textures/HUD.filenames, CommonUI.filenames, …) and give the human-readable name of each sub-texture in the atlas, in array order — so UI/Scaleform code can look a sprite up by name and resolve it to a texture index.

OffsetLengthTypeNameDescription
0x0varcstring[]namesConsecutive NUL-terminated ASCII strings; index = position in the list. Read to EOF.
$ od -A d -t x1z UI/HUD/Textures/HUD.filenames | head -3
0000000 42 75 6d 70 65 72 43 61 6d 5f 62 61 6e 6b 65 64  >BumperCam_banked<
0000016 47 6c 6f 77 42 6c 75 65 00 42 75 6d 70 65 72 43  >GlowBlue.BumperC<
0000032 61 6d 5f 62 61 6e 6b 65 64 47 6c 6f 77 52 65 64  >am_bankedGlowRed<
          [0] "BumperCam_bankedGlowBlue"  \0  [1] "BumperCam_bankedGlowRed" …
The entry order in the .filenames list matches the sub-texture order packed into the paired .textures atlas, so the two are read together: the atlas supplies pixels, the .filenames table supplies the lookup keys.

dirlist — directory manifest

Confirmed A bare run of NUL-terminated relative paths naming the contents of the folder it lives in — structurally identical to .filenames (no count header, read to EOF), just used for directories rather than atlas sub-textures. The loader uses it to enumerate a directory without a real filesystem scan (the engine has matching Find files matching / Find directories matching / Reading log strings). Seen at Environments/Levels/<Level>/ColorCubes/dirlist, listing the available time-of-day colour-cube subfolders.

Entries are separated by 0x00, not spaces. Piping the file through cat makes the NUL bytes look like spaces in a terminal, which is misleading — a byte-accurate dump shows the real terminators.
$ od -A d -c Downtown/ColorCubes/dirlist           # 49 bytes
0000000  A  f  t  e  r  n  o  o  n  \  d  e  f  a  u  l  t \0
0000018  S  u  n  r  i  s  e  \  d  e  f  a  u  l  t \0 S  u
0000036  n  s  e  t  \  d  e  f  a  u  l  t \0
          → 3 entries: Afternoon\default, Sunrise\default, Sunset\default

$ od -A d -c docks/ColorCubes/dirlist              # 28 bytes
0000000  c  r  a  s  h \0 d  e  f  a  u  l  t \0 N  i  g  h  t  \  d  e  f  a  u  l  t \0
          → 3 entries: crash, default, Night\default  (entries need not be "X\default")
OffsetLengthTypeNameDescription
0x0varcstring[]entriesConsecutive NUL-terminated relative paths (Windows-style \ separators); index = position. No count, file ends after the last entry's NUL. Read to EOF.
Paths use back-slashes and point at sibling subfolders/files (each holding a .textures colour-cube). Parse by splitting on the 0x00 bytes — exactly the same way you'd walk a .filenames list.
