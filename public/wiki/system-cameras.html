Cameras & Camera Director

Split/Second's whole identity is the camera. The action-movie framing — slamming into a crash, whip-panning onto an exploding building, the slow-mo rewind on a takedown — is driven by a single data-driven subsystem the engine calls the CameraDirector (eboot class cCameraController). It owns a library of reusable cameras, each built from a stack of behaviors, and a sequencer that stitches them into multi-shot cinematic modes and sequences triggered by gameplay events. Almost everything is plaintext XML + .params, so the configured values quoted below are read straight from the devkit data.

Data layout under Deferred/Cameras

The camera data tree splits a small set of common, always-resident cameras from per-mode and per-event overrides:

PathContents
Cameras/CommonCameras/The global camera library: Cameras.xml, Cameras.params, CameraGroups.xml, CameraSplines.xml. Loaded with BatchedLoad="true".
Cameras/Params/The director's tuning files: CameraStyleConfig.params, CrashMode.params, PausedMode.params, OnlineIntroMode.params, TakeDownCameraMode.params, ScreenTransitionTimes.params.
Cameras/GameModes/<Mode>/<Event>/Per-gamemode, per-event camera sets (Race, Elimination, Demolition, Nemesis, Chopper, TimeTrial, CarCapture1–4, Online). Each has its own four-file camera bundle + an event .params (e.g. StartLine_P, FinishLine_P, Intro_P, Outro_P).
Cameras/UtilCameras/Debug/dev cameras: DefaultFreeCamera, DefaultTweakCamera, DefaultMousetrapCamera — the QA build's flycam toolkit.
Environments/Levels/<Level>/Cameras/Per-level cameras, plus PowerplayCameras.xml, PowerplayCameraSubjects.xml and the TakedownSequences/ folder of named powerplay cutscenes.
The _P suffix that recurs everywhere (StartLine_P, PA01_Collapsing_Building_P) is the studio's "P = production / proper" tag used across the data, distinguishing finalised assets from work-in-progress variants.

Cameras, behaviors & the .params twin

A camera is declared in Cameras.xml as a typed node with a list of behaviors. The XML carries only identity and structure; the actual numbers live in a sibling Cameras.params, keyed by the same timestamp-style Name/Data id (e.g. 1229362827.9.params). The FriendlyName attribute is the human label the director references.

<Camera Type="FixedToWorldCamera" Name="1260818036.-66344278"
        Data="1260818036.-66344278.params" FriendlyName="CrashFrontCamPlayer">
   <Behavior Type="LookAtBehavior"     Name="1260818053.-66344277"  Data="…params"/>
   <Behavior Type="AutoZoomBehavior"   Name="1260890696.716593860"  Data="…params"/>
   <Behavior Type="ShakeV2Behavior"    Name="1260891292.716593861"  Data="…params"/>
</Camera>

Camera types

Three base camera types appear in the data and in the eboot (cFixedToSubjectCameraData, cFixedToWorldCameraData, cFreeCameraData). In CommonCameras the split is 42 FixedToSubjectCamera / 15 FixedToWorldCamera / 1 FreeCamera:

TypeBehaviour
FixedToSubjectCameraRides relative to a subject (the car). Used for onboard/gyro/chase rigs.
FixedToWorldCameraPinned in world space; the car drives through frame. Crash & trackside cams.
FreeCameraFree-flying — pause-menu intro flythroughs, debug flycam.

Behavior stack

Behaviors are mix-and-match modifiers layered onto a camera. Each has its own cXxxBehaviorData loader in the engine. The base transform comes from a .params block of position/quaternion/FOV — note the game tunes separate FOVs for 16:9 and 4:3:

Cameras/Data/1263396175.123916999.params:
    Position.x = 0.0;  Position.y = 0.0;  Position.z = 0.0;
    Orientation.qx = 0.0;  Orientation.qy = 0.7071068;
    Orientation.qz = 0.0;  Orientation.qs = 0.7071068;
    FieldOfView16To9 = 32.841499 (0.0, 180.0);
    FieldOfView4To3  = 37.5      (0.0, 180.0);
    UseSetPosition = true;  UseSetOrientation = false;
BehaviorRole
LookAtBehavior / OrbitLookAtBehavior / HumanLookAtBehaviorAim the camera at the subject; orbit and "human operator" hand-held variants.
TrackHeightBehaviorRaycasts to the ground (MinHeight 0.5, MaxHeight 20.0, cast from +1.0 to −20.0) to keep the cam above terrain.
AutoZoomBehavior / HumanAutoZoomBehavior / RandomZoomBehaviorDynamic FOV — keep the subject framed / add operator jitter / random zoom for heli shots.
DampingBehaviorSmooths position/orientation tracking.
ShakeBehavior / ShakeV2Behavior / ShakeOnAnyCarProximityBehaviorCamera shake — impact, proximity, and a v2 model.
FollowSplineBehaviorDrives the camera along a CameraSplines.xml path (pause-menu fly-bys).
ClipPlaneBehaviorAdjusts near/far clip per shot.
RaiseUpMultiCarBehaviorPulls back/up to keep several cars in frame.
OffsetToSubjectBehavior / LinearMovementBehavior / DeltaTimeBehaviorPositional offsets, dolly moves, time-scrub control.
Post-FX behaviors — these let a camera drive screen post-processing directly: DOFBehavior/DOFLinearBehavior (depth of field), SSMotionBlurBehavior, VignetteBehavior, BloomBehavior/BloomLinearBehavior, BlurLinearBehavior, FadeLinearBehavior, GradientBehavior, ToneMapperBehavior, ColorCubeBehavior (drives the per-area colour grade — see Post-Processing DB and Tracks & Levels).

Camera splines

CameraSplines.xml stores named spline paths used by FollowSplineBehavior. Each <Spline> has an id, a total Length, a base Transform (3×3 rotation + position row), and a list of <Control_Point>s, each with its own 4-row transform matrix and a SegmentLength:

<Spline ID="2639553593" Length="9.9252911">
   <Control_Point ID="2125945358" SegmentLength="3.3305974">
      <Transform>
         <Row3 X="-0.0075565" Y="1.3200014" Z="-5.6494946"/>
      </Transform>
   </Control_Point> …

The eboot resident-list confirms a frontend variant at UI/Frontend/Cameras/CameraSplines.xml used for menu camera moves.

Cinematic sequences: the shot model

The director's signature feature is the shot sequence. A sequence is an ordered list of ShotN entries; each shot names a camera style and carries three criteria blocks that the director evaluates frame-to-frame: Cut (when to advance), Rewind_Time (jump the replay clock relative to impact), and Slow_Motion. This is what produces the rewind-and-slow-mo replay on every big hit.

/CameraDirector/CameraModes/CrashMode/Head-on/Shot2:
    '1a; Primary Camera Style' = 'Low Down Crash Point';
    '4a; Enabled' = true;
/CameraDirector/CameraModes/CrashMode/Head-on/Shot2/Cut_Criteria:
    '1a; Cut After Specified Time' = true;
    '1b; Time To Cut' = 0.6;
/CameraDirector/CameraModes/CrashMode/Head-on/Shot2/Rewind_Time_Criteria:
    '1a; Use Rewind Time' = true;
    '2a; Time Offset From Major Event (Impact)' = -0.25;

Camera styles (CameraStyleConfig.params)

The shot's Primary Camera Style resolves to a style defined under /CameraDirector/CameraStyles/. Styles are procedural framing rules (distances, heights, velocity prediction) rather than fixed cameras. Some carry a full multi-criteria evaluation block (occlusion, screen-size, rule-of-thirds weighting) used to score candidate framings. Real styles present in the data:

The director also exposes the eight evaluation axes in CameraDirector/Evaluation/DebugText/: Rule Of Thirds · ScreenSize · Direction · ScreenPosition · LOS (line-of-sight) · MultiCar · CamVsGround · BackgroundObjects — the heuristics it scores framings against.

Powerplay sequences & per-level takedowns

Big environmental powerplays (a collapsing building, the helicopter tunnel attack) get bespoke, hand-authored camera cutscenes. These are driven by the PowerplaySequenceDirector (eboot class) and stored per level. Each level's Cameras/PowerplayCameras.xml lists the powerplay events that have a custom camera, e.g. for Downtown: PA01_Collapsing_Building_P, PA03_Heli_tunnel_Track_A_P, PP09_CraneGirderDrop_1/2, PP10_PierExplode_TimeSq, PP12_road_explode_P (14 entries). The matching cutscene definitions sit in Cameras/TakedownSequences/ (24 .params files for Downtown).

A takedown sequence extends the shot model up to 10 shots and adds a Debug block with the trigger name and a respot definition (where to teleport the pack after the event — see Difficulty & Race Director):

/CameraDirector/TakeDownSequences/pa03_heli_tunnel_track_a_p/Debug:
    '1a; Powerplay Trigger Name' = '22: PA03_Heli_tunnel_Track_A_P';
    '1b; Powerplay Trigger Time' = 1.4;
    '2a; Respot Position' = (-39.0, -7.0085, 173.178);
    '2c; Respot Speed'      = 100.0;
    '2d; Respot Num Players' = 6;
/CameraDirector/TakeDownSequences/pa03_heli_tunnel_track_a_p/Shot1:
    '1a; Primary Camera Style' = 'Powerplay Detonation';

Shots reference cameras by name ('Select By Name' + a friendly id like 22: PA03_Heli_tunnel_Track_A_P_CAM_2) so artists can place exact cinematic cameras in the level and call them from the sequence. The director can rewind from the most recent rewind marker (Use Most Recent Rewind Time Marker) to land the cut right on the explosion. See Powerplay System and Powerplays format.

Director conditions & debug

The QA build exposes a deep debug menu. The director gates when it may grab a cinematic camera through CameraDirector/CameraConditions/:

ConditionMeaning
Use Drifting Or Jumping Conditiononly cut on stylish state
Use Min Time Between Takedowns Conditionthrottle (paired with Min Time Between Takeout Cameras)
Use Can Trigger Other Powerplay Conditiondon't steal the camera from a bigger event
Use Victim To Close / In Loaded Sector / Powerplay Is Most Recentspatial & relevance gates
Use Slow Mo On Takedown / Use Pause On Takedownglobal slow-mo & freeze toggles

Other notable debug toggles: Disable Sequences, Disable Takedown Sequences, Disable AI Control, Test Bullet Time, Test New Crash Sequence, an Override/ free-position rig, an orthographic mode (Debug/Ortho), and Letter Box. Screen transition timing between modes lives in ScreenTransitionTimes.params (3 active transitions: 0.366 / 0.1 / 0.366 s). The pause flythrough (PausedMode.params) sequences Left/Center/Right intro splines then cycles Bonnet→Door→Roof→Boot showcase cams at 4 s each.

Because every camera, behavior, style, group and sequence is plaintext, the configured values here are read directly off disk — confidence is Confirmed for the structure and the quoted numbers. What's Theory is the exact runtime scoring math the eight evaluation axes feed into; only the inputs and weights are visible in data.
