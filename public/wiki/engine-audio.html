Audio Engine

Split/Second's audio runs through LLAudio2 — Black Rock Studio's in-house wrapper around FMOD Ex 4.28.00 and its Event System. LLAudio2 adds a data-driven mixing layer (the DynamicMixer), reverb-zone management, pooled event and sound-bank managers, and a completely bespoke granular engine synthesizer called Tonic that generates car engine audio in real time. Nearly all of this is configured from plaintext XML on disk, so the values below are quoted directly from the game data.

The audio system was built in its own branch (C:/Projects/SplitSecond/ss_audio_dev/Code/contrib/FmodEx/42800/…). The 42800 directory name is the FMOD Ex version — 4.28.00. The wrapper proper lives in the main tree at c:/Projects/proj_1/ws_0/Code/libs/Audio/Wrappers/LLAudio2/.

Architecture: LLAudio2 over FMOD Ex

Gameplay code never touches FMOD directly. It talks to LLAudio2, which manages a set of sub-systems, each updated once per frame from LLAudio2::Update. The update order is visible in the profiler markers:

OrderSub-updateManager / source file
1EventSystem_UpdateFMOD Event System tick
2EventManager_PreUpdate → EventManager_UpdateEventManager/EventManager.cpp
3GroupManager_UpdateGroupManager/GroupManager.cpp
4LiveUpdateManager_UpdateFMOD Designer live auditioning
5ReverbManager_UpdateReverbManager/ReverbManager.cpp
6NetworkManager_UpdateNetwork audio link (see below)

Underneath these sit four pools that own the FMOD handles and recycle them to keep allocation off the hot path:

  • SoundBankManager — loads .fsb banks into fixed-size slots (SoundBankSlot.cpp, SoundBankPool.cpp). Logs SoundBank [%s] is larger than slot size [%d > %d] when a bank overflows its slot.
  • EventManager — owns EventPool / EventInstance / EventChannels and streams event data via EventMemoryStream. Tracks live events under /EventManager/PlayingEvents/%s/{State,ChannelsPlaying,PositionMS,LengthMS,Audibility}. When it runs out it "steals" the least-audible instance (EventInstance [%s] Stolen).
  • GroupManager — the 15 mixer channel groups (volume / pitch / reverb send / DSP slots), driven by the DynamicMixer.
  • DSPPool — recycles FMOD & custom DSP effect instances into the per-group slots (DSPPool.cpp).
A custom FMOD memory allocator and file-system callbacks (Initialise/MemoryCallbacks.cpp, FileSystemCallbacks.cpp) hand FMOD a dedicated heap so it never calls the OS allocator directly — the audio system reports /AudioSystem/Allocator/%s/{Size,Used,Free,Alloc} in the debug HUD.

On-disk data

The Deferred/Audio/ tree holds 73 .fsb banks and 73 matching .fev event projects (one .fev per .fsb), plus the XML config and the Tonic synth data. Verified magic bytes:

Most folders carry a parallel DEBUG_*.fsb/.fev pair (e.g. DEBUG_Collision, DEBUG_Surface, DEBUG_particles) for QA tone testing — a hallmark of this devkit build. The frontend loads locale banks lazily; the EBOOT marks them (noassert)|Audio/Locales/%s/%s.fev so a missing locale is non-fatal.
The engine supports localized voice-over: the EBOOT contains per-language speech paths Audio/Speech/{English,Dutch,French,German,Italian,Spanish}/Speech.fsb. This particular devkit build, however, ships only a single flat Audio/Speech/Speech.fsb (~2 MB) with no language sub-folders, and the EBOOT also carries a fallback (noassert)|Audio/Speech/Speech path. So the six-language layout is the retail/runtime scheme, not what is present on disk here.

DynamicMixer

The DynamicMixer (libs/Audio/Controllers/DynamicMixer/) is the heart of the mix. It is configured entirely by Audio/XML/DynamicMixer.xml (110 KB of XML) and applies priority-ordered stacks and transient presets on top of the static channel groups. It mixes 15 audio groups:

Per-group DSP chains

Each group has 7 DSP slots (slot 7 is reserved for BlackRock_VolumeMeter). The chains are declared in the <AudioGroups> block. Real examples from DynamicMixer.xml:

<Music>       FMod_Lowpass, FMod_Highpass, …, BlackRock_FrequencyFilter, BlackRock_VolumeMeter
<PlayerEngine> FMod_Lowpass, …, BlackRock_FrequencyMeter, BlackRock_VolumeMeter
<Powerplays>  FMod_Lowpass, BlackRock_Flanger, …, BlackRock_VolumeMeter
<Master>      (6× Empty), BlackRock_VolumeMeter

The DSP palette mixes stock FMOD effects with custom Black Rock ones. The FMOD effects compiled in include: Lowpass, Highpass, Compressor, Distortion, SFXReverb, ParamEQ, Lowpass Simple, ITLowpass, Echo, ITEcho, Flange, Chorus, Normalise, PitchShift and Oscillator. Custom DSPs seen in the XML are BlackRock_VolumeMeter, BlackRock_FrequencyFilter, BlackRock_FrequencyMeter, BlackRock_Flanger and BlackRock_Reverb.

Stacks (global mix states)

Stacks are priority-ordered global mood layers — only the relevant ones blend in, each with a <Transition Time="…" Curve="…"/>. The <Stacks> block holds eight of them: Frontend (priority 10), Game (priority 0), GamePaused / GameResults (7), GameCrashCam / GameRubberNeckCam / GamePowerPlayTakeDownCam (5) and GameHudSlowDownTime (3). When in the frontend, for instance, the mixer ducks almost everything to silence and leaves Music up:

<Frontend Priority="10">
  <Music>       Volume = -10 dB </Music>
  <HUD>         Volume = -14 dB </HUD>
  <PlayerEngine, CPUEngine, Swooshes, Surfaces, …> Volume = -60 dB (effectively off)
</Frontend>

In-world acoustics and one-off events are not stacks but presets (see below), e.g. TUNNEL_ENTER/TUNNEL (filter the music to 1857 Hz on its low-pass Slot2 and duck ambience −10 dB), LANDING_PLANE, Tonic (I3DL2 reverb on PlayerEngine), and a ReverbPreset preset that selects the named DangerZone reverb (Blend="1") during powerplays.

Presets (transient one-shots)

Presets are short-lived ducking/focus reactions to game events, tagged with one of three named priorities (HIGH=0, MEDIUM=128, LOW=256; lower wins). Most use MEDIUM, with the loud stingers on HIGH. Real preset names from the file:

  • SOLO_HYPERREAL, FOCUS_HYPERREAL, FOCUS_HYPERREAL2 — duck everything but the hyperreal layer for a cinematic moment.
  • FOCUS_RUMBLE, FOCUS_POWERPLAY, POWERPLAY_GENERIC_EXPLOSION, POWERPLAY_TRUCK_HYPERREAL — powerplay stingers.
  • EXPLOSION, DEBRIS — impact ducking, triggered from the particle system.

The particle audio bindings in Audio/XML/Particles.xml hook particles to FMOD events and to these mixer presets. For example the explosion particle plays particles/particles/explosion and fires the EXPLOSION mix preset; fire (hyperreal) fires FIRE; debris fires DEBRIS — each with a mix_preset_speed_mix so the ducking scales with car speed.

Reverb & DirectionalReverb

LLAudio2 has two reverb managers: ReverbManager.cpp and DirectionalReverbManager.cpp. Reverb is placed in the world as ReverbArea volumes (Sphere / Box) with a ReverbPreset, a Transition and a Blend; the runtime crossfades presets as the listener moves between zones. The full I3DL2 reverb parameter set is supported (EnvSize, EnvDiffusion, RoomHF/RoomLF, DecayTime, DecayHFRatio, Reflections, ReverbDelay, EchoTime/Depth, ModulationTime/Depth, AirAbsorptionHF, Density, Diffusion, …).

Per-level reverb is authored in Audio/Locales/<level>/<level>_Audio_Reverb.xml as a list of AudioTrigger box volumes. Each names a reverb type and a wet mix, e.g. from downtown_Audio_Reverb.xml:

Trigger labelreverb_typereverb_wet_mix

Triggers also carry a dynamic_mixer_preset hook, a 3-axis falloff and a force_fmod_settings flag, so the same authored volume can drive both reverb and a mixer state. The PatchBay.xml maps live game parameters (e.g. Player Speed (0-200), Trigger Proximity (0-1)) onto FMOD event parameters through clamp/scale/response rows — this is how speed-reactive sounds like Whoosh and Triggerzone are wired.

Tonic — the granular engine synthesizer

The most distinctive part of the audio engine is Tonic (Audio::Tonic, runs as a SPURS job JobTonic with its own TonicHeap). Rather than looping recorded engine samples, Tonic synthesizes each car's engine in real time using granular synthesis (SettingsUseTonicInput, SettingsTonicGrainWindow). It is driven by the live physics state and a per-car state engine defined in XML.

The compiled .tonic files are big-endian binaries. The header of Hypercar_03.tonic begins with what look like count fields and a sample rate (0x0000AC44 = 44100 Hz); the .tonic2 twin uses a lower rate (0x00005208 = 21000), suggesting two quality tiers of grain data:

What the state engine controls

The _StateEngine XML is remarkably detailed tuning — far beyond a normal sample set. Reading Engine_States.xml (FileVersion 3.00) shows Tonic models a complete drivetrain:

  • Aggression bands — up to 4 bands (SettingsAggressionBandCount=4) that blend exhaust/engine/thumper/boost volumes and the DSP filter settings by how hard the engine is working (e.g. SettingsExhaustVolume_1..4 = 0.6 → 1.0).
  • A DSP voice chain per band: low-pass (SettingsLowPassFreq_=200), band-pass (SettingsBandPassFreq_=1400, with Wet/Q per band) and saturation (SettingsSaturationAmount_/Mix_).
  • Simulated gearbox — SettingGearCount=4 with per-gear RPM entry/exit ranges, time-in-gear randomisation, speed hints and per-gear aggression scaling (e.g. 1st gear RpmHighExit=5900, SpeedHint=30).
  • One-shots — Pop/backfire, gear up/down engage & exit, enter/exit boost and a 16-step Thumper sequencer, each with ducker gain/release and L/R/LFE speaker sends.
  • Per-camera mixes — separate exhaust/engine/oneshot/rear/LPF/band-pass/saturation scalers for Normal, Bumper and Bonnet cameras (_Camera_1/2/3), so the engine sounds different from each viewpoint.
  • Rev limiter, skid, lost-gear sub-states with their own RPM thresholds and timings.
Because Tonic is parametric, the AI cars and the player share the same synth; the PlayerEngine and CPUEngine mixer groups simply route different Tonic instances. The Tonic mixer stack even applies its own I3DL2 reverb to the player engine (Decay_time=1.5, Reflect_delay=0.01, Reverb_delay=0.015).

Live auditioning & debug

This devkit build keeps the full FMOD Designer live-update path. LLAudio2 exposes a network audio link (Audio/LLAudio/Network/{NetworkAddress,NetworkPort,Connect} and a NetworkManager_Update) so a sound designer's host PC can audition events on the running console; the oscpack (OSC/UDP) library is linked for this kind of tooling traffic. A rich debug HUD is wired to the debug-menu under Audio/DebugHUD/DynamicMixer/… (panel layout, per-DSP name colours, the active preset/instance/slot lists) and Audio/Tonic/… (car picker, multichannel speaker gains, grain window, "Playback Debug Capture" that dumps a WAV/RAW of the synth output). There is also a recorded command stream for deterministic replay: Audio/Replay/audio_commands_recording.bin.

A note on .lip files

Despite the nav summary, the .lip files in this build are not voice lip-sync. The only .lip files on disk live under Environments/Levels/<level>/Event/RACING/TrackLogic/<A-D>/Track.lip and are big-endian offset-table binaries holding track-logic / timeline data, not audio. See Lip Sync (.lip) for the byte layout. Voice-over ships as ordinary Speech.fsb banks.
