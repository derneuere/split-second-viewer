Vehicles

Every drivable car in Split/Second is a data-driven bundle of loose files rooted under Deferred/Vehicles/. A single master roster — VehicleDatabase.xml — names each car, points it at its physics, camera and audio data, and supplies the frontend "balance" stats. Each car then owns a folder of art and physics assets (model, Havok collision, deformation model, breakable dynamic parts), while its handling lives in three parallel .params trees under Vehicles/Handling/. This page documents the roster, the file bundle, the paint/decal system and the damage/deformation pipeline, all sourced from the devkit build's plaintext data; for how the cars fit into the wider runtime see the engine architecture overview.

Vehicle classes

Cars are grouped into five families, distinguished by ID prefix in VehicleDatabase.xml and by the shared chassis they sit on. Black Rock framed the three core classes around the famous "rock–paper–scissors" balance — Trucks are heavy and strong, Supercars are fast and fragile, Musclecars sit in the middle with the best drift. The eboot exposes the class names as runtime constants (NemesisTruck, Supercar, Supercar_player, plus the boss modes HELICOPTER and DEMOLITION).

The Prototype chassis is used by exactly one car: Unique_04, whose stat line in the roster is the most extreme in the game — Speed="10" Drift="1" Strength="1" and PlayerSkillLevel="1.0". Musclecar_Drone is the pack-filler AI car (it borrows Musclecar_01's body/physics but its own low-skill AI profile); the dedicated Truck_Demolition and the DEMOLITION/HELICOPTER tags drive the boss Nemesis & Heli modes.

VehicleDatabase.xml — the master roster

Vehicles/VehicleDatabase.xml is the single source of truth for which cars exist. It is a flat list of <Car> elements under a <Root FileVersion="1.0">. Each element wires the car to its handling, cameras, audio and frontend presentation. The attribute names are confirmed against the eboot's XML parser strings (HumanPhysicsFile, AIPhysicsFile, ChaseCamFile, BumperCamFile, BonnetCamFile, AudioFile, DevonName, Strength, ActionScriptImage, DCBModifier, PlayerSkillLevel, Locked, Player).

<Car ID="Musclecar_01" HumanPhysicsFile="Musclecar_01" AIPhysicsFile="AIMusclecar_01"
     ChaseCamFile="MuscleChase" BumperCamFile="DefaultBumper" BonnetCamFile="DefaultBonnet"
     Lives="3" ActionScriptimage="11" Locked="false" DCBModifier="0"
     PlayerSkillLevel="0.2" AudioFile="Truck_03" Speed="2" Drift="4" Strength="4"/>

Shared chassis

Bodies are visual shells; the rolling chassis underneath is shared. ChassisName.txt in each body folder names which of the seven entries in Vehicles/Chassis/ to mount. A chassis folder (e.g. Coupe/) holds its own .model, .deform, shaders and textures. Verified mapping for every body:

The Nemesis truck breaks the standard bundle. Bodies/NemesisTruck01/ has no .phys, .deform, .mainColl or _dynamicParts.params. Instead it ships an .rdata + .xml pair (eboot: Vehicles/Bodies/%s/%s.rData, %s.xml) and its physics live in GameLogic/Nemesis/NemesisTruckPhysics.params with a dedicated debug tree (MGF/NemesisTruckPhysics/…) and class cNemesisTruckAIComponent / NemesisTruckInstance. Its articulated trailers are separate bodies (NemTruck1Trai1, NemTruck1Trai2) joined by ragdoll-style constraints. See Nemesis & Heli Modes.

Paint & colour palettes

Vehicles/VehicleColors.params (≈5,160 lines) defines the selectable paint jobs. It is one .params file with a group per palette under /Vehicles/ColorPalettes/<ID>/<NN>. Forty cars are present (every roster entry except the two Nemesis trailers), with up to nine palettes each (00–08; Musclecar_01 has nine). Each palette is a three-layer car-paint model — a base Paint Color, a Flake Color (metallic sparkle) and a Lacquer Color (clear-coat tint) — each an RGB triple in the 0–1 range, consumed by the vehicle paint shader via the _bodyPaint texture mask.

/Vehicles/ColorPalettes/Musclecar_01/01:
    '@Paint Color'   = ':Paint Color:color:'Paint Color_r' 'Paint Color_g' 'Paint Color_b' ';
    'Paint Color_r'  = 0.4117647 (0.0, 1.0);   ; a deep red base
    'Paint Color_g'  = 0.0666666 (0.0, 1.0);
    'Paint Color_b'  = 0.0666666 (0.0, 1.0);
    'Flake Color_r'  = 0.92549  (0.0, 1.0);    ; gold-ish metallic flake
    'Flake Color_g'  = 0.38431  (0.0, 1.0);
    'Flake Color_b'  = 0.05882  (0.0, 1.0);
    'Lacquer Color_r'= 0.074509 (0.0, 1.0);    ; near-black clear coat
    'Lacquer Color_g'= 0.023529 (0.0, 1.0);
    'Lacquer Color_b'= 0.023529 (0.0, 1.0);

The '@…' lines are editor metadata describing how the three component sliders compose into a colour swatch (the :Name:color:… form is the tuning tool's colour-picker hint).

Decals & achievement liveries

Cars also carry composited decal liveries driven by Vehicles/Compositing/AchievementsDatabase.params with the shared Stickers.textures sheet and GameCompositingInfo.params. Each achievement maps to a progression of decal textures applied to the body, e.g. Achievement_00 "Triple Trouble" uses Decal_70/Red_Dot, and Achievement_01 "Power Up" cycles Decal_141/Green_Dot/Red_Dot across its stages. The runtime manager is the Vehicle Compositor (eboot directory /VehicleCompositor/Achievements).

Handling & physics configuration

A car's drive feel is not in the Havok .phys file alone — that supplies the rigid body and collision; the tuning lives in three parallel .params trees under Vehicles/Handling/, all selected by the HumanPhysicsFile/AIPhysicsFile keys from the database (loader path Vehicles/Handling/Configs/):

TreeContentsGranularity
Handling/Configs/The full driving model: engine, brakes, drift, steering, suspension, weight transfer, collision response, crash states, nudge behaviour, per-surface frictionOne file per car (+ AI* variants, Default.params)
Handling/Havok/The editor "star rating" stats + ride heights, CoM and suspension hardpoints fed into the Havok rigOne file per car
Handling/Tuning/DefaultTuning.params: the engine/gearbox upgrade matrix (Engine 0–4 × Level 0–3)Shared default

Most parameters are response curves stored as 10- or 20-entry arrays plus a scale and a max, e.g. drift counter-steering, grip-vs-slip, suspension force-vs-extension and engine AccelerationVsSpeed. There are also CrashState/* blocks (HeadOn, ForwardFlip, BackwardFlip, YawThenRoll, Drone takedown, Crushed…) that script the cinematic post-crash motion, and a full HandlingModifiers/SurfaceMaterials/* table (cobbles, concrete, dirt, metal, sand, track, water, wood, wet variants) giving per-surface friction/drag — the hook for the wet-track and dirt sections of each level.

The upgrade matrix (DefaultTuning)

Handling/Tuning/DefaultTuning.params defines five engine families (Engine0–Engine4), each with four upgrade Level0–Level3. Every level carries an AccelerationStatBase/SpeedStatBase, a BoostMaximumSpeed, a DriveTorqueVsRpm curve, a ChangeUpRpm, gear ratios, and three sprocket presets (Long/Short/Standard) plus three brake presets. The acceleration base climbs with level — for Engine0: 8 → 18 → 28 → 38 — and boost top speed rises 92 → 96 → 102 → 109 mph, illustrating the season-mode progression.

Dynamic parts (breakables)

<ID>_dynamicParts.params defines every part that can break off or flex on impact, as a set of Havok rigid bodies and constraints hung off the body. For Musclecar_01 there are 44 rigid bodies and 16 constraints across these named parts:

BODY · BOOT · BUMPER_F · CHASSIS_CABIN/FRONT/REAR · DISC_{F,B}{L,R} · DOOR_{L,R}(+_MIRROR,_WINDOW) · ENGINE · EXHAUST_{L,R} · HOOD · HUB_{F,B}{L,R} · >LIGHT_{F,B}{L,R} · RADIATOR · SPOILER_B · TYRE_{F,B}{L,R} · WHEEL_{F,B}{L,R} · WING_F{L,R} · WISHBONE_{F,B}{L,R} (44 named parts total)

Every part has a Rigid body block (Mass, Friction 0.5, Restitution 0.4, Angular damping 0.05, Allowed penetration −1.0); hinged parts add a Constraint1 (single-axis: min/max angle, friction torque, Tau factor) and doors add a two-axis Constraint2. The masses encode what the part "feels" like when it tears loose:

/Vehicles/DynamicParts/Musclecar_01/BODY/Rigid body:
    Mass = 1000.0 (0.1, 1000.0);     ; the shell
/Vehicles/DynamicParts/Musclecar_01/ENGINE/Rigid body:
    Mass = 200.0;
/Vehicles/DynamicParts/Musclecar_01/DOOR_L/Rigid body:
    Mass = 25.0;
/Vehicles/DynamicParts/Musclecar_01/DOOR_L/Constraint1:
    'Maximum angle' = 85.0 (-180.0, 180.0);  'Minimum angle' = 0.0;  'Tau factor' = 0.5;

The runtime managers are DynamicParts RB Data Mgr and DynamicParts Constr Data Mgr; the eboot also exposes a dev/crash sandbox (Vehicles/DynamicParts/Dev/Reset, Vehicles/DynamicParts/Crash activation/*). The part list maps onto the .parts hierarchy (a custom big-endian binary holding part IDs and 4×4 attach transforms) and the .deform mesh-deformation model — together they make a Split/Second car shed panels and crumple. Unique_04 (Prototype) and the Nemesis truck have no _dynamicParts.params, so they do not shed standard panels.

Surface damage & deformation

Two systems progressively dirty and crumple the car:

Surface damage planes

Bodies/<ID>/SurfaceDamage.params defines five oriented damage planes (Front/Back/Left/Right/Upper, each with azimuth, zenith and distance) bounding where dirt and scrape marks accumulate on the _damageMap texture. For Musclecar_01 the front/back planes sit 1.9 m out, the sides 0.75 m, and the upper plane 0.825 m at an 86.5° zenith (there is no lower/underside plane):

/Vehicles/SurfaceDamage/Musclecar_01/DamagePlanes:
    'Front - azimuth angle' = 0.0;   'Front - distance' = 1.8999999;
    'Left - azimuth angle'  = 90.0;  'Left - distance'  = 0.75;
    'Upper - zenith angle'  = 86.5;  'Upper - distance' = 0.8249999;

The shared rates live in Vehicles/Generic/SurfaceDamageCommon.params, split into Standard (gradual driving wear) and Crash (impact) profiles for both Dirt and Scrapes — e.g. crash dirt has an Increase rate of 0.1 up to a Maximum level of 0.8, while standard scrapes only add 0.001875 per contact. The eboot exposes the Surface damage database and a Vehicles/Surface damage render vehicle dev toggle.

Deformation (.deform)

The geometric crumpling is the .deform file — a custom DFM\x02 container (magic 44 46 4D 02, big-endian, ~31 KB for a car). It stores the vertex-deformation model the physics drives when the car is hit. Unlike .phys/.mainColl it is not a Havok packfile, so it is Black Rock's own format.

How a car loads (data flow)

Putting it together, when a car is spawned the engine:

  1. Reads the <Car> entry from VehicleDatabase.xml.
  2. Loads the visible mesh Bodies/<ID>/<ID>.model and the two wheel models, plus shaders, textures and the _bodyPaint/_damageMap layers (managers cVehicleModelManager, cWheelModelManager, cVehicleShaderInterface).
  3. Mounts the shared chassis named by ChassisName.txt.
  4. Loads the .phys rigid body + .mainColl shape, then applies the handling tree selected by HumanPhysicsFile/AIPhysicsFile.
  5. Builds the breakable rig from _dynamicParts.params + .parts + .deform.
  6. Applies a paint palette from VehicleColors.params and any composited decals.
  7. Binds cameras (Cameras.xml) and the AudioFile engine bank.
