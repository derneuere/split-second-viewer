Powerplay Info (<Level>_powerplayInfo.dat)

Every level carries one <Level>_powerplayInfo.dat in its root (Environments/Levels/<Level>/). It is a flat placement table: a counted list of records, each naming a Catnip powerplay timeline (.cat) and giving the world-space transform at which that powerplay is instanced in the level. It is the level-side index the Powerplay system reads to know which powerplays exist on a track and where they fire.

Top-level layout

Confirmed (header) The file begins with a big-endian entry count, followed by that many variable-length records. Each record is length-prefixed by its name, so the file must be walked record by record (there is no offset table).

OffsetLengthTypeNameDescription
0x00x4uint32_tentryCountNumber of powerplay placements. Downtown 0x58=88, Graveyard 0x31=49, docks 0x51=81.
0x40x4uint32_ttableHash?80 73 33 3c — a 4-byte value that precedes the first record (a CRC/hash of the table or first id). Theory
0x8varrecord[]entriesentryCount variable-length records (see below).

Record layout

Partial Each record is a length-prefixed name followed by a fixed 4×3 affine transform matrix (12 big-endian float32) and a short trailing block (two words + a per-entry id). The name length and string are certain; decoding the 12 floats of the first Downtown record yields a near-identity 3×3 rotation with the world position in the translation row, which is why the record is a TM34 matrix — not a quaternion+position pair as a first glance at the leading ≈1.0 float suggests.

OffsetLengthTypeNameDescription
+0x00x4uint32_tnameLenLength of the timeline name in bytes (e.g. 0x19=25). Confirmed
+0x4nameLenchar[]nameCatnip timeline file, e.g. pp12_road_explode_mpo.cat. Not NUL-terminated. Confirmed
+…0x30float32_t[3][4]transform4×3 (TM34) affine matrix: rows 0–2 = 3×3 orientation (near-identity in the sample), row 3 = world-space translation (x,y,z). Theory (matrix order)
+…0x8uint32_t[2]fieldsTwo trailing words — observed 00 00 00 00 then 00 00 00 01 (a flag/count?). Theory
+…0x4uint32_tidPer-entry trailing id/hash (e.g. 84 fe d8 1b), immediately preceding the next record's nameLen. Theory

The whole record is 4 + nameLen + 0x30 + 0x8 + 0x4 bytes. For record 0 that is 4 + 25 + 48 + 8 + 4 = 89 bytes (offsets 8…96 here), and the next record's nameLen (0x1C=28) lands exactly where predicted — confirming the record stride even though the matrix-row order is not yet pinned to a known in-game powerplay.
The name length cross-checks exactly: 0x19=25 = len("pp12_road_explode_mpo.cat"), and the following record's 0x1C=28 = len("pp10_pier_bld1noswap_mpr.cat"). Counting records forward from offset 8 reaches entryCount entries and lands exactly on EOF.

What the names tell us

The referenced files are Catnip powerplay timelines — the .cat assets under Powerplays//Catnip/PowerplayTimelines that script a powerplay's sequence of FX, physics and camera beats. Their naming encodes the powerplay and its variant:

TokenMeaningExample
pp##Powerplay numberpp12_road_explode
pg## / pa##Other event/scenario prefixespg07_cars_6_mpo.cat
_mpo / _mprMultiplayer-online vs multiplayer-race variant…_mpo.cat, …_mpr.cat
noswapGeometry-swap behaviour flagpp10_pier_bld1noswap_mpr.cat

The TakedownSequences .params in the level's Cameras/ folder carry matching PP##/PA## names, so a powerplay's placement (here), its timeline (the .cat), and its takedown camera are linked by the shared id.

The 4-byte word at offset 4 (80 73 33 3c) and the trailing per-entry id (84 fe d8 1b, …) are plausibly CRC32 name hashes — the title's .crcs tables use the same hashed-name lookup scheme — but this has not been confirmed by hashing the strings. The record stride is solid (the predicted nameLen of each following record lands exactly), but the order of the matrix rows and whether the two trailing words are flags vs. part of the id stay Theory until a record is matched to a known in-game powerplay position.

Relationship to the powerplay system

At load, the Powerplay system (cPowerplayManager / cPowerplayController) reads this table to instantiate each placed powerplay, binding its .cat timeline to the given transform and to the surrounding triggers / Catnip entities. The separate top-level sectordata.dat + sector_scenarios_overlay.dat are used by the Megabowles test harness to auto-generate powerplay scenarios and are unrelated to this per-level placement file.
