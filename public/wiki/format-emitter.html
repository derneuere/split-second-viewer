Emitters (.emitterControllers / .emitterSurfaces)

These two files are the heart of Split/Second's GPU particle system. A .emitterControllers file is a table of emitter controllers — each one a named spawner with a transform and a fat block of tuning floats (lifespan, velocity, LFOs, size/rotation envelopes). A .emitterSurfaces file is a triangle mesh that an emitter can scatter particles across instead of emitting from a point or volume. Both are loaded by the in-house particle manager (cSSParticleSystemManager / SScParticleSystemManager).

Where they live & how they group

Particle assets sit under each level's Particles/ tree, sorted into the engine's five particle classes. Every class is a folder containing a matching set of files (Standard.data, Standard.emitterControllers, Standard.emitterSurfaces, Standard.fxc, Standard.shaders, Standard.textures, Standard.timelineInfo):

Environments/Levels/docks/Subtracks/A/Particles/TimelineParticles/
  Standard/   Standard.*        (cStandardGameParticleDataGroup)
  Geometry/   Geometry.*        (cGeometryGameParticleDataGroup)
  Light/      Light.*           (cLightGameParticleDataGroup)
  Screen/     Screen.*          (cScreen…  "Screen Timeline Particles")
docks/Particles/TimelineParticles/Flare/   Flare.*  (cFlareGameParticleDataGroup)
docks/Particles/EnvironmentParticles/default/   Standard.*  (always-on ambient FX)

The five class names are confirmed verbatim in the EBOOT (cStandardGameParticleDataGroup, cFlareGameParticleDataGroup, cLightGameParticleDataGroup, cGeometryGameParticleDataGroup, and "Screen Timeline Particles"). Each folder's .timelineInfo twin sequences the controllers over time; its header count equals the controller count exactly (see below), so the index spaces line up 1:1.

The loader log strings tell the story: Loading '%s' particle emitter controller file... → %d bytes of standard emitter controller data, alongside Loading '%s' particle data file... and Loading '%s' particle emitter controller file.... Files are matched by the patterns %s.emitterControllers, .emittersurfaces, %s.timelineInfo and %s.data.

.emitterControllers format

Layout is: an 8-byte header, a fixed-width directory of named controllers, then one variable parameter block per controller (each a 4×4 transform + a float parameter array, terminated by an FF FF FF FF sentinel), then a short chunk trailer.

Header

OffsetLengthTypeNameDescription
0x00x4uint32_tversionAlways 0x00000005 across all 83 samples. The EBOOT has a "Particle Data Group file, '%s', version mismatch - got %d, expected %d" assert, so this is a checked version field.
0x40x4uint32_tcontrollerCountNumber of controllers. Verified: docks Standard reads 0x000007A9 = 1961, and its Standard.timelineInfo header holds the same 1961 at offset 0x4.

Controller directory (one 0x50 entry each)

Immediately after the header, controllerCount records of 80 bytes each. Verified by stepping the offsets: GUID strings appear at 0x08, 0x58, 0xA8 (stride 0x50).

OffsetLengthTypeNameDescription
+0x00x28char[40]guidNUL-padded ASCII "id" string, e.g. 1237993346.3 or 1251996606.-920208392 — a millisecond-timestamp + hash the editor assigns to each emitter instance.
+0x280x28char[40]nameNUL-padded ASCII display name, e.g. MSC_ElectricGlow, SMO_SMokeLitInside, SPA_BlueSparkS, LIT_OrangeFire. Prefixes encode the effect family (SMO=smoke, SPA=spark, DEB=debris, LIT=light, MSC=misc).

Per-controller parameter block

After the directory, one block per controller in the same order. Each block opens with a short preamble, then a row-major affine transform (4 rows of 4 floats; identity = 1.0 on the diagonal), the big tuning-float array, and is padded and capped with FF FF FF FF. In the 3-controller UI/Frontend/.../Light.emitterControllers (1608 bytes) the param region starts at 0xF8 (right after the 3×0x50 directory) and the three sentinels land at 0x2B0, 0x460 and 0x610 — i.e. block lengths 0x1B8, 0x1B0, 0x1B0. The block length is therefore not a fixed stride; treat each block as variable-length and walk it by the FF FF FF FF sentinel, or use the chunk-offset trailer at the end of the file (it lists the start offset of each block).

OffsetLengthTypeNameDescription
+0x00x40float32_t[4][4]transformLocal placement matrix. Identity in the simple files (diagonal 3F 80 00 00 = 1.0, translation row zero).
+0x40varfloat32_t[]paramsTuning array, schema below. First non-matrix floats seen are 41 20 00 00=10.0, 40 40 00 00=3.0, then three 3F 00 00 00=0.5 — i.e. the EmitterVolume / spread group.
—0x4uint32_tsentinel0xFFFFFFFF marks end of block.

Empty / stub files

Many class folders that exist for completeness ship a 48-byte stub: header 00 00 00 05 with controllerCount = 0, then only the chunk trailer. Of the 10 Flare.emitterControllers files, 9 are 48-byte stubs and only airport_test_03 carries a real (single) controller; several "Geometry" Nemesis-level files are 48-byte stubs too.

.emitterSurfaces format

An emission surface is a small triangle mesh. The loader is owned by cEmitterSurfaceDatabase; when a controller's EmitterVolumeType selects surface emission the system scatters particles across these triangles (the EBOOT warns "WARNING: Unable to find emitter surface" when the named surface is missing).

Header

A 16-byte file header, then a fixed 0x20-byte block that describes (at least) the first surface, then the per-surface vertex and triangle arrays, closed by a 12-byte chunk trailer.

OffsetLengthTypeNameDescription
0x00x4uint32_tversionAlways 0x00000001. Checked by "Emitter surfaces file, %s, version mismatch - got %d, expected %d".
0x40x4uint32_tsurfaceCountNumber of surfaces. docks Standard = 0x00000007 = 7; the small nem_downtown Standard = 1.
0x80x8uint32_t[2]hash / sizeTwo per-file words, e.g. 5B BE 29 CD + 00 00 03 DC in docks, C8 52 9B D6 + 00 00 05 EC in nem_downtown. Not a total vertex count — the value differs from the real vertex totals (see warning below); treated here as a content hash / section size.
An earlier draft labelled 0x0C as a global vertexCount = 988. That is wrong: the single-surface nem_downtown file stores 41 verts yet carries 0x000005EC (1516) at 0x0C, so 0x0C is not a vertex count. The real per-surface vertex/triangle counts live in the descriptor block below.

Surface descriptor (0x10)

A 0x20-byte block at 0x10 carries the geometry counts and the offset to the triangle data. Offsets are relative to 0x10. (For multi-surface files this describes surface 0; how the remaining surfaces are indexed is not yet fully mapped — hence the page stays Partial.)

OffsetLengthTypeNameDescription
0x100x4uint32_t—Zero in both samples.
0x140x4uint32_tvertexCountVerts in this surface. docks surface 0 = 20 (0x14); nem_downtown = 41 (0x29).
0x180x4uint32_tvertexStride?Constant 0x20 (32) in all samples.
0x1C0x4uint32_ttriangleCountTriangles in this surface. docks surface 0 = 20; nem_downtown = 26 (0x1A).
0x200x4uint32_ttriangleOffsetByte offset of the triangle array, relative to 0x10. docks = 0x160 → abs 0x170; nem_downtown = 0x2B0 → abs 0x2C0. Both check out: 0x30 + vertexCount×16 = triangleOffset.
0x240xC—padZero; vertex array begins at 0x30.

Vertex array (16-byte records)

Starts at 0x30. Each record is three float32 (XYZ) + a 4-byte pad (zero). Values are model-space (e.g. docks vertex 0 = 40 BB 22 F0=5.847, BE 89 21 B7=-0.268, BF 7A 30 C9=-0.977). Verified: 20 verts × 16 = 0x140 takes docks from 0x30 to the triangle offset 0x170 exactly.

OffsetLengthTypeNameDescription
+0x00xCfloat32_t[3]positionVertex XYZ in local space.
+0xC0x4uint32_tpadZero.

Triangle / connectivity array (32-byte records)

After the vertex array the file switches to 32-byte triangle records: a float32[3] (a unit-ish vector, almost certainly the triangle normal) + pad, followed by three uint32 vertex indices + pad. Verified in nem_downtown (1 surface, 41 verts, 26 tris): the first record at 0x2C0 is 3E BA EE 84 BF 21 EF A3 BF 2E DC C3 00000000 (normal) then 00000002 00000003 00000001 00000000 (indices 2,3,1). 26 records × 32 = 0x340 fills 0x2C0…0x600 exactly before the trailer.

OffsetLengthTypeNameDescription
+0x00xCfloat32_t[3]normalFace normal (often a clean axis like 3F 80 00 00=1.0 in axis-aligned meshes).
+0xC0x4uint32_tpadZero.
+0x100xCuint32_t[3]indicesThree vertex indices into this surface's vertex array.
+0x1C0x4uint32_tpadZero.

The file closes with a 12-byte chunk trailer (00 00 00 02 00 00 00 08 00 00 00 10 — 2 entries), the same trailer convention as .emitterControllers.

An earlier draft described 16-byte triangle records ("two float32 then three uint32"). That cannot be right — two floats plus three uint32 is 20 bytes, and the data is laid out on a 32-byte stride (a normal+pad quad followed by an indices+pad quad). The record size is 32 bytes.

Empty surface files

The overwhelmingly common case is an 8-byte stub: 00 00 00 01 00 00 00 00 (version 1, zero surfaces). Classes that emit from points/volumes only — and the vehicle particle set (Vehicles/Particles/Standard/VehicleParticles.emitterSurfaces, verified to be exactly 8 bytes) — ship this stub. Real surface files are rare: only the per-level Standard and Screen classes carry geometry.

Runtime & debugging

The particle manager exposes a live stats overlay confirming the object model: ActiveParticleCount, RenderParticleCount, ActiveSystemCount, RenderSystemCount, ActiveEmitterCount, ParticleDataCount, plus "Emitters Active: %d/%d/%d". Debug toggles live under Particles/Debug/* (system bounding boxes, render-bucket BBoxes, particle velocity draw) and Particles/Lighting/Debug/*. Particles are lit through the same deferred path as the world — see particle_shadow_map and the Lights page.

The matrix/parameter byte offsets within a parameter block are inferred from the simple files (UI Light, airport Flare) where the array is short and mostly default. In the large multi-controller files the param array is long and not every field has been individually located, so the exact float index of each named parameter above is Theory beyond the leading EmitterVolume group.
