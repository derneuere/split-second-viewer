Collision (.hkColl / .mainColl)

Split/Second stores all of its physics collision geometry as Havok 5.5 packfiles. .hkColl holds static level collision (the drivable world, walls, terrain) and .mainColl holds per-vehicle collision (the car bodies that bounce, scrape and take damage). Both share the identical Havok container header — they differ only in which Havok shape classes the data section instantiates.

Both formats are standard Havok packfiles — see Havok Packfiles for the full container header breakdown. This page focuses on what is specific to collision: the shape class hierarchy and how level vs. vehicle data differ. The two extensions are just file-naming conventions used by the loader; their bytes are produced by the same Havok serializer.

Where the files live

The engine loads these by string templates found in the EBOOT (eboot_strings.txt):

Path template (EBOOT)ExtensionRole
environment|Physics/%s.hkColl.hkCollWhole-level static collision
environment|Subtracks/%s/tl.hkColl.hkCollPer-subtrack collision ("track-local")
Resident|Vehicles/Bodies/%s/%s.mainColl.mainCollPer-vehicle collision (always resident)

A third variant, cl.hkColl ("common"), sits in Levels/<name>/Common/. In the samples examined its data section is an empty hkpPhysicsData (worldCinfo = null, systems = 0) — a placeholder container, with the real geometry living in the Physics/<name>.hkColl and Subtracks/*/tl.hkColl files. The Resident|…mainColl prefix tells the streaming system the car collision is never paged out (see Resources & Streaming).

Section table

Three section headers follow at 0x40, each a 16-byte tag plus a run of big-endian uint32 offsets (data start + fixup/boundary offsets). The tags are always, in order, __classnames__, __types__, __data__: 

000040  5f 5f 63 6c 61 73 73 6e 61 6d 65 73 5f 5f 00 00   __classnames__..
000050  00 00 00 ff 00 00 00 d0 00 00 01 50 ...            data@0xD0  end@0x150
000070  5f 5f 74 79 70 65 73 5f 5f 00 00 00 00 00 00 00   __types__.......
000080  00 00 00 ff 00 00 02 20 ...                        data@0x220
0000a0  5f 5f 64 61 74 61 5f 5f 00 00 00 00 00 00 00 00   __data__........
0000b0  00 00 00 ff 00 00 0b 70 ...                        data@0xB70
The internal field split of each section header (absolute-data-start vs. the local / global / virtual fixup offsets and the exports/imports table) is the generic Havok packfile layout — it is documented on the Havok Packfiles page and marked Partial there. Nothing about it is collision-specific.

Level collision — .hkColl

The __data__ root is an hkRootLevelContainer. For levels it carries a hkpPhysicsData plus a baked material table: one HavokMaterialExportContainer holding several HavokMaterialExport → cHavokMeshMaterial pairs (Black Rock's custom per-surface material/friction table stored alongside the geometry). This table appears in both the whole-level Physics/<name>.hkColl and the per-subtrack tl.hkColl twins — it is not subtrack-only. The collision body chain is:

ClassRole
hkpPhysicsDataTop container; holds the physics systems.
hkpPhysicsSystemNamed "EnvironmentCollision"; one rigid body.
hkpRigidBodyNamed "environment_collision", MOTION_FIXED (static). Material friction 0.5, restitution 0.4.
hkpMoppBvTreeShapeMOPP bounding-volume tree wrapping the mesh for fast broadphase queries.
hkpMoppCodeThe compiled MOPP byte-code (a large uint8 array, e.g. 32768 entries) + a quantisation offset vector.
hkpExtendedMeshShapeThe actual triangle soup, organised into hkpExtendedMeshShapeTrianglesSubpart blocks (vertex + index buffers).

This MOPP BV-tree over an extended triangle mesh is the textbook Havok pattern for large static worlds: the mesh stores the polygons, the MOPP code is a precomputed spatial index so the car-vs-world ray/shape casts stay cheap. I verified the hierarchy by reading the __data__ section of Downtown/Subtracks/A/tl.hkCollXml, where #0070 hkRootLevelContainer → #0071 hkpPhysicsData → #0079 hkpPhysicsSystem → #0080 hkpRigidBody → #0081 hkpMoppBvTreeShape → #0082 hkpMoppCode + #0083 hkpExtendedMeshShape appear in exactly that order. The big Downtown.hkColl (6.97 MB) uses the same six collision classes — confirmed by enumerating its XML twin — plus the same baked material table (HavokMaterialExportContainer + 12 HavokMaterialExport/ cHavokMeshMaterial pairs, vs. 6 in subtrack A).

Sample: the MOPP code array

The MOPP byte-code is the bulk of a level file. From tl.hkCollXml (the human-readable twin of the same bytes): 

<hkobject name="#0082" class="hkpMoppCode" signature="0x72ee59f8">
  <hkparam name="info">
    <hkparam name="offset">(-1630.96 -19.82 -323.69 9270.97)</hkparam>  <-- quantisation origin + scale
  </hkparam>
  <hkparam name="data" numelements="32768">
    40 0 182 39 0 8 38 0 255 16 143 153 3 12 0 63 ...               <-- MOPP opcodes (binary in .hkColl)
  </hkparam>
</hkobject>

In the binary .hkColl this same array is a raw run of bytes inside __data__; the .hkCollXml twin is the serializer's debug dump of it. The trailing 205 (0xCD) bytes seen at the end of the dump are uninitialised-fill padding from the exporter, not opcodes.

Vehicle collision — .mainColl

Vehicle bodies do not use a triangle mesh — they use a small set of hkpConvexVerticesShape hulls. Unlike the level files, a .mainColl contains no hkpPhysicsData, hkpPhysicsSystem or hkpRigidBody: its __data__ root is a bare hkRootLevelContainer holding a list of hkRootLevelContainerNamedVariant entries, each naming one convex hull. The rigid body, mass and handling that animate these shapes live separately in the .phys file. There is no XML twin for vehicles — only the level .hkColl files ship a .hkCollXml.

Every vehicle examined (Musclecar / Supercar / Truck / Unique, plus the Drone) carries exactly four named convex hulls, with these literal names embedded in __data__:

Hull nameShapeApparent role (Theory)
Vs_EnvironmenthkpConvexVerticesShapeHull used when colliding against the static world (.hkColl).
Vs_ObjecthkpConvexVerticesShapeHull used against dynamic props / powerplay objects.
Vs_VehiclehkpConvexVerticesShapeHull used in car-vs-car contacts.
CorehkpConvexVerticesShapeA central / inner hull (likely the chassis core volume).

The Vs_ prefix and the per-target naming strongly suggest these hulls are selected by collision target (a per-pair filter), not by damage zone. The classname table also lists the supporting hkpConvexShape, hkpSphereRepShape, hkpConvexVerticesConnectivity and hkpConvexVerticesShapeFourVectors (SIMD-packed vertices) used to build each hull.

Correction. An earlier draft of this page stated each vehicle has "8" convex hulls. That number was the raw count of the string hkpConvexVerticesShape across the whole file, which also counts the __classnames__ and __types__ reflection entries (and the substring inside hkpConvexVerticesShapeFourVectors). The __data__ section holds four actual hull instances — verified by counting hkpConvexVerticesShape occurrences past the __data__ offset (0xB70) in Musclecar / Supercar / Truck / Unique samples.

Each __classnames__ entry is { uint32 signature; uint8 tag(0x09); char[] name '\0' }; the 09 byte before every name is the Havok classname tag. The convex-hull choice is the classic Havok approach for moving rigid bodies: cheap to collide and to compute inertia, unlike the static world's triangle mesh. The four target-named hulls feed the Vehicle Physics (.phys) rigid body (mass, inertia, handling) and the Deformation (.deform) model that turns impacts into visual damage.

Quick way to tell the two apart without parsing: grep the file for class names. If you see hkpMoppBvTreeShape / hkpExtendedMeshShape it is a level .hkColl; if you see hkpConvexVerticesShape it is a vehicle .mainColl.

Confidence

Confirmed — the container header, section table, version string, the level collision chain (hkpPhysicsData → hkpRigidBody "environment_collision" → MOPP BV-tree + extended mesh, friction 0.5 / restitution 0.4, MOTION_FIXED) and the baked HavokMaterialExport table are all directly verified against the binary and the .hkCollXml twins, across small and large samples.

Partial — the vehicle .mainColl hull count (four) and their literal names (Vs_Environment, Vs_Object, Vs_Vehicle, Core) are verified from the binary, but vehicles have no XML twin, so the per-hull role is inferred from the names (Theory). The byte-exact split of each section header's fixup-offset run is generic Havok and is covered on the Havok Packfiles page.
