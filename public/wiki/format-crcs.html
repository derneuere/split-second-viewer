CRC Tables (.crcs / .tex.crcs)

A .crcs file is the simplest format in the game: a flat array of big-endian 32-bit CRCs and nothing else — no header, no magic, no count field. Each CRC is a texture-name hash that ties an asset (a level backdrop, a track-side prop, a powerplay animation) to the textures it depends on. The same 32-bit keys appear as the crc field of each .textures descriptor and in that file's C2NM name trailer, so a .crcs list is effectively a "textures required by this object" manifest.

Overview

Two naming conventions exist for the same format:

ExtensionTypical locationPurpose
X.texture.crcsEnvironments/…/Backdrop/, Environments/Objects/TL_Instances/Textures a backdrop or track-side instance references
X.tex.crcsPowerplays/Animations/…Textures a powerplay animation prop references

The eboot builds the backdrop path with environment|Backdrop/%s_backdrop.texture.crcs, confirming these are loaded per-level/per-object to resolve a texture working set before the matching .textures / .streamtex are paged in.

Structure

There is no header. The file is fileSize / 4 consecutive big-endian uint32 CRCs. This was confirmed by surveying many files: every size is a clean multiple of 4 (0, 20, 24, 32 bytes → 0, 5, 6, 8 entries) and the bytes are plainly CRC-shaped (high entropy, no ASCII, no offset patterns).

OffsetLengthTypeNameDescription
0x000x4uint32_tcrc[0]First texture-name CRC (big-endian).
0x040x4uint32_tcrc[1]Second CRC.
…0x4uint32_tcrc[N-1]Repeats to end of file; N = fileSize / 4.
An empty .texture.crcs (0 bytes, e.g. Downtown_cl.texture.crcs) is valid and means "this object references no streamed textures of its own."

Shared CRCs & the name link

The same CRC recurs across unrelated objects when they share a texture. 5d 8c eb 49 appears in the powerplay bus, the powerplay tram and the Downtown backdrop list above. That CRC resolves to a real name via the C2NM trailer of the level's _po.textures, which ends with:

2e3080  72 74 5f 6c 61 75 6e 63 68 65 72 5f 30 31 5d 8c   rt_launcher_01].
2e3090  eb 49 00 00 00 14 63 75 62 65 6d 61 70 5f 47 65   ..I....cubemap_Ge
2e30a0  6e 65 72 69 63 5f 73 77 61 70                      neric_swap

Annotation: 5d 8c eb 49 = CRC, 00 00 00 14 = length 20, "cubemap_Generic_swap". So that shared key is the generic reflection cube map every prop samples — exactly the kind of common dependency a CRC list exists to declare. This round-trip (a CRC in a .crcs list → the same CRC + name in a C2NM trailer → the same CRC in a TEXS descriptor) is what proves all three formats use one CRC namespace.

CRC algorithm

The exact hash is not yet pinned. The values are 32-bit and clearly name-derived (the same name yields the same CRC everywhere it occurs), consistent with the engine's generic PackedRGBA/asset-name hashing referenced in the eboot's Crayon2 code. Black Rock's engine elsewhere uses standard CRC-32; whether these are CRC-32 of the lowercase name, a custom polynomial, or a seeded variant is left Partial pending a brute-force match against the known C2NM name→CRC pairs (e.g. "cubemap_Generic_swap" → 0x5D8CEB49, "Musclecar_01" → 0xB7B2A83B).

How we know

  • Flat uint32 array: every sampled file size is a multiple of 4 (0/20/24/32 B); contents are high-entropy with no ASCII/offset structure.
  • CRCs = texture names: each 4-byte value matches a crc field in a TEXS descriptor and a CRC in a C2NM name record.
  • Shared keys: 0x5D8CEB49 appears in three independent .crcs files and resolves to "cubemap_Generic_swap" in a _po.textures trailer.
  • Purpose: eboot path environment|Backdrop/%s_backdrop.texture.crcs shows they are loaded per object to declare a texture working set.
