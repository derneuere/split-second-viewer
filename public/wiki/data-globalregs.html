Global Registry (default.global_regs)

default.global_regs is the renderer's master register-block database: a single binary file at the root of Deferred/ that declares every named shader global and texture sampler the Crayon2 renderer is allowed to bind, together with each one's default slot. The eboot loads it at render-init with the message Loading register block sets from %s. and the data path data|default.global_regs, sitting amongst cShaderManager, cGlobalShaderContextManager and cMaterialComboManagers. Despite the "registry" name this is firmly a graphics file, not a gameplay config — it is the dictionary that maps shader uniform names to engine-side slots.

Role in the engine

Split/Second is a deferred renderer whose shaders are assembled at runtime from the 2.4 MB global_shader_combos.fxc übershader database and patched live by cShaderPatcher. Every shader that wants to read a camera matrix, a light parameter, a bloom threshold or a G-buffer texture refers to it by name. default.global_regs is the authoritative list of those names — the "global register block set" — so the runtime can resolve a name to a hardware constant/sampler slot and supply a default. Think of it as the symbol table that binds the shader übershader to the engine's per-frame globals.

There is exactly one .global_regs file in the build: Deferred/default.global_regs (65,354 bytes). The default. prefix and the FREG magic (File-REGisters) imply the format could carry alternate named sets, but only the default is shipped. Inside, the same register vocabulary is repeated across five record blocks (the name light_ambient recurs five times, at 0x18, 0x384a, 0x546c, 0x8c9e and 0xe6be) — presumably one set per shader stage / material-combo context. Summed across the five blocks the file holds ~1,500 records (1,503 FFFFFFFF slot markers), not the ~1,000 a single block suggests.

Binary layout

The file is big-endian and walkable. A 20-byte preamble (8-byte FREG+version, the GLBB tag, then two count/size dwords) is followed by a run of variable-length records, then a GLBE footer. Reading the head with od -A x -t x1z shows the magic, version, the GLBB ("Globals Block Begin") tag and the first three register entries:

The second header dword, 1500 (0x5DC), is almost certainly the total record count: walking the file turns up 1,503 FFFFFFFF slot markers, i.e. 1,503 records, which rounds to the stated 1,500. (This corrects the earlier guess that the file holds only ~1,000 entries — that figure counts a single block; the vocabulary repeats five times.) The first dword, 512 (0x200), is still Theory — it does not match any count we measured, so it reads as a hash-table bucket count or per-block capacity hint. Note the same 00 00 02 00 / 00 00 05 dc pair recurs mid-file at 0x5460, so it behaves more like a sub-block header than a one-time file preamble.

Register record

Each entry is a length-prefixed, NUL-terminated name followed by two dwords. light_ambient at 0x14 is 00 00 00 0e (nameLen 14) + the 13 ASCII bytes "light_ambient" + its NUL terminator (13 + 1 = 14) + NUL padding so the following dword is 4-byte-aligned + ff ff ff ff + 00 00 00 01:

…0x4uint32_tregCountnumber of float4 constant registers the global occupies Partial
The trailing dword is not a two-valued type tag. An earlier reading called it "1 for scalars/vectors, 4 for matrices", but walking one full block (340 records) turns up many values: 1 (291×, e.g. light_ambient), 2, 3 (e.g. NormalTrans), 4 (30×, e.g. Trans/View/Proj), 8, 12, 13, 32, 35 (e.g. bonebind_BindToJointMatrices), 64 and 105 (e.g. bonebind_JointToModelMatrices).

The values line up with how many float4 constant registers each global needs: a scalar or single vector is 1; a 4×4 matrix (Trans, View, Proj, ViewProj) is 4 rows; NormalTrans is 3 rows; and arrays of N vectors/matrices (sample-offset tables, bone-bind matrix palettes) carry the larger counts (8, 16, 32, 64, 105, …). So this field is best read as a register/vec4 count, not a discrete type enum — hence still Partial until the array vs single-matrix cases are fully mapped. (Note: NormalTrans ends in 00 00 00 03, not …04 as previously stated.)

Footer

The block closes with GLBE ("Globals Block End") immediately followed by the ASCII marker END:

What the registry holds

Walking the names gives a near-complete map of the deferred pipeline, because the registers are grouped by the shader/pass that owns them — the prefix before the first underscore is the pass name. The largest groups in the file:

A second class of entries are the always-present frame globals bound for every draw: the camera/world matrices Trans, View, Proj, ViewProj, TransViewProj, NormalTrans; the camera/light vectors WorldCameraPos/Right/Front/Up, ObjectCameraPos/Right/Front/Up, WorldLightDir, ObjectLightDir; and the base material lights light_ambient, light_diffuse, light_specular. These appear first in the file, before the per-pass groups.

The pass prefixes here line up one-to-one with the deferred-pipeline stages documented on Rendering Pipeline — G-buffer → screen-space mask/SSAO → deferred lighting → shadows → HDR bloom/tone-map → DOF/motion-blur → edge post. The registry is effectively a parts list for that pipeline.

Relationship to other render data

default.global_regs declares the names and slots; the actual shader byte-code lives in the .fxc compiled effects and the global übershader combo DB, instanced through .shaders / .shaderinst and the texture sets in .textures. At runtime cShaderPatcher stitches these together using this registry as the binding dictionary. Because every defaultSlot in the shipped file is −1, the engine assigns slots dynamically — the file fixes the vocabulary, not the layout.

Open questions

  • Why the register vocabulary is repeated across five blocks — per shader stage, per material-combo context, or per render path? The block sizes differ (340 / 172 / 340 / 565 / 86 records), so they are not identical copies.
  • Meaning of the first header dword, 512 (0x200). It matches no measured count, so it is taken as a bucket/capacity hint — unproven. (The second dword, 1500, is now read as the total record count: 1,503 markers were found.)
  • The exact semantics of the regCount dword for the array cases (8 / 16 / 32 / 64 / 105 …) and whether texture samplers are encoded distinctly from constants at all.
  • Whether defaultSlot is ever non-−1 — in this build it is uniformly −1, so the "default" semantics can only be inferred.
