Lip Sync (.lip)

Naming correction. Despite the extension and this page's catalogued title, .lip in Split/Second is not voice lip-sync / viseme data. The only .lip files in the build are named Track.lip and live in each level's Event/RACING/TrackLogic/ folder, alongside Track.checkpoints, Track.linkorigins, Track.logicinfo and a human-readable Track.debug.txt. The file is the compiled TrackLogic node graph — the AI racing-line / respot / crash-zone network used by SSAI and the race director. "LIP" is best read as Logic / racing-line data Pack; there is no speech in it.

This page documents what the format actually contains, verified against the debug twin. For the related track files see Route Data and Logic Info.

Provenance — how it was identified

The eboot string table makes the lineage explicit. Under the TrackLogic loader the format strings are emitted side-by-side:

environment(noassert)|Event/%s/TrackLogic
environment(noassert)|Event/%s/TrackLogic/%s
%s/Track.lip
%s/Track.linkorigins
%s/Track.checkpoints
%s/Track.logicinfo

Surrounding log tags are TrackLogic_Error/Warning/Info, with messages like "No checkpoints found", "Found Start Grid @ …", "AITrack data file is empty". The neighbouring AI debug menu (AI/Debug/TrackRender/…) renders "Racing Line", "Block Logic" and "No Respot" — exactly the per-node flags found in the binary. None of the audio/FMOD/lip-sync strings reference .lip.

Ground truth — the debug twin

Each Track.lip ships next to a Track.debug.txt that dumps the same data as text, which makes verification direct. The opening of Downtown/A/Track.debug.txt:

Track
=====
Node [0]
Pos (-1342, 8, 254)
Crash zone [ ]
Can respot [x]
DropZone False
TriggerZone False
Node [1]
Pos (-1314, 8, 239)
…

So a node carries a 3D position and four booleans: Crash zone, Can respot, DropZone, TriggerZone. These map directly onto the binary records below.

File header

The first 0x24 bytes are a big-endian directory. The first word at 0x00 is 0x50 = the per-node record stride (80 bytes), confirmed below by measuring the distance between consecutive node position triples. The node array begins immediately at 0x24; the words at 0x10–0x1c are a small section table whose larger offsets (0x114, 0x11b) point past the node array into the link/checkpoint sub-arrays — i.e. the file is not entirely node records.

Node record (0x50 bytes)

Records repeat on an 80-byte (0x50) stride — the value held in the header word at 0x00. Offsets below are relative to the record start (record n begins at file 0x24 + n×0x50). Only the fields decoded directly against the Track.debug.txt twin are marked confirmed; the rest are honestly left as unknown.

OffsetLengthTypeNameDescription
+0x000x10float32_t[4]tangent / dirFour floats; the first is small and signed (can be ±), the second ≈0.85, the third is 0.0, the fourth ≈0.51 — a normalised direction/tangent (and a planar component) at the node. Theory
+0x100x4float32_tposXWorld X. c4 a7 c5 b0 = -1342.18 → matches Node[0] "(-1342, …)". Confirmed
+0x140x4float32_tposYWorld Y. 41 08 2b 7b = 8.51 → "(…, 8, …)". Confirmed
+0x180x4float32_tposZWorld Z. 43 7e 1c e2 = 254.11 → "(…, …, 254)". Confirmed
+0x1C0x24uint32_t[9]links / sub-indicesMostly zero, with a couple of small ascending indices (e.g. one word equals the node's own index+1) — adjacency / link references into the sibling sub-arrays. Partial
+0x400x4float32_twidth / radius41 68 d4 fe = 14.55 (varies a little per node) — node corridor half-width. Theory
+0x440x4uint8_t[4]flagsTypically 01 01 00 00. Packs per-node booleans — but see the warning below: it does not by itself distinguish "Can respot" on/off, so the exact bit assignment is unconfirmed. Partial
+0x480x4uint32_tnodeIndexSequential node id (0, 1, 2, 3 …), one per 0x50 record. Confirmed
+0x4C0x4uint32_tfield_4CZero in sampled records.
The 0x50 stride was confirmed by locating consecutive position triples: node[0] floats at file-offset 0x34, node[1] at 0x84, node[2] at 0xd4 — each exactly 0x50 apart — and all decode to the coordinates printed for Node[0]/[1]/[2] in the debug twin. The nodeIndex word at +0x48 likewise runs 0,1,2,… in lock-step with the debug "Node [n]" headings.
Earlier drafts of this page claimed a 0x40 (64-byte) stride; that was wrong — measuring three consecutive nodes gives 0x50. Also note Node[46] of Downtown/A is the only "Can respot [ ]" node in that track yet its +0x44 bytes are still 01 01 00 00, identical to its respot-able neighbours — so the respot boolean is not carried in that byte pair (it may live in a sibling sub-array such as Track.linkorigins). The four-boolean packing is therefore still open.

What is still open

The position triple, the sequential nodeIndex (+0x48) and the 0x50 stride are confirmed against the debug twin; corridor width (+0x40) and the leading direction floats are well-supported but not cross-checked against text. Still uncertain: the exact bit assignment of the flags word at +0x44 (it does not by itself encode "Can respot"), the meaning of the +0x1C…+0x3F link/sub-index words, and the trailing section table at header 0x10 (the 0x114/0x11b offsets point into checkpoint/link sub-arrays shared with Track.linkorigins and Track.checkpoints). A full bit-for-bit decode — including which sibling file actually carries the four per-node booleans — is left for a later pass.
