Networking & Online

Split/Second's online multiplayer is built on Quazal Net-Z and Quazal Rendez-Vous (the same middleware family later rebranded "NEX"), layered on top of Sony's PlayStation Network (sceNpMatching2, NPBasic, NPScore). Black Rock wrapped all of it in an in-house module called ATVNet, whose source lives at c:/Projects/proj_1/ws_0/Code/games/SplitSecond/Shared/ATVNet/…. Game state is shared between consoles using Net-Z's Duplicated Object (DO) replication model: every networked thing in the race — players, vehicles, the host's game setup, the room, even individual physics objects — is a cATVNetDO* class that Net-Z keeps in sync across all stations.

All strings, class names and debug-menu paths quoted here come from the devkit/QA EBOOT (eboot_strings.txt). Because this is a debug build, the entire matchmaking/replication state machine prints itself, so the online model is unusually well documented for a shipped game.

The middleware: Net-Z and Rendez-Vous

The EBOOT carries two distinct Quazal libraries from the contrib/Quazal/ tree:

  • Net-Z (contrib/Quazal/NetZ) — the peer-to-peer object-duplication engine. Identifies itself at runtime as Quazal Net-Z / Version 2008.9 SP2 with Copyright (C) 1998-2009, Quazal Technologies Inc. It owns the DuplicationSpace, the SessionClock (a replicated clock used to timestamp updates), and the ObjDup subsystem.
  • Rendez-Vous (contrib/Quazal/RV) — the back-end / matchmaking framework (Quazal Rendez-Vous). It is reached through a Python-flavoured RPC core (PythonCore) and exposes named protocols: RVAuthentication, RVSecure, SessionDiscoveryProtocol, SandBoxMgmt, Routing and NATEcho. The game enforces "You must be logged in to SonyNP prior to logging in to Rendez-Vous."

Secure Rendez-Vous streams can use RC4 encryption and ZLib compression, both toggleable — the EBOOT logs "set RC4 encryption engine for Rendez-Vous Secure streams" and "set ZLib as compression engine for Rendez-Vous Secure streams" (with corresponding "no encryption / no compression" variants).

PSN as the transport layer

On PS3, Rendez-Vous rooms ride on top of Sony's matching service. The network manager calls sceNpMatching2 directly for room lifecycle (create / join / leave / kick / transfer owner / hide-or-close room) and pulls room events with sceNpMatching2GetEventData. Other PSN services in use:

PSN serviceUsed for
sceNpMatching2*Rooms, server lists, member updates, owner migration
NPBasic (sceNpBasic*)Block list, custom game invitations

cATVNetManager spins NPBasic and NPScore up and down on demand (EnsureBasicAndScoreRunning() / TerminateBasicAndScore()), so the title only holds those PSN contexts open while online features are actually live.

ATVNet: the Black Rock wrapper

Game code never talks to Net-Z directly; it goes through ATVNet. The two top-level managers are:

  • cNetworkManager — owns the matchmaker mode and the NetZ instance lifecycle. It has a four-state matchmaker (MATCHMAKER_LAN, MATCHMAKER_ONLINE, and two transition states MATCHMAKER_SWAPTOLAN / MATCHMAKER_SWAPTOONLINE) and can hot-swap between LAN and Online by tearing down and recreating NetZ. It also drives UpdateNetZ() each frame and asserts loudly if that stalls.
  • cATVNetManager — the ATVNet-level session controller: Start(), host migration (SessionMigrateHost()), score reset broadcasts, and the player-slot bookkeeping below.

A small helper thread, cKeepNetZAlive, runs a worker loop ("worker running / waiting / quitting") so Net-Z keeps getting serviced during otherwise-busy phases like level loading.

NetZ creation can be async / deferred

Because tearing NetZ down and standing it back up takes time, the manager guards against races: creating while already created destroys-and-recreates; creating during a pending termination is held up until termination finishes. The QA build exposes these as debug toggles debug/AsyncNetIssues/HoldUpNetZCreation and debug/AsyncNetIssues/HoldUpNetZTermination for testing the deferred paths.

Duplicated Objects: the replication model

Net-Z's ObjDup keeps a DuplicationSpace in which each replicated object has one duplication master (the authoritative station) and any number of duplicas on other stations. When a master leaves, the object migrates to a new master ("emigration" / "approving emigration" / host migration). Each DO class is defined by a *DDL.cpp file (Net-Z's Data Definition Language) and pairs a behaviour class (cATVNetDO*) with a plain data struct (cATVNetDS*) holding the replicated fields.

What a PlayerUpdater actually carries

The DDL type table in the EBOOT exposes the replicated field names and their wire types. The high-frequency cATVNetDSPlayerUpdater rides the unreliable protocol behind an InGameOnlyFilter and packs movement and input into a compact, timestamped payload; the updater explicitly rejects stale packets ("out-of-order update detected. last time stamp … this time stamp …"). mScore and player mState are not in the PlayerUpdater — they live on slower-changing objects: mScore (uint32) is a field of cATVNetDSPlayerData (the PlayerSetup struct), and mState is the state field of both cATVNetDSRoomStateData (uint64) and cATVNetDSPeerStateData.

Other replicated DS fields, by owning struct (from the same DDL type table): mTrack, mGameModeIndex, mAIDifficulty (GameSetupData); mScore, mBuildNumber, mRaceForm, mRaceFormGuide, mAnchovies (PlayerData); mVehicleName, mColour (VehicleData, referenced from PlayerData's mVehicle); mState, mNumPlayers, mNonce, mEventStartTime, mRankedMatchStartTime, mHandles, mPhysicsHandle (RoomStateData); and mIsHumanPlayer, mPeerState, mPlayerSetup, mPlayerUpdater, mEventUpdater (the dohandles in PlayerDOHandles). mAnchovies (uint64, a member of PlayerData) is an internal codename consistent with Black Rock's food-themed naming (cf. "Catnip", "Megabowles"; the PS3 trophy backend is also cAnchoviesPS3).

Because AI cars are also full DOs, the game can migrate AI drivers between consoles when a host leaves — cATVNetEventSlotManager::HostMigrateAIUpdatersLoadBalanced() spreads the bot updaters across remaining peers, and warns if "there's no one we can migrate AI players to!"

Slots, host migration & load balancing

A race is organised as a fixed array of player slots managed by cATVNetEventSlotManager. On the host, Host_PopulateSlots() fills empty slots with AI DOs; on a client, Peer_SetupPlayerControls() walks the slots and classifies each as my player, one of my AIs, a remote human or a remote AI. Each occupied slot points at a cATVNetDOPlayerDOHandles that ties together that player's Setup, Updater and PeerState DOs.

When the duplication master of these objects disappears, Net-Z performs host migration: the managers approve/deny emigration and "fault recovery" per object based on whether the local peer is in a valid state to accept game DOs, and cATVNetManager::SessionMigrateHost() hands off the PSN room ownership in parallel (the PSN side fires HandleRoomEventRoomOwnerChanged()).

State machines: PeerState and RoomState

Two replicated state machines drive the lobby→race→results flow. RoomState is the host's global view; PeerState is each station's local view. The QA build prints every transition ("Room State is now: %s" / "Local Peer State is now: %s").

Arbitration is the host-authority handshake that gates the start of a ranked match; EXCHANGING_DATA is the phase where peers swap their PlayerDOHandles and verify everyone's in-game DOs before going to LOADING ("We've got everyone!"). If peers time out loading, the host can start early and drop the stragglers, or bounce the room back to the lobby ("No peer is at LOADING! Returning room to lobby").

Ranked vs. Quick Match

ATVNet distinguishes two online flows, and even suppresses error popups differently in each ("Not showing error message due to being in ranked mode" / "… due to being in quick match"):

  • Quick Match — search for an existing room, and if that fails, create one ("QuickMatch menu: search for rooms failed, so maybe create one"). Search vs. create criteria are mapped via ATVNetMappings, which sets a GAME_NAME CRC and a GROUP_NUMBER when building the room descriptor.
  • Ranked Match — uses an auto-start countdown (Online/RankedMatchCountdown), records mRankedMatchStartTime, and feeds the Race Form reputation system below.

Online results adjust a per-player "Race Form" rating, with an explicit cheat / quit penalty. These are tunable from the debug menu under ATV/Online/Race Form/…:

Voice chat (AVStreams)

Voice runs over Net-Z's AVStreams extension (contrib/Quazal/NetZ/lib/Extensions/…), not through the game's FMOD/LLAudio2 engine (the many Audio/Voice/… debug entries refer to FMOD audio channels, a separate concept). The voice path is a replicated VoiceChannel object plus a pair of low-bitrate speech codecs:

  • LPC10 — …/AVStreams/Codecs/LPC10/LPC10Codec.cpp
  • OpenLPC — …/AVStreams/Codecs/OpenLPC/OpenLPCCodec.cpp

Both are classic ~2.4 kbit/s linear-predictive vocoders — tiny, intelligible-but-robotic speech, exactly what you want sharing the wire with gameplay traffic. VoiceChannel (with its DDL twin) manages stream allocation and routing: OpenLocalStream / CloseLocalStream, GetAvailableStream / ReclaimStream / ReleaseStream, and packet handling (HandleVoicePacket, HandleJoin, HandleLeave). Its replicated ChannelInfo struct carries the codec id and stream parameters:

On the game side, each talker is a cNetChatParticipant whose cNetChatParticipantInfo exposes mActiveIndex and mIsHeadsetAttached. The frontend mirrors this with debug flags for both lobby and in-race voice: ATV/Online/Voice/Room - Has Headset / Is Talking / Muted and the matching … In Game - … set.

Errors & fault handling

Net-Z and Rendez-Vous surface a rich error taxonomy. Rendez-Vous login errors include NotAuthenticated, InvalidUsername, InvalidPassword, AccountDisabled, AccountExpired, ConcurrentLoginDenied, EncryptionFailure and InvalidOperationInLiveEnvironment; transport errors include ConnectionFailure, Timeout and NATEcho-related connectivity tests. Join failures distinguish JoinDenied from ConnectivityTestFailed and report the raw Quazal error code. If a station is orphaned (loses its host) it logs "We've become orphaned - leaving room."

This is an offline-only era of PSN: the original Rendez-Vous back-end servers for Split/Second are long dead, so none of this is reachable today. The value here is historical / reverse-engineering — the build documents the protocol, not a live service.
