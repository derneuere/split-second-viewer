Vehicle Physics (.phys)

Each drivable car in Split/Second ships a .phys file beside its .model and .mainColl. It is a Havok 5.5 packfile whose top-level object is an hkpPhysicsSystem — i.e. the car's simulated rigid body, its collision shapes, its mass & inertia, and a wheel constraint. Read the Havok Packfiles page first; this page only documents the vehicle-specific payload that lives in the packfile's __data__ section.

There are 21 .phys files — eleven Musclecars (incl. Musclecar_Drone) and ten Supercars — sizes ranging ~51 KB (Supercar_05, 51 312 B) to ~62 KB (Musclecar_01, 62 208 B). Across the five samples diffed (Musclecar 01/02/Drone, Supercar 01/10) the packfile header, the two section headers for __classnames__/__types__, and the entire __classnames__ and __types__ payloads (bytes 0x00–0x51A0) are byte-identical between cars. Only the __data__ section — its header (sizes/fixup offsets) and its payload — differs. That tells us the per-car tuning lives entirely in the data section; every .phys shares the same 41-class schema and the same __data__ start offset of 0x51A0.

Important scope note. Despite the name, a .phys file does not contain the high-level handling model (engine torque curve, gearbox, aerodynamics, suspension stiffness, steering). The Havok Vehicle Kit classes (hkpVehicleData, hkpVehicleDefaultEngine, hkpVehicleDefaultSuspension, hkpVehicleDefaultAerodynamics, hkpVehicleDefaultSteering …) exist in the EBOOT but are not serialized in the .phys — none of those class names appear in the file's embedded registry. The car's hkpVehicleInstance is built at runtime by Black Rock code, most likely fed from .params tuning. The .phys supplies the rigid-body chassis it attaches to.

So a .phys is one chassis rigid body, with sphere/convex collision shapes and exactly the constraint plumbing for a wheel. The __types__ section spells out the members of each of these — the field names quoted below are taken verbatim from that registry.

Rigid body, mass & inertia

The hkpRigidBody (via hkpEntity → hkpMotion) carries the dynamics state. The mass/inertia data is in the motion object, whose registry members include:

FieldTypeMeaning
typeenum MotionTypeSolver model — e.g. MOTION_BOX_INERTIA (see enum below).
motionStatehkMotionStateTransform, swept transform, objectRadius, linearDamping, angularDamping, maxLinearVelocity, maxAngularVelocity.

The motion's centerOfMassLocal (in the nested hkSweptTransform) sets the chassis CoM — a key handling lever even though it lives in the rigid body rather than a "handling" struct.

Collision shapes

The chassis collidable points at a shape. The registry shows both hkpSphereShape (a single sphere, members radius + inherited hkpConvexShape.radius) and the hkpConvexVerticesShape family (an arbitrary convex hull: aabbHalfExtents, aabbCenter, rotatedVertices, numVertices, planeEquations, connectivity). A car's broad-phase proxy is a sphere; its actual chassis hull is the convex shape. The detailed drivable-surface mesh is in the separate .mainColl.

Material (friction & restitution)

The hkpMaterial on the body holds three members, in this declared order (confirmed from the embedded __types__ registry at 0x1B28/0x1B40/0x1B50): responseType, friction, restitution. They are directly readable as adjacent values in the data section, and the values found are the classic Havok defaults — which is itself evidence the parse is correct:

FieldTypeValue (Musclecar_01)How known
responseTypeenumRESPONSE_SIMPLE_CONTACT (1)First member; value 0x00000001 = index 1 in the ResponseType enum in __types__.
frictionfloat32_t0.5 (3F 00 00 00)Second member; decodes to 0.50 big-endian. This is the Havok default friction (0.5).
restitutionfloat32_t0.4 (3E CC CC CD)Third member; decodes to 0.40 big-endian. This is the Havok default restitution (0.4).
Order matters here: the floats appear in memory as responseType=1, 0.5, 0.4 — i.e. friction=0.5 then restitution=0.4, matching both the hkpMaterial member order and Havok's stock defaults. (An earlier draft of this page had the two floats swapped.)

Wheel constraint

The registry includes the constraint plumbing classes (hkpConstraintInstance, hkpConstraintData, hkpConstraintAtom, hkpModifierConstraintAtom), and the ConstraintType enum in __types__ spells out the full Havok set — CONSTRAINT_TYPE_BALLANDSOCKET … CONSTRAINT_TYPE_WHEEL … CONSTRAINT_TYPE_POWERED_CHAIN (21 entries, verified by string scan). In the canonical Havok 5.5 ordering CONSTRAINT_TYPE_WHEEL is value 7; the in-file enum value table is interleaved and was not cleanly byte-traced, so treat the specific value as Theory.

Caveat. The presence of these classes in the __classnames__ registry proves the schema is available, not that a wheel constraint instance is actually serialized in this file's __data__. Whether hkpPhysicsSystem.constraints is non-empty (and, if so, which constraint type it carries) was not confirmed by walking the data graph. The wheel simulation proper (raycast wheels, suspension spring response) is in any case the runtime hkpVehicle* code referenced in the scope note above, which is not in this file.

Annotated hex — data section start

The __data__ section of Musclecar_01.phys begins at file offset 0x51A0 (its header's absoluteDataStart). Below is the region where the material floats sit; the 0.4/0.5 pair is what identifies it as the hkpMaterial:

0052e0  00 00 00 01 00 00 00 01 41 e6 64 10 01 00 00 00  ........A.d.....
0052f0  00 00 00 00 00 00 00 01 41 e6 64 10 01 00 00 00  ........A.d.....
005300  00 00 00 00 00 00 00 00 3b 83 12 6f 00 00 00 00  ........;..o....
005330  01 00 00 00 3f 00 00 00 3e cc cc cd 00 00 00 00  ....?...>.......

Decoding the marked big-endian floats:

  • 41 E6 64 10 → 28.799 (appears twice — a shape/AABB extent or radius pair).
  • 3B 83 12 6F → 0.004 (a small tolerance, consistent with allowedPenetrationDepth).
  • 01 00 00 00 → 1 = responseType (RESPONSE_SIMPLE_CONTACT).
  • 3F 00 00 00 → 0.5 = friction (second member).
  • 3E CC CC CD → 0.4 = restitution (third member).
The material decode is the cleanest anchor in the file: responseType (=1) followed by two textbook Havok defaults (friction 0.5, restitution 0.4) sitting adjacent, in exactly the order the hkpMaterial registry declares them. That both floats land on round values under a big-endian read is strong confirmation the endianness and field mapping are right.

Confidence

Confirmed: container/header (see Havok Packfiles), the root object is hkpPhysicsSystem (header offset resolves to it at 0x17C), the complete 41-class embedded class set, the hkpMaterial values (responseType=RESPONSE_SIMPLE_CONTACT, friction 0.5, restitution 0.4 — corrected from an earlier swapped draft), and that header + classnames + types are byte-identical across cars while only __data__ varies. Partial: exact byte offsets of individual members within each object (names/order confirmed from __types__, offsets depend on Havok layout rules and were not all traced). Theory: the 28.799 and 0.004 field assignments; that an actual wheel-constraint instance is serialized (only the schema is proven present) and that CONSTRAINT_TYPE_WHEEL = 7 (canonical, not byte-traced here); and the claim that per-car handling is driven from .params rather than the .phys (well-supported — no vehicle-dynamics classes are serialized — but the exact runtime wiring was not traced).
