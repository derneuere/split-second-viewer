TrackTivity Telemetry (.track)

TrackTivity is Black Rock's in-house QA telemetry / replay subsystem. The .track files left loose in the Deferred/ root are its simplest on-disk artefact: a captured driving path — a flat list of world-space line segments tracing where a vehicle drove during a session. Each file is a tiny, self-describing binary blob: a single 32-bit count followed by that many 24-byte records, with no magic and no embedded strings. The format is fully recovered here — every byte is accounted for across all 111 samples on disk.

Do not confuse this format with the richer TrackTivity Replay serialization (the cTrackTivityEvent / cITracktivitySerializable object stream used for the keyframed ghost-replay system). That is a typed, named event buffer; the .track files on disk are the plain geometry dump only. Both are produced by the same subsystem — see Relationship to the replay system below.

All multi-byte values are big-endian (PS3 PowerPC). Offsets in the tables are byte offsets from the start of the file. Verified against the full set of 111 .track files in USRDIR/Deferred/, ranging from 1 612 bytes (67 records) to 11 476 bytes (478 records).

Where the files come from

The EBOOT string table (eboot_strings.txt) carries the exact printf templates used to name these files:

TracktivityData_%c_%s_%d_%d_%d_%d_%d_%d.%s
TracktivityData/TTTData_%c_%s_%d_%d_%d_%d_%d_%d.%s

Decoding against a real filename TracktivityData_P_PS3-_2010_1_10_14_18_32.track: %c is a single platform/category char (always P in the samples), %s is a build/branch tag, the six %d are year / month / day / hour / minute / second of capture, and the trailing %s is the extension (track). The branch tags seen on disk are PS3- (81 files), Unkn (28) and boff (2). The first template writes straight into Deferred/; the second (TTTData_…) targets the Deferred/TracktivityData/ subfolder, which exists but is empty in this build.

The same code path references a Disney QA upload endpoint — http://brmilk.emea.wdpr.disney.com/add_tracktivity_file/ — and the toggle game_setup/enable_TrackTivity, confirming these are automatically captured QA session recordings meant to be harvested off devkits, not shipping game content. A live profiler timer "Tracktivity Update" (alongside "Tracktivity Replay System Update" and "Nemesis Update") shows the capture runs every frame.

File layout

There is no header beyond a single count. The body is a tightly packed array; there is no trailing data, no alignment padding, and no footer.

OffsetLengthTypeNameDescription
0x00x4uint32_trecordCountNumber of segment records that follow. Big-endian; the high two bytes are 00 00 in every sample (max observed 478 = 0x01DE).
0x4recordCount × 0x18TrackSegment[]recordsThe path itself — see segment record.
Size invariant (how the record size is known): for every one of the 111 files, 4 + recordCount × 24 == filesize holds exactly, with zero trailing bytes. Dividing (filesize − 4) / recordCount yields a clean 24 in all cases (e.g. (1612 − 4) / 67 = 24, (11476 − 4) / 478 = 24). That is what pins the record at 24 bytes and the header at 4.

Segment record (24 bytes)

Each record is six big-endian float32 values: two 3D points, a segment start and end, in world space. Decoding the records and plotting them confirms a connected driving line: within a continuous stroke the end of record N equals the start of record N+1, so consecutive segments share an endpoint (this is the "duplicated pairs" pattern visible in the raw bytes). Y is the vertical axis and stays nearly constant within a run (ground height); X and Z sweep across the track.

OffsetLengthTypeNameDescription
0x000x4float32_tstart.xSegment start, world X (metres).
0x040x4float32_tstart.ySegment start, world Y (vertical; ≈ ground height).
0x080x4float32_tstart.zSegment start, world Z (metres).
0x0C0x4float32_tend.xSegment end, world X. Equals start.x of the next record within a stroke.
0x100x4float32_tend.ySegment end, world Y.
0x140x4float32_tend.zSegment end, world Z.
Why "two points" and not "pos + previous pos per sample": if the six floats were (position, previous-position) of one moving sample, the second triple of record N would equal the first triple of record N+1. The data shows the opposite — the first triple of N+1 equals the second triple of N — i.e. records are directed edges (A→B), (B→C), (C→D)…. So a record is a line segment, not a doubled sample.

Strokes (path breaks)

The end-equals-next-start chaining holds for most records but not all: in the small sample below 56 of 66 adjacencies chain cleanly, and in the largest file 426 of 477 do. The breaks are not corruption — they are genuine "pen-up" jumps where one captured stroke ends and another begins (e.g. respawns, restarts, or separate path layers within one session). A reader should therefore treat the array as a set of polyline strokes: walk the records, and start a new polyline whenever record[i].start ≠ record[i-1].end. There is no explicit stroke-count or per-stroke length field — stroke boundaries are inferred purely from endpoint continuity.

Because stroke boundaries are implicit, the file format alone cannot tell you which stroke belongs to which lap, vehicle, or event. That association lives in the richer replay event stream, not in the .track geometry.

Relationship to the replay system

The .track geometry is the visible tip of a larger telemetry/replay system. The EBOOT string table also lists a typed, serialized event hierarchy — all subclasses of cTrackTivityEvent / cITracktivitySerializable — that the replay loader deserializes into a time-indexed events buffer:

ClassRecords
cDrivingStateEventPer-tick driving model state (position / orientation / steering — drives the keyframed ghost).
cLapFinishedEvent / cRaceFinishedEventLap and race completion markers.
cPowerplayTriggeredEventA Powerplay firing; carries a powerplay_transform_xx/xy/xz… matrix.
cCrashEventCrash / takedown marker (rendered with mCrashEventColor).
cPlayerScoreEvent, cGameInfoEventScoring and game-info samples.
cPlayerContactVehicle / cPlayerContactEnvironmentContact events vs. other cars / the world.

This stream is what the debug-menu TrackTivity / Replay page replays — its tuning keys (Use Keyframed recording, Use Sweep Position (in the physics) instead of Set Position, interpolation error thresholds, Time size of the events buffer (in secs)) all describe how the recorded driving model is played back over the live one. A separate ASCII serializer (the "ASCII File To HDD Serializer Worker Thread") can dump the same data as text using the tagged form <<<type="TrackTivity", name="…", value0="…">>> with helpers %s v3(%f %f %f) / %s v2(%f %f) / %s c(%d %d %d %d). Neither the serialized event stream nor the ASCII export ships in this build's Deferred/ root — only the binary path dumps do.

Confidence split. The .track binary geometry is Confirmed (size law holds for all 111 files; floats decode to a sensible world-space path; chaining is consistent). The event-stream layout of cTrackTivityEvent subclasses is Theory here — the class names and field names are known from the EBOOT, but no serialized event file is present on disk to verify their byte layout.

Playback rendering

When played back in the debug build, TrackTivity draws the path and overlays per-sample telemetry. The renderer's tunables (also from the EBOOT) name the channels it can visualise: mShowVelocity / mVelocityScale, mShowOrientation / mOrientationScale, mShowSteering / mSteeringScale / mSteeringColor, mUseColorsForAccelAndBrake (mAccelColor / mBrakeColor), plus per-player toggles mShowPlayer[%d] and time controls mCurrentTime / mMinTimeToRender / mMaxTimeToRender. The plain .track file only carries the path line; the velocity/steering/colour overlays come from the richer event stream above.
