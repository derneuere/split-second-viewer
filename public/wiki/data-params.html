.params Tuning System

.params is Split/Second's universal plaintext tuning format. Almost every game system — AI, scoring, catch-up, cameras, area-of-effect explosions, game modes — is configured by one or more .params files rather than by hard-coded constants. Each file is a flat, human-readable list of named values with a min/max range attached, which lets the in-game QA debug build expose every value as a live tweak slider. Because the files are ASCII, the values quoted on this page are read directly from the shipped data — this is the single most legible format in the whole game.

Why a text format

This is a devkit / QA debug build (title id NPXX00575) with qa_menus=true, so the data ships loose and un-compiled. Designers edited these files and the runtime parsed them on load; the eboot exposes the loader through strings such as Load Params / Load Tweaks, Tweakvars and the map-maker variant (Map maker: Loading tweaks… / mapmaker.params). A retail disc build would normally bake these into a binary blob, but here you can read — and in the QA menus, edit — every value.

There are 727 .params files under Deferred/ in this build. They range from a 224-byte stub (GameLogic/Catchup.params) to 50 KB+ per-game-mode setups, and appear everywhere from SSAI/ driving tables to per-camera behaviour files.

File anatomy

Every file opens with a !directory: header that names the logical tree the file populates, then one or more section groups. A group is a path ending in a colon, followed by tab-indented value lines. The complete GameLogic/GameplayParameters.params file shows the whole grammar in one screen: 

!directory: /GameplayParameters

/GameplayParameters/AI Powerbar/Drafting:
	'000;Minimum distance from player to start gaining powerbar' = 5.0 (1.0, 10.0);
	'001;Maximum angle difference with player to start gaining powerbar' = 5.0 (0.0, 90.0);
	'002;Units gained per second' = 4.0 (0.0, 100.0);

/GameplayParameters/AI Powerbar/Drifting:
	'000;Units gained per second' = 14.0 (0.0, 100.0);

/GameplayParameters/LDC:
	'000;Speed Intervals' = 60.0 (1.0, 300.0);
	'001;Initial Speed' = 60.0 (1.0, 300.0);
	'002;Final Speed' = 180.0 (1.0, 300.0);

/GameplayParameters/Powerplays/Dynamic Trigger:
	'000;Use the new dynamic trigger system implementation' = true;
	'001;Extra extent to add on the danger zones' = 0.0 (0.0, 50.0);

/GameplayParameters/Teleporting:
	'000;Teleport distance behind the player' = 100.0 (0.0, 5000.0);
	'001;Teleport distance ahead the player' = 200.0 (0.0, 5000.0);

The !directory: header

The first non-blank line is always !directory: /SomePath. The path is a virtual location in the unified tweak tree — it does not have to match the file's location on disk. For example GameLogic/Catchup.params declares !directory: /Prototype/Catchup, and a per-game-mode file such as GameLogic/GameModes/Elimination.params declares !directory: /RaceSetup. Several different files can therefore contribute to the same /RaceSetup namespace, which is how a single game mode is assembled from a base file plus overrides.

Section groups

Each /path/section: line begins a group; everything indented under it belongs to that group until the next group line. Groups nest by slash, e.g. /RaceSetup/Logic/Respot/TrackCurvature:, so the namespace can be arbitrarily deep. A group whose path is the bare directory root (e.g. /AI/LDC/AISupercar_LDC:) holds the "top level" values for that file.

Value forms

A value line is key = value;, optionally followed by a (min, max) tuning range. The key is either a quoted human label or a bare identifier. Across the 727 files the following value forms appear — all verified against shipped data.

The 'NNN;label' key convention

Many keys are quoted strings of the form 'NNN;Human description', where NNN is a zero-padded ordinal and the text after the semicolon is the designer-facing label. The ordinal is what the code reads (it is a positional index into a struct), while the description is purely for the tweak UI. This is the same grammar the eboot's debug menus use — e.g. Lighting/Debug/GBuffer/a;Disable albedo writes or TrackDamage/Debug/i; Intensity — just with a numeric ordinal instead of a hotkey letter. Other files skip the ordinal entirely and use the full label as the key (e.g. 'Max torque gain' = 0.0 (0.0, 1.0); in Catchup.params), or use bare identifiers with no quotes (HudMessageTime = 2.5 (0.0, 10.0);).

The (min, max) pair is tooling metadata, not gameplay data — it only bounds the QA slider. Floats are written in plain decimal and frequently show the artefacts of 32-bit storage, e.g. 1.5700001 (π/2), 0.6133329, 1.2999999. Treat those as the designer's round number rounded by float precision.

Worked example: an SSAI LDC table

The clearest demonstration of the format driving a real system is a Low-level Driving Config (LDC). SSAI/AISupercar_LDC.params declares !directory: /AI/LDC/AISupercar_LDC and contains one group per speed band — Speed 60.000000, Speed 120.00000, Speed 180.00000 — each with the same 20 indexed variables. How it drives the QA tweak menus

On load the runtime parses each .params file and grafts its !directory: tree into a global tweak hierarchy ("Tweakvars"). Every value line becomes a node: the key supplies the label, the value supplies the current setting, and (min,max) supplies the slider bounds. Booleans become checkboxes; enums become drop-downs; @-keyed entries become buttons that fire the encoded :Label:action:'…' command. The same machinery backs the in-eboot debug menus, whose paths share the identical grammar: 

CameraDirector/Debug/GameEventSequences/Intro
Lighting/Debug/GBuffer/a;Disable albedo writes
TrackDamage/Debug/g; Damage Radius

The QA menu is reachable through the Cheats/qa_menus path seen in the eboot. Because the tree is rebuilt from the files, designers could change a value, save, and reload — the loader strings Load Params / Load Tweaks sit right next to the cPowerplayManager (load) init code.

.params is not the same as default.global_regs. The global registry is a compiled binary file (magic FREG) holding shader register defaults; .params is the editable text tuning layer used by gameplay systems. The two are unrelated formats despite both being "config".

Open questions

  • Whether the parser is whitespace-significant (tabs vs spaces) or tolerant — all shipped files use a single tab for indentation, so this could not be tested from data alone.
  • The exact action-command grammar inside @ entries (:Label:action:'…') — the trailing payload appears to re-state the label, but other verbs besides action may exist.
