Threading & JobStream

Split/Second spreads work across the PS3's PPU threads and its six available SPUs using an in-house job system called JobStream (libs/System/JobStream), layered directly on Sony's SPURS runtime. A small number of PPU threads — a main/game thread plus dedicated loader threads — drive gameplay and submit SPU job chains for the heavy parallel work: deferred lighting (SPULight), shader patching, geometry decompression (Sony Edge) and Havok physics. Everything below is taken from the assert/source strings in EBOOT.BIN (QA debug build NPXX00575).

SPURS foundation

SPURS (the SPU Runtime System) owns the SPU thread group; JobStream creates job chains on top of it. The chain attribute set-up is fully visible in the source paths, and two call sites pin down the exact parameters:

// libs/System/JobStream/JobStream/Spurs/SpursJobChain.cpp
cellSpursJobChainAttributeInitialize(
    &attr, mCommandList, mJobSize, mMaxGrab, mPriorities,
    mNumSPU, true, 0, 1, false, 256, 1 );
cellSpursCreateJobChainWithAttribute( mSpurs, mJobChain, &attr );

// Crayon2 shader-patch chain (ShaderPatching.cpp) — concrete values:
cellSpursJobChainAttributeInitialize(
    &attr, mCommandList, sizeof( CellSpursJob128 ), 16,
    spuPriorities, spuCount, true, 0, 1, false, 256, 1 );

From these we know each job is a CellSpursJob128 (a 128-byte job descriptor), a SPU grabs up to 16 jobs at a time (mMaxGrab = 16), and the chain runs across spuCount SPUs with a per-SPU spuPriorities array. Chains are run, shut down and joined explicitly:

SPURS callRole in JobStream
cellSpursRunJobChain( mJobChain )Start a chain executing on the SPUs
cellSpursJobGuardInitialize(…) / cellSpursJobGuardNotify(…)Barrier: PPU/other jobs wait until a count of jobs complete
cellSpursShutdownJobChain → cellSpursJoinJobChainTear down and join a finished chain
cellSpursJobGetInputList(&job->workArea.dmaList[n], …)Build the per-job DMA input list (e.g. shader + patch data)

Event handling thread

Chain completion is delivered through a dedicated PPU helper thread, spurs_job_event_handler_%d, created with an 8 KB stack:

// SpursJobEventHandler.cpp
sys_ppu_thread_create( &mThread, ThreadHandler, (u64)(uintptr_t)this,
    mPriority, (1024 * 8), SYS_PPU_THREAD_CREATE_JOINABLE, threadName );
sys_event_queue_create( &mQueue, &attr, SYS_EVENT_QUEUE_LOCAL, 127 );
cellSpursAttachLv2EventQueue( mSpurs, mQueue, &mEventPort, true );

It attaches an LV2 event queue (depth 127) to SPURS and blocks in sys_event_queue_receive, signalling the rest of the engine when SPU jobs finish. A terminating event port (sys_event_port_send( mTerminatingPort, … )) lets the main thread shut it down cleanly.

There are two job-dispatch abstractions in the build: JobStream::cJobScheduler / cJobThread (the SPURS job-chain wrapper) and a higher-level cJobManager with PPU worker threads (cJobManager::cWorkerThread). "Job Manager" appears as a profiled subsystem under both the in-game and front-end controllers, so it is a long-lived service, not a per-frame object.

PPU threads

The profiler thread/category table in the EBOOT names the long-running PPU threads. The build calls initialized profiling for thread %s (%p) as each one starts. The notable ones:

Thread / subsystemRole
Main / game threadRuns the frame: FrameBegin → Update → FrameEnd, then RenderToTexture / RenderAlpha / RenderDebug. Submits SPU job chains.
Loading ThreadGeneric background loader (Background Loader)
Level (Loader Thread)Streams the current level / track data while racing
Front End (Loader Thread)Loads front-end / menu resources independently of gameplay
RenderingRender submission / GPU pacing (cRenderSynchronizer, C2:EndScene, C2:Wait)
spu_printf_handler_%dPPU thread servicing SPU printf output (debug build)
spurs_job_event_handler_%dSPURS completion event pump (see above)

Splitting the loaders into a Level loader and a Front End loader is deliberate: the front-end (menus, car select, season screen) can stream its assets on its own thread while the level loader continues to bring in track data, so the game can transition between the menu and a race without one starving the other. The EBOOT confirms both as distinct profiled threads (Level (Loader Thread), Front End (Loader Thread)), alongside a plain LoaderThread token and a Loading Thread.

Note the division of labour on SPURS: JobStream uses SPURS job chains (cellSpursCreateJobChainWithAttribute), while Havok plugs in through SPURS tasksets (cellSpursCreateTaskset / CreateTask). Both share the same underlying cellSpurs instance and event-queue plumbing.

Sony Edge

The geometry-processing SPU library is Sony's Edge middleware (SPU libs), version v0_4_3, used here as EdgeGeom. Its three core source files — edgegeom.cpp, edgegeom_compress.cpp, edgegeom_decompress.cpp — appear in the assert paths, confirming Split/Second decompresses (and likely back-face/cluster culls) vehicle and environment geometry on SPUs as a JobStream job before handing index/vertex data to the RSX. This is why .model geometry is stored Edge-compressed.

Window of opportunity

The "SPULight Window of opportunity" marker captures the core design idea of the whole system: while the SPUs chew through a job chain, the PPU main thread is free to do other work and only blocks (SPULight Stall Global Light) when it actually needs the result. The job-guard barriers (cellSpursJobGuardNotify) are what let the PPU defer that wait as late as possible, maximising PPU/SPU overlap each frame.

Source-path evidence: c:/Projects/proj_1/ws_0/Code/libs/System/JobStream/JobStream/Spurs/SpursJobChain.cpp (JobStream) and E:/SplitSecond/ss_dev/Code/contrib/Havok/Havok550/Common/Base/Spu/Util/Ps3/hkPs3SpuUtil.cpp (Havok's SPU util — note the alternate E:/SplitSecond/ss_dev/ tree for contrib code). The job-system structure is Confirmed; the exact SPU count and per-job priorities are Partial (passed as runtime variables spuCount/spuPriorities, not constants in the strings).
