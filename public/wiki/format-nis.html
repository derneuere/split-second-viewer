Sequences (.nis / .gbx)

Two small binary formats sit in the track tree rather than the movie tree. Despite the "sequence"-sounding extensions, investigation of the actual files shows neither is a cutscene script: .nis is the AITrack/TrackLogic route table (a short list of named track/event zones with a per-zone flag) and .gbx is a small big-endian light-rig override table (extra ambient-light boxes and prop draw-distance tweaks per time-of-day rig), most of whose instances are empty.

The cluster brief speculated .nis = "Non-Interactive Sequence (cutscene script)" and .gbx = unknown. The on-disk evidence does not support a cutscene reading — every .nis lives under Event/RACING/TrackLogic/<route>/, the EBOOT loads it via a TrackLogic subsystem, and its contents are track/event-zone codes, not animation/camera commands. This page documents what the files actually contain.

Where it lives & what it is for

Each playable level has an Event/RACING/TrackLogic/ folder containing one or more route variants in sub-folders A, B, C, D — these are the alternate paths a Split/Second track can branch into when powerplays reshape the circuit. Every route folder holds exactly one Track.nis, loaded alongside the other per-route tables (Track.checkpoints, Track.linkorigins, Track.logicinfo, Track.lip). The EBOOT confirms the path, the owning subsystem, and — crucially — what "NIS" denotes here: it is an AITrack route/identifier file, not a cutscene:

%s/Track.nis · environment(noassert)|Event/%s/TrackLogic · environment(noassert)|Event/%s/TrackLogic/%s · TrackLogic_Error/Warning/Info · AITrackData NIS file '%s' is corrupt · AITrack route identifier file is empty, '%s' · AITrack data file is empty, '%s'

The file is a flat list: a record per track zone, each pairing a 16-bit id with a NUL-terminated ASCII zone code and a one-byte flag. The zone codes are the engine's own named zones — some are short sector codes (PA03-1, PP13F-2, RCA2-4, HC_01) but many are descriptive English names for powerplay set-pieces (e.g. Powerplant uses 1st_Ceiling_01, 2nd_Cooling_01, Chimney_01, Door_01; Graveyard uses PP16_RockFall_*). A .nis therefore reads as "for this route, here are the track/event zones that participate, and a per-zone flag."

File layout

The header is just two bytes; records pack immediately after with no padding. The structure was verified by parsing the full byte stream of three samples to completion — there are zero leftover bytes, and the record count in byte 1 matches the parsed record count exactly for all 18 files.

OffsetLengthTypeNameDescription
0x000x1charmagic0x69 = ASCII 'i'. Constant across all 18 files. (Same "i" letter as the Bink codec but unrelated — different format.)
0x010x1uint8_trecord_countNumber of segment records that follow. Verified equal to the parsed record count for every file (e.g. Downtown/A = 10, docks/A = 27, docks/D = 34). 0 for empty Nemesis tracks.
0x02variesrecord[n]recordsrecord_count back-to-back segment records (see below).

Each record:

OffsetLengthTypeNameDescription
+0x00x2uint16_tsegment_idBig-endian segment/zone id. Increases roughly in track order (Downtown/A: 1, 2, 6, 8, 9, 10, 11, 12, 13, 16). Stable references into the level's section list rather than a dense 0..n index.
+0x2varieschar[]zone_codeNUL-terminated ASCII zone name. Often a short sector code ("PA03-1", "PP13F-2", "RCA2-4", "HC_01") but sometimes a long descriptive name ("2nd_Cooling_01", "PP16_RockFall_Small_1"). Length ranges roughly 6–22 bytes incl. terminator.
+n0x1uint8_tflagPer-zone boolean, only ever 0 or 1 across all 181 records parsed (79× 0, 102× 1) — read as enabled / active-on-this-route.

Every Nemesis boss track (nem_*) ships a 2-byte 69 00 stub: magic + zero records. Boss chases are linear scripted pursuits with no powerplay route-branching, so they need no TrackLogic segment list — a clean corroboration that the file drives route/segment logic rather than playing a cutscene.

Each level's LightRigs/ folder holds one sub-folder per time-of-day preset (midday, sunrise, sunset, twilight, nighttime, midday_v02, …). Inside each preset folder, the .gbx sits alongside the files that carry the bulk lighting data:

An earlier draft of this page claimed every .gbx is "exactly four bytes, all zero" and a registration stub. That is wrong. Of the 18 .gbx files, 13 are empty (4 bytes, record_count = 0) but 5 carry real structured data (212–491 bytes): Downtown/sunset, docks/midday, Graveyard/midday, Graveyard/sunset and airport_test_03/sunrise.

File layout

The .gbx is a big-endian table of named light-rig overrides. The header is a single big-endian uint32 record count; each record is a pair of {32-bit hash, length-prefixed ASCII type-name} + {32-bit hash, length-prefixed ASCII instance-name} followed by a run of big-endian float32 values (positions / parameters). The two type-name strings observed — "Ambient Light" and "Prop Draw Distance" — are both real engine strings present in the EBOOT, confirming the file feeds the lighting/prop systems, not a stub.

OffsetLengthTypeNameDescription
0x000x4uint32_t (BE)record_countNumber of override records. 0 in 13 files (the 4-byte ones); 2–5 in the populated files. Confirmed
0x04…variesrecord[n]recordsPer record: uint32 type-hash, uint32 str-len + ASCII type ("Ambient Light" / "Prop Draw Distance"), uint32 name-hash, uint32 str-len + ASCII instance name, then big-endian float32 fields. Partial (float field count/meaning not fully mapped).

Graveyard/midday goes further, with "Prop Draw Distance" records keyed to powerplay props such as PP16_RockFall_Small_1 and PP04_FlatBedTruck — i.e. per-rig, per-prop draw-distance overrides for that time of day. So .gbx is a small per-light-rig override/extras table (extra ambient-light boxes and prop draw-distance tweaks) layered on top of the bulk .lights data; the empty 4-byte files simply have no overrides for that preset.

The EBOOT references the .gbx alongside the rest of the rig folder:

environment(noassert)|LightRigs/%s/%s.gbx · %sLightRigs/%s/%s.lights · %sLightRigs/%s/%s.params · Ambient Light · Prop Draw Distance · game_setup/subtracklightrigIndex · Environments\SubTrackLightRigs.txt · Env '%s' : SubTrack '%s' : LightRig '%s'

The engine selects a SubTrack × LightRig combo (e.g. SubTrack "A" + LightRig "midday") and loads that rig folder by name, reading .lights, .params and the .gbx overrides together.

Naming aside: .gbx is also the extension Nadeo's TrackMania uses for its "GameBox" files, but that is unrelated — Split/Second is a Black Rock title with its own engine, and these .gbx files share none of the GameBox structure (which begins with a GBX ASCII magic; these begin with a big-endian record count).

Confidence

Confirmed: the .nis magic, record-count byte, and per-record id / string / flag layout — verified by parsing all 18 files to completion with no residual bytes and matching the count field (181 records total, flag only ever 0/1). The .gbx big-endian record_count header and the length-prefixed type/name string structure are also Confirmed (the type strings "Ambient Light" / "Prop Draw Distance" match EBOOT strings). Partial: the semantic meaning of the .nis segment_id values and the exact role of its flag; and the count/meaning of the trailing float32 fields in each .gbx record. Theory: the zone-code family meanings. We can firmly rule out the original "cutscene script" theory for both formats — their location (TrackLogic / LightRigs), EBOOT references and contents are unambiguous.

Open questions

  • Do .nis segment_id values index the level's sectorInfo / route tables, or a TrackLogic-internal section list?
  • Is the .nis flag "active on this route" or "is a branch/junction" — does route B's .nis describe the divergent path only?
  • How many float32 fields follow each .gbx record, and what do they encode (box centre + extents for "Ambient Light"; a single distance for "Prop Draw Distance")? Are there .gbx record types beyond the two observed?
