Physics Packs (.hkPPs / .hkPPsXml)

The .hkPPs ("Physics Pack") is the largest per-subtrack physics asset: a single Havok 5.5 packfile whose root container bundles eight named sub-systems — the dynamic-physics world, gameplay trigger phantoms, area-of-effect (explosion) volumes, animated rigid bodies and their animation containers, plus MOPP-accelerated collision meshes. It is the runtime backbone that connects the Catnip / Powerplay scripting layer to physics. Each file ships next to a human-readable XML twin (.hkPPsXml).

Where it lives & how it loads

One .hkPPs exists per subtrack (the A/B/C/D directional/seasonal variants of a level), beside the matching .hkRBs and the route data. The EBOOT loads it from the path template

environment|Subtracks/%s/%s.hkPPs

and reports it through the generic packfile loader (HK PACKFILE %s: SIZE = %d kb). The binaries are sizeable — Downtown subtrack A is 2.88 MB (3,021,456 bytes; its XML twin is 11.3 MB / 11,299,746 bytes), versus the much smaller .hkRBs at 140 KB — because the pack carries the level's MOPP collision geometry in addition to the gameplay phantoms.

The environment| prefix is the resource-residency tag used throughout the build (see Resources & Streaming). %s is the subtrack letter / level name.

Container layout

Like every Havok asset in the game this is a standard packfile: a fixed 57 E0 E0 57 header, the Havok-5.5.0-r1 version string, and the three serialized sections __classnames__, __types__, __data__. The header is byte-for-byte identical to .phys, .hkColl and the other Havok files through offset 0x25 (verified identical across every environment .hkPPs/.hkRBs/.hkRBXs/.hkHBXs sample); the first byte that varies between files is the 32-bit field at 0x24 — the Havok header's contentsClassNameSectionOffset (its value fits in 16 bits, so only the word at 0x26–0x27 changes). This is a class-name section offset, not a data or file size. See Havok Packfiles for the full header walk-through.

First 0x50 of Downtown/Subtracks/A/Downtown.hkPPs. Bytes 0x00–0x25 are byte-identical across every .hkPPs, .hkRBs, .hkRBXs and .hkHBXs sample examined; the only varying header field is the contentsClassNameSectionOffset word at 0x26–0x27 (e.g. 0x350 for Downtown A and docks A, 0x2b0 for Downtown B). The ASCII Havok-5.5.0-r1 begins at 0x28.

Root container — the eight variants

Reading the __data__ section of the XML twin, the hkRootLevelContainer exposes eight namedVariants. This is what makes the "Pack" a pack: a normal Havok scene would carry only hkpPhysicsData; Black Rock appends six custom export user-data containers that bind Havok objects back to game events.

#classNameRole
0hkpPhysicsDataThe physics world: an array of hkpPhysicsSystem (72 in Downtown A) holding rigid bodies, phantoms and collision shapes.
1HavokTriggerPhantomExportUserDataContainerBinds each trigger phantom to a Catnip event name (see below).
2HavokTriggerSystemExportUserDataSystem-level wrapper for the trigger phantom system.
3HavokAreaOfEffectExportUserDataContainerExplosion / force volumes with impulse + duration data.
4HavokAreaOfEffectSystemExportUserDataSystem-level wrapper for the AOE system.
5HavokAnimatedRigidBodyExportUserDataContainerKeyframed / scripted moving bodies (drawbridges, gates, kinematic powerplay pieces).
6HavokAnimatedSystemExportUserDataContainerSystem-level wrapper for the animated bodies.
7HavokAnimationContainerContainerThe hkaAnimation clips driving variant 5 (Havok Animation, Havok 5.5).

Trigger phantoms

Each HavokTriggerPhantomExportUserData record pairs a Havok phantom volume with a gameplay event. The container in Downtown A holds 37 of them; a typical record looks like:

<hkobject>
  <hkparam name="type">TRIGGER_PHANTOM_TYPE</hkparam>
  <hkparam name="trigger">#0198</hkparam>            ← ref to the hkpRigidBody/phantom
  <hkparam name="eventName">1249478979.1976132399</hkparam>   ← Catnip event id pair
  <hkparam name="triggeredBy"></hkparam>
  <hkparam name="triggeredByDelay">0.000000</hkparam>
</hkobject>

The eventName values are the same dotted <crc>.<crc> hashed-name pairs used as rigid-body names (see .hkRBs); they resolve into the Catnip trigger graph that fires powerplays.

Area-of-Effect (explosion) volumes

The HavokAreaOfEffectExportUserData records (75 in Downtown A) describe the radial force volumes that throw cars during powerplay detonations. Confirmed fields:

FieldTypeDescription
typeenumAlways AREA_OF_EFFECT_TYPE in samples.
arearefThe Havok phantom/shape the effect is bound to.
eventNamestringComma-separated list of Catnip event-id pairs that fire it.
centerImpulsefloat32_tForce applied at the centre of the volume.
boundaryImpulsefloat32_tForce at the boundary (falls off to here).
delayfloat32_tSeconds before the impulse fires.
loopedboolWhether the AOE re-applies each frame.
durationfloat32_tHow long the volume stays active.

These map directly to the Area of Effects system and the EBOOT's HavokAreaOfEffectExportUserDataContainer class.

Collision geometry inside the pack

Beyond the gameplay phantoms, the __data__ section also stores real collision shapes: hkpMoppCode (the MOPP mid-phase BV tree) paired with hkpStorageExtendedMeshShape (the triangle soup). This is the same MOPP+mesh pairing documented for .hkColl / .mainColl; here it travels with the dynamic objects so a subtrack can be streamed as one physics unit.

A quick way to tell the four environment-physics formats apart without reading the data is the root container's variant list: .hkPPs has 8 variants (triggers + AOE + animation), .hkRBs has 4 (furniture/single/compound bodies), .hkRBXs has 3 (respot boxes) and .hkHBXs has 3 (handling boxes).

Confidence

Confirmed for the container format, root-variant list and the trigger / AOE record fields — every claim here is read straight from the XML twin and corroborated by the EBOOT class names and load path. The exact binary byte offsets of each custom user-data struct inside __data__ are Partial: they are reconstructible from the __types__ reflection but were verified through the XML rather than by hand-parsing the packed blob.
