Rendering Pipeline

Split/Second runs a deferred renderer built on Black Rock Studio's in-house render layer, codenamed Crayon2 (libs/Render/Support/Crayon2). Geometry is rasterised once into a multi-channel G-Buffer, screen-space ambient occlusion and shadow masks are generated, lighting is accumulated on the PS3's SPUs ("SPULight" jobs), and the result is tone-mapped through an HDR post chain (bloom, motion blur, edge, colour-cube, lens flare). Every fact and value on this page is taken directly from the loose data files and the EBOOT.BIN strings of the QA debug build (NPXX00575), built Jan 30 2010 at 02:33:01.

Frame structure

The per-frame render is bracketed by the FrameBegin / Update / FrameEnd phases (driven from the main thread), with the GPU command list closed at C2:EndScene and the CPU waiting at C2:Wait. The EBOOT's profiler timer table enumerates the render passes in execution order; the major ones are:

The render runs entirely in big-endian on the RSX through Sony's Cell GCM low-level driver (cGPUCallbackManager, cRenderSynchronizer). The debug build reports the display through a Monitor info dump and supports Display Resolution: %d x %d with 4:3 / 16:9 aspect detection.

The G-Buffer

The geometry pass writes into a packed set of render targets. The EBOOT exposes the channel layout both as named targets (g_color_0, g_color_1, g_color_2, g_depth) and as a debug menu that can disable each MRT write individually (Lighting/Debug/GBuffer/a…e). Cross-referencing those gives the channel assignment:

MRTContentsDebug toggleFullscreen visualiser
g_color_0Albedo (base colour)GBuffer/a; Disable albedo writesFullscreen - Albedo
g_color_1"Unlit" (emissive / un-lit colour)GBuffer/b; Disable unlit writesFullscreen - Unlit
g_color_2Normals (+ packed data)GBuffer/c; Disable normal writesFullscreen - Normals
(packed)Motion-blur velocity IDs (gbuffer_MotionID, MotionBlurID)GBuffer/d; Disable motion blur writesFullscreen - Motion Vector IDs / Motion Vectors
(packed)Specular intensity & powerGBuffer/e; Disable specular writesFullscreen - Specular Intensity / Specular Power
g_depthScene depth—Fullscreen - Depth

This matches the five canonical channels the brief expected: albedo, unlit, normal, motion-blur, specular. The screen-space masks add three more visualisers (Fullscreen - SSMask0/1/2) — the depth-shadow and particle-shadow masks. The GPU also keeps a GBuffer_RT resolve target and a separate ss_deferred_gbuffer_globalFace pass for the global directional contribution.

The G-Buffer lives in RSX tiled memory (cTiledMemoryManager, "Tiled: %d"). Because the deferred lighting reads depth, the engine explicitly restores depth and ZCull after the MRT resolve (GBuffer Depth Restore, GBuffer ZCull Restore, plus a Z-Cull Reload Helper renderable) — ZCull state is destroyed by the resolve and must be rebuilt for the later forward passes.

SSAO

Screen-space ambient occlusion is generated immediately after the MRT resolve (GBuffer SSAO → GBuffer Blur, shaders GenerateSSAO and BlurSSAO). The QA "lighting/SSAO" debug page exposes the full tuning set:

SPU lighting (SPULight)

The headline feature of the renderer: local lights are culled, batched and accumulated on the SPUs rather than on the RSX. The deferred lighting stage splits lights into three buckets and reports the split each frame:

Light LODs viewport %d, num lights %d, combined %d, full %d, qtr %d, spu %d

So a viewport's lights are partitioned into combined, full-res, quarter-res and SPU sets. The SPU path is gated behind Lighting/LocalLights/SPULight/Enabled and is timed by these markers:

SPULight markerMeaning
SPULight Build JobsMain thread assembles the light job chain
SPULight Copy DepthCopies the depth buffer to LS-friendly source (spu_input_depth)
SPULight Stall Kick JobsKicks the SPU job chain
SPULight Stall Global LightMain thread blocks on the global-light result
SPULight Window of opportunityTime budget during which SPUs run lighting while the PPU does other work

The SPU jobs themselves are named JobLight0…JobLight16 (spot/combined lights) and JobPointLight1…JobPointLight16 (point lights) — i.e. up to 16 of each per chain. Their inputs/outputs are the LS buffers spu_input_depth, spu_input_normal, spu_output and spu_output_blank. Whatever the SPUs cannot take is rendered on the RSX as full- or quarter-res light volumes, using the geometry shaders PointLight / PointLight_Low_LOD, SpotLight / SpotLight_Low_LOD and the vehicle headlight shader VehicleLight / VehicleLight_Low_LOD (with a projected vehicle_light_gobo_tex).

The combined-light shaders are generated per light count: CombinedLight_%dSpot_%dPoint (and …_QtrResLocal / …_Frag1 variants). This is classic Crayon2 shader patching — the engine builds the exact permutation it needs at runtime from the übershader DB instead of shipping thousands of pre-compiled programs.

Lighting register inputs

Deferred lighting is driven by a fixed set of shader constants (visible as the deferred_lighting_* register names), including deferred_lighting_light_pos_range, deferred_lighting_light_dir_angle, deferred_lighting_light_specular, deferred_lighting_light_falloffScale, deferred_lighting_shadow_fade, deferred_lighting_InvViewProj and the three G-Buffer samplers deferred_lighting_Target0/1/2TexPS3. Defaults for the global registers live in default.global_regs (e.g. light_ambient, light_diffuse, light_specular).

Shadows & light scattering

Shadows use Cascaded Shadow Maps (the QA Lighting/CSM page exposes "Visualise splits", "Stable projection", "Last split is fixed projection", a "Quality level" and per-split frustum-test counters such as Split: %d - %d/%d - Dist: %.1f m). They are projected through the screen-space mask passes (SSMask 1st/2nd Depth Shadow, shadow_projection, shadow_edge). Volumetric atmospherics are a separate deferred pass: DeferredLightScattering_Extinction / _Inscattering (and ATV variants) driven by lightscattering_inscattering_color_from/to, …_per_metre, lightscattering_extinction_per_metre and lightscattering_vLightDir.

HDR & post-processing

The lit scene is rendered in HDR (RenderHDRToHDR; the front-end uses RenderHDRToDefault) and resolved down through a tone-mapper and a stack of screen-space effects. The order of the post passes (from the profiler timers) is Bloom → Motion Blur → Edge → Depth of Field → ColorCube → Low Res → Lens Flare → Particle Flare. Each effect is data-driven from PostProcessDatabase and the global Generic/Params/PostProcess.params. Real default values from those files:

Tone mapping (HDR)

The screen-space tone-mapper (SSToneMapper) does auto-exposure with a configurable key and luminance clamp. From the Area Of Effect preset in PostProcessDatabase.params:

/PostProcessDatabase/Area Of Effect/SSToneMapper:
    'a; Enabled' = true;
    'b; White Luminance' = 0.8867188 (0.0, 5.0);
    'c; Exposure Key' = 0.3789063 (0.0, 5.0);
    'd; Exposure Speed' = 0.05 (0.0, 1.0);
    'f; Min Average Luminance' = 0.2 (0.0, 5.0);
    'g; Max Average Luminance' = 5.0 (0.0, 5.0);
    'h; Average Luminance Blur Passes' = 4 (0, 10);
    'i; Filmic Enabled' = false;
    'j; Filmic Toe' = 1.5  'k; Filmic Shoulder' = 0.3;

Auto-exposure measures average scene luminance by repeatedly blurring it down (Average Luminance Blur Passes = 4); a filmic tone curve (Toe / Shoulder) is available but off by default in this preset. The crash/takedown camera preset pushes white luminance up to 2.3984375 and the exposure key to 0.5507813 for its overexposed look.

Bloom

Bloom (SSBloom) is a thresholded bright-pass blurred and added back (SSBloomBrightPassExp → SSBloomAdd → Render_BloomWithToneMapping):

/PostProcessDatabase/Area Of Effect/SSBloom:
    'a; Enabled' = true;
    'b; Threshold' = 0.941406 (0.0, 5.0);
    'c; Tone_r/g/b' = 2.353 / 2.581 / 3.0 (0.0, 500.0);
    'd; Blur Passes' = 1 (1, 6);
    'e; Blur Amount' = 4.0 (1.0, 10.0);
    'f; Curve' = 1.0 (0.5, 10.0);

Motion blur

Per-object/camera motion blur (SSMotionBlur) reads the motion-ID channel and a camera matrix (ss_deferred_motion_blur_matMotionPS3). Default tuning:

/PostProcess/SSMotionBlur:        (global default)
    'a; Enabled' = true;
    'b; Velocity Multiplier' = 8.0 (0.0, 10.0);
    'c; Blur Amount' = 0.5 (0.0, 3.0);

The in-game Area Of Effect preset raises the blur amount to 1.25 for a stronger sense of speed.

Edge post & colour grading

The edge pass (Post Processing Edge Render) is the stylised ink outline. It is on by default in the global chain but off in most per-area presets:

/PostProcess/EdgeRender:           (global default)
    'a; Edge Render Enabled' = true;
    'b; Edge Render Color Scale' = 4.5 (0.0, 10.0);
    'c; Edge Render Width' = 0.0023399 (0.0, 1.0);
    'f; Edge Render Tone' = (0.133, 0.133, 1.0);

Colour grading is a 3D LUT lookup (ColorCube, ColorCubeRenderTarget, shader Blend_ColorCubes) that blends between two cube textures, e.g. color_cube_airport_test_03 blended with color_cube_CRASHCAM_airport_test_03. The chain finishes with Vignette, a screen Gradient, depth-of-field (DoF, focus near/mid/far planes) and the SSLensFlare / SSParticleFlare sun and particle flares. A per-area "Overwrite … Configuration" flag set decides which effects an area-of-effect trigger overrides versus inherits from the global default.

Each effect can be authored per-camera as a behaviour with start/end keys for cinematic transitions: cBloomBehaviorData / cBloomLinearBehaviorData, cSSMotionBlurBehaviorData, cDOFBehaviorData / cDOFLinearBehaviorData, cColorCubeBehaviorData, all loaded under Cameras/Data/%s/…. See Cameras & Director.

Anti-aliasing

The final resolve supports several RSX multisampling modes, selectable in the Lighting/Debug menu and named in the shader DB:

ModeShader
2× QuincunxMultiSampledAntiAliasing2X_Quincunx (+ alt / colour-only)
2× AccuviewMultiSampledAntiAliasing2X_Accuview
4×MultiSampledAntiAliasing4X (+ _offsetMSAA)

The deferred path resolves the MSAA fragments individually (Lighting/Debug/e;MSAA Fragment 0, f;MSAA Fragment 1, PerformDeferred_fragment_%d, GlobalLightPass_Frag0/1) and projects shadows over the second fragment (MSAA Project shadows over Frag 1). MSAA sample offsets are tunable via Renderer/MSAAOffset and Renderer/MSAAResolveOffset.

Crayon2 & the übershader

All materials are resolved against a single compiled effect database, global_shader_combos.fxc (2,382,848 bytes ≈ 2.27 MB) with an index global_shader_combos.shaders (194,958 bytes), both loaded from data|global_shader_combos/…. At runtime the cShaderPatcher patches register blocks into the right combo (Crayon2_Init → cMaterialComboManagers / cMaterialComboFactory); a missing material is logged as "Can't find material combo with CRC 0x%08x.. replacing with error combo!". Shader patching is itself offloaded to an SPU job (see Threading & JobStream). See FX Compiled (.fxc) and Shaders for the on-disk layout.
