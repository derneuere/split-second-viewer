FX Compiled (.fxc)

The .fxc ("FX compiled") file is the compiled shader microcode half of the Crayon2 material system. It holds the RSX vertex- and fragment-program binaries — the "clean shader" the .shaders register-block sets patch — together with the named constant/sampler symbol table that says where each uniform lives. It sits beside the renderable's .model and is paired exactly 1:1 with its .shaders twin (2357 of each on disk; there are slightly more .model files, 2425, since some meshes share a material set).

Where it sits

The loader resolves three sibling paths for every body — "can't find model,fxc or shaders for single body instance %s" (eboot_strings.txt). The .fxc supplies the SPURS shader patcher's first DMA input, info.mCleanShader (see .shaders for the full patch-job listing). In other words: .fxc = the program binary; .shaders = the relocation/patch table that maps engine uniform names onto the program's registers. The master copy of every combo is global_shader_combos/global_shader_combos.fxc (2.27 MB, 263 combos) — the übershader database referenced directly in the EBOOT as data|global_shader_combos/global_shader_combos.fxc.

Header (0x14 bytes + name)

The header mirrors the .shaders header field-for-field, which is how the two are matched. All integers are big-endian.

OffsetLengthTypeNameDescription
0x00x4char[4]magic"\0FXC" (00 46 58 43 — leading NUL byte). Constant across all 2357 samples (vehicle, wheel, env materials.fxc, global DB).
0x40x4uint32_tversion1 in every observed file. Format version.
0x80x4uint32_tcombo_countNumber of compiled shader/material combos. Verified equal to the count field of the paired .shaders (Musclecar_01 = 23, _WheelBack = 9, global DB = 263).
0xC0x4uint32_tname_lenLength of the set-name string incl. terminator (0x0B = 11 here).
0x10name_lenchar[]set_nameThe set-level CRC, e.g. "0xe4249692". Byte-identical to the name in the paired .shaders — this string is the join key between the .fxc and .shaders twins.

Per-combo body

After the header, the file lays out combo_count compiled combos back-to-back. Each combo begins with a small constant/symbol table (a list of named uniforms with their register slots), then the RSX microcode for the vertex and fragment stages, terminated by an "END" tag. There is no top-level offset table; the constant table for combo n is what lets the patcher find combo n's registers without walking the binary.

Constant / symbol table

Each combo opens with a u32 symbol count followed by that many symbol records. Names seen at the head of a combo include the global engine uniforms Trans, TransViewProj, NormalTrans, WorldCameraPos, WorldCameraRight, WorldCameraFront, WorldCameraUp, anim_frame_count, cloudshadow_Cloud_Shadow_MatU/MatV, lightscattering_*, shadowheightmap_*, and the material-specific VehiclePaintAndDamage_scrapePlane* / scrapeLevels* set. Each record is roughly { several u32 metadata words; u32 strlen; char[] name } and is immediately followed by the hardware register name it maps to — the constant-register tokens C19, C20, C21, … (e.g. WorldCameraPos→C19, WorldCameraRight→C20). The exact width of the metadata word group varies between the first entry and the rest, so it is left Partial. These are the same symbol names the .shaders register blocks reference — confirming the two files describe one program from two angles.

RSX microcode

The vertex program declares its inputs by Cg attribute semantic — ATTR0 → i.position_half_float_0, ATTR1 → i.half_float_0, ATTR2/3 → further half-float streams (matching the half-float vertex layout used by the .model). Sampler bindings appear as TEXUNIT0… and output semantics as COLOR0. The bulk of each combo is then 16-byte (4-word) NV40/RSX fragment-program instruction words — the recurring 04 18 00 00 0c b8 00 00 10 06 ff ff ff ff … pattern is the opcode/dst/src encoding of RSX fragment microcode. The whole file is closed by the ASCII tag "END".

The RSX microcode itself is platform GPU bytecode (NV40/RSX fragment + vertex programs). Decoding individual opcodes is out of scope and marked Opaque; the container framing (header, per-combo symbol tables, the ATTR*/TEXUNIT*/COLOR0/END markers) is the documented part.
How a combo's symbols map onto registers, samplers (TEXUNIT*) and the .textures CRCs a material binds — the material / texture binding scheme — is documented on the .shaders page, since the join keys (set-name CRC, per-combo CRC, texture CRC) live in both files. The ATTR0 → i.position_half_float_0 / ATTR1 → i.half_float_0 vertex inputs correspond to the half-float P4 vertex layout the .model auto-detect recovers.

.fxc vs .shaders — division of labour

OffsetLengthTypeNameDescription
——file.fxcCompiled microcode ("clean shader") + symbol table. Microcode slots are full.
——file.shadersRegister-block / patch tables only. Microcode SDRS…SDRE slots are empty (length 0).

Because the count and the set-name string are identical between the pair, you can validate a dump by checking those two fields agree. The übershader pair global_shader_combos.{fxc,shaders} (CRC 0xdfe85d60, 263 combos) is the authoritative library every per-car set is sliced from.
