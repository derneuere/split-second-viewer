Sectors (.sectorInfo)

Each level ships exactly one <Level>.sectorInfo under Environments/Levels/<Level>/Sectors/. It is the level's sector database: the spatial partition that the engine uses to stream geometry and textures, cull renderables, run occlusion queries and answer gameplay questions like "is the victim in a loaded sector?". It is the data behind the EBOOT's Sector Streaming, Sector Visibility Update and Streaming Sector Render systems.

Do not confuse the sector partition (streaming/rendering, this page) with the route section (a run of AI links). They are independent partitions of the same level — sectors are volumetric and used by the renderer/streamer; sections are 1-D and used by lap timing and the AI.

What a sector is

The Sectors subsystem (EBOOT page Sector Debug Page, tweakables under Sectors/Streaming/*, Sectors/Culling/*, Sectors/Occlusion/*, Sectors/Visibility/*) treats the world as a grid/graph of sectors, each with sub-sectors. The runtime keeps a visibility map (Sector From → Sector To adjacency) so that standing in one sector it knows which others can be seen and must be resident. Key tunables that this file feeds:

  • Sectors/Streaming/e; Max Sector Stream Dist — how far ahead sectors are pulled in.
  • Sectors/Streaming/i; Look Ahead Dist / j; Look Ahead Damp — predictive streaming along the route.
  • Sectors/Culling/Cull Renderables + Cull Shadow Caster Renderables — per-sector cull lists.
  • Sectors/Occlusion/Use Occlusion Queries — per-sector occluder geometry.
  • CameraDirector/.../Use Victim Is In Loaded Sector Condition — gameplay reads sector residency.

The runtime log line Sector %d %s subsectors visible %d invisible %d confirms the sector → sub-sector hierarchy and the visible/invisible bookkeeping.

File layout

Partial The file opens with a small fixed header, an embedded source-asset path, then a long run of q###-tagged chunks (one per sector). Strings are stored as UTF-16LE (little-endian, unusual for this big-endian title), while geometry/counts are big-endian. The leading header carries two constants shared by every level (1.9200, 300.0) followed by a big-endian sector count that — verified across five levels — equals the number of q### tags in the file (the one solidly-pinned header field).

Sector chunks (q###)

Theory Sector records are introduced by an ASCII tag q000, q001, … (Downtown contains exactly 128 such tags — matching the header sectorCount — though the numeric ids are not contiguous: they run up to q132, so a few indices are skipped). Between tags each record carries: a block of big-endian floats that read as an axis-aligned bounding box / extents (the values -1394.3, 148.8, -1591.7 … are world-space and within the level's coordinate range seen in Track.debug.txt), several small big-endian integer counts (00 00 00 04, 00 00 00 01 …) and a UTF-16LE name fragment. The integer arrays are almost certainly the per-sector renderable / occluder / adjacency index lists referenced by the culling tunables. The exact field order inside a chunk is not yet fully resolved.

OffsetLengthTypeNameDescription
+0x00x4char[4]tag"q###" sector id (3-digit, ids may be sparse — Downtown's 128 sectors run q000…q132).
+0x40x18float32_t[6]aabbmin.xyz / max.xyz bounds of the sector (big-endian). Theory
+…varuint32_tcountsOne or more index-list counts (renderables, sub-sectors, neighbours).
+…varuint32_t[]indicesPer-list index arrays (asset/renderable ids, neighbour sector ids).
+…varutf16lenameOptional sector/source name fragment.
The UTF-16LE name runs sometimes read "scrambled" in an 8-bit hex dump (e.g. q001 \0 \0 \ p r … with high bytes interleaved). That is just the little-endian 16-bit text inside an otherwise big-endian container — decode the string columns as UTF-16LE and they resolve cleanly.

Related streaming data

Sectors are the consumer side of the streaming pipeline; the asset side lives in the .stream / .streamtex files and is described under Resources & Streaming. A separate top-level sectordata.dat / sector_scenarios_overlay.dat pair (EBOOT GenerateSectorScenarios) is used by the Megabowles powerplay-test harness to drive scenarios "per sector" — that is test tooling, not the per-level .sectorInfo documented here.
