Triggers (.triggers)

The list of powerplay trigger volumes for one subtrack — the "buttons" a car drives into to fire a powerplay. Like its .powerplays sibling this is a plain UTF-8 XML file, not a binary blob. Each <Trigger> is a positioned spherical volume plus activation rules and a reference to the powerplay (and specific timeline) it should set off. The EBOOT calls these TriggerButtons.

At a glance

Each file is a flat list of <Trigger> elements under a versioned root. Several Nemesis subtracks have no powerplay triggers at all and ship the minimal 32-byte self-closing root <Root FileVersion="1.0000000"/> — a clean tell that the format is text and that an empty list is legal. The largest seen is airport_test_03 at ~23 KB. The EBOOT confirms both the load path (environment(noassert)|Subtracks/%s/%s.triggers, Resident(noassert)|UI/Frontend/Powerplays/frontend.triggers) and the runtime label Powerplays/TriggerButtons/%s.

A trigger only points at a powerplay; it carries none of the powerplay's content. The matching definition (placement, timelines, props) lives in .powerplays, looked up by the shared id/idCRC.

<posAndScale>

A single 4-component vector defining the activation volume: X/Y/Z is the world-space centre and W is interpreted as the (uniform) activation radius. Across all 394 triggers W is always a positive scalar ranging 1.0–193.8 with a median of ~35, consistent with a car-scale radius; the spherical reading is inferred from the single W scalar and the name posAndScale, not confirmed in code. Y is not uniformly ground level — it spans roughly −27 to +245 (only ~37% of triggers sit within ±1 of zero), reflecting elevated and multi-level track sections.

AttributeTypeDescription
X, Y, ZfloatWorld-space centre of the trigger volume. Y varies widely (−27…+245), so volumes are placed at the track height, not clamped to the ground.
WfloatActivation radius (inferred). Always positive; observed 1.0–193.8, median ~35.
The <posAndScale> name and lone W scalar point to a uniform sphere, but whether W is strictly a radius (vs. some other uniform scale) is inferred from its value distribution rather than read out of the runtime. Partial on the exact geometry.

<Powerplay> / <Timeline> reference

The payload: which powerplay this trigger fires, and which timeline within it. Both are reference-only (id + idCRC, no name here) — they resolve against the .powerplays file's matching id. A single trigger references exactly one <Powerplay>, which in turn names one <Timeline> to run.

ElementAttributesDescription
<Powerplay>id, idCRCThe powerplay definition to fire. Matches a <Powerplay id> in .powerplays.
<Timeline>id, idCRCThe specific timeline of that powerplay to play.

triggerType enum

Only two values occur across all 394 triggers in the shipped data set (388 + 6). The EBOOT, however, lists the enum as three consecutive names — Standard, FireBestChild, VictimSelector — so a third type exists in code but is never instanced on disk.

NameCountNotes
Standard388Ordinary single-shot button: enter the sphere → fire the referenced powerplay.
FireBestChild6Selects and fires the most appropriate child/variant powerplay rather than a fixed one — used where one button can drive different outcomes (e.g. route-dependent picks).
VictimSelector0Defined in the EBOOT enum but never used in shipped data. Name suggests it targets a specific victim/opponent rather than firing on entry; behaviour unverified.
The two names Standard and FireBestChild (and the unused VictimSelector) are confirmed verbatim in eboot_strings.txt, but FireBestChild's child-selection logic lives in cPowerplayController, not in this file — the behavioural description here is inferred from the name and its scarcity. Theory on the exact selection rule.

Empty / minimal file

Six subtracks have no powerplay triggers (nem_downtown/A, nem_graveyard/A, nem_graveyard/B, nem_storm/A, nem_training/B, nem_warehouse/A) and ship the 32-byte stub:

000000  3c 52 6f 6f 74 20 46 69 6c 65 56 65 72 73 69 6f   <Root FileVersio
000010  6e 3d 22 31 2e 30 30 30 30 30 30 30 22 2f 3e 0a   n="1.0000000"/>·

The self-closing /> and trailing newline make exactly 32 bytes — confirming the file is text and that an empty trigger list is a valid, deliberate state rather than a missing file.

How we know

Files are readable XML from byte 0. Attribute and element names are taken verbatim from nem_training/Subtracks/A/nem_training.triggers and docks/Subtracks/C/docks.triggers, and cross-checked against frontend.triggers. The triggerType counts (Standard=388, FireBestChild=6) are an exhaustive tally across every .triggers file. The EBOOT (eboot_strings.txt) supplies the load paths and the Powerplays/TriggerButtons/%s label, plus the owning classes cPowerplayManager/cPowerplayController and the cPowerplayTriggeredEvent event type.

Open questions

  • FireBestChild's exact selection algorithm is unknown. The triggerType enum is now known to include a third value, VictimSelector (from the EBOOT), but it is never instanced in shipped data so its behaviour and data shape are unverified.
  • The owner/ownership semantics (dontSetOwner, enableForOwningPlayer) tie into the networked cPowerplayTriggeredEvent replication — see Networking & Online for the duplicated-object angle.
  • Whether posAndScale volumes are always spheres (vs. W meaning something else for non-Standard types) is unverified, but every observed value is consistent with a radius.
