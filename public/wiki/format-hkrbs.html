Rigid Bodies (.hkRBs / .hkRBXs / .hkHBXs)

Three closely-related Havok 5.5 packfiles carry the environment's rigid bodies, and the suffix tells you which kind. .hkRBs is the destructible furniture / smashable-prop set; .hkRBXs ("RB eXtra") holds the invisible respot boxes; .hkHBXs ("Handling-Box eXtra") holds the handling-modifier boxes. All three are ordinary packfiles whose root container wraps hkpRigidBody objects with Black Rock's custom export user-data, and all three ship with XML twins (.hkRBsXml, .hkRBXsXml, .hkHBXsXml).

The three suffixes at a glance

The EBOOT loads all three from explicit path templates, which also reveal where each lives:

environment|Subtracks/%s/%s.hkRBs      ← per-subtrack, beside .hkPPs
environment|Physics/%s.hkRBXs          ← per-level
environment|Physics/%s.hkHBXs          ← per-level
SuffixMeaningHoldsRoot variants
.hkRBsRigid BodiesDestructible furniture, smashable fences, signs, crates…4
.hkRBXsRigid Body eXtra (respot boxes)Box volumes that detect crashes & trigger respotting3
.hkHBXsHandling-Box eXtraBox volumes that limit speed / override surface material3
"RBXs / HBXs" are not hinge bodies. Despite the cluster-hint guess of "rigid / extra / hinge", the actual data shows the X stands for the auxiliary eXport user-data box volumes (respot & handling). Hinge/ragdoll constraints (hkpHingeConstraintData, hkpRagdollConstraintData) do exist in the engine, but they live inside the relevant systems — there is no dedicated hinge-body file extension in the shipped data.

Shared container header

All three are the same Havok packfile as everything else in the Havok cluster: a constant 57 E0 E0 57 header, the Havok-5.5.0-r1 version string, and the __classnames__ / __types__ / __data__ sections. Bytes 0x00–0x25 are byte-identical to .phys and .hkPPs; the only varying header field is the Havok contentsClassNameSectionOffset at 0x24 (a 16-bit-sized value, so only the word at 0x26–0x27 changes). It is a class-name section offset, not a data/file size — the byte at 0x22 is always 00.

.hkRBs — destructible furniture & props

This is the file most people mean by "the rigid bodies". The hkpPhysicsData holds one hkpPhysicsSystem per object instance — Downtown A has 20 systems with names like wooden_crate_large. Four root variants are present:

classNameRole
hkpPhysicsDataThe systems (one named hkpPhysicsSystem per prop) and their hkpRigidBody+shape.
HavokFurnitureObjectExportUserDataMarks the static "furniture" body the props rest on (furnitureBody ref).
SingleBody_UDContainerArray of SingleBody_UD — one per single-piece destructible.
CompoundBody_UDContainerArray of CompoundBody_UD — one per multi-piece destructible.

SingleBody_UD record

Each single-body record names a destructible instance and lists its physics body IDs. Confirmed from the XML twin:

FieldTypeDescription
typeenumAlways SBUD_EXPORT_TYPE.
familyuint32_tMaterial/behaviour family (0,1,2 seen) — drives breakoff & debris.
breakabilityuint32_tHow easily it shatters (0–3 seen; 0 = solid, 3 = fragile).
instanceNamestringe.g. wooden_crate_large, smashable_Fence_double_04_c, sign_small_01.
shapeKeyint32_tHavok shape key for the body within the shared shape.
bodyIds[]uint32_t[]Hashed IDs of the constituent broken pieces.
bodyParentIds[]uint32_t[]Parent ID per piece (0xFFFFFFFF = root / no parent).
 
<hkobject name="#0562" class="SingleBody_UD">
  <hkparam name="type">SBUD_EXPORT_TYPE</hkparam>
  <hkparam name="family">2</hkparam>
  <hkparam name="breakability">3</hkparam>
  <hkparam name="instanceName">wooden_crate_large</hkparam>
  <hkparam name="bodyIds" numelements="7"> 926042105 3729908428 … 6921164 </hkparam>
  <hkparam name="bodyParentIds" numelements="7"> 4294967295 … 4294967295 </hkparam>
</hkobject>

CompoundBody_UD record

Compound bodies are props made of several jointed pieces (e.g. smashable_Fence_double_04_b, smashable_Fence_02). They add a numBodiesInSystem count and store breakabilities[] / families[] per piece:

FieldTypeDescription
typeenumAlways CUD_EXPORT_TYPE.
instanceNamestringInstance name of the compound prop.
numBodiesInSystemuint32_tPiece count in this compound.
breakabilities[]uint32_t[]Per-piece breakability.
families[]uint32_t[]Per-piece family.
bodyIds[] / bodyParentIds[]uint32_t[]Piece IDs and parent links.
shapeKeyint32_tShape key into the shared compound shape.

These two record types are exactly the SingleBody_UD / CompoundBody_UD classes the EBOOT references, and the engine's debug menu has matching all.SingleBodies / all.CompoundBodies resource managers plus "make furniture shape debris on load" tuning. This is Split/Second's smashable-scenery system, distinct from the powerplay-driven .mcl prop collision.

.hkRBXs — respot boxes

Tiny by comparison (Downtown is 33 KB). The root container exposes 3 variants: hkpPhysicsData (a single system named respotBoxes_system), HavokRespotBoxExportUserDataContainer and HavokRespotBoxSystemExportUserData. Each rigid body is an hkpBvShape wrapping an hkpBoxShape (an oriented box, e.g. halfExtents (177.2 2.0 48.1 …)) placed along the track. The user-data is minimal:

FieldTypeDescription
typeenumRESPOT_BOX_TYPE.
rigidBodyrefThe box body this entry annotates.
forceCrashboolIf true, entering the box counts as a crash and respots the car.

These drive the respot / teleport logic — the EBOOT class cInstantRespotBox and the "RespotBoxes/Debug Render" debug toggle confirm the use.

.hkHBXs — handling-modifier boxes

Same shape (box volumes, 3 root variants), but the user-data describes how a car's handling changes inside the volume. Only Powerplant ships one (33 KB). The system is named handlingModifierBoxes_system and each HandlingModifierBoxExportUserData carries:

FieldTypeDescription
typeenumHANDLING_MOD_BOX_TYPE.
rigidBodyrefThe box body this entry annotates.
limitSpeedboolWhether the box caps the car's speed.
speedLimitfloat32_tThe cap (e.g. 95.0, 85.0).
overrideMaterialuint32_tSurface-material index forced while inside (e.g. 4).
 
<hkobject class="HandlingModifierBoxExportUserData">
  <hkparam name="type">HANDLING_MOD_BOX_TYPE</hkparam>
  <hkparam name="rigidBody">#0065</hkparam>
  <hkparam name="limitSpeed">true</hkparam>
  <hkparam name="speedLimit">95.000000</hkparam>
  <hkparam name="overrideMaterial">4</hkparam>
</hkobject>

The EBOOT's "Handling Modifier Boxes" / "physics handlingboxes" debug groups and the HandlingModifierBoxExportUserDataContainer class match exactly. These are the off-track / slow-zone volumes that gently nudge handling without an explicit collision.

Confidence

Confirmed — every record field above is read directly from the matching XML twin and cross-checked against EBOOT class names and load paths. The suffix meanings are derived from the data, not assumed. Exact in-packfile byte offsets of each user-data struct are Partial (they follow the __types__ reflection but were confirmed via XML, not by hand-parsing the binary).
