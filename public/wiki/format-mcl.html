Mesh Collision (.mcl)

.mcl files live under Powerplays/Animations/<Level>/… and were catalogued in the physics cluster on the assumption they were powerplay/prop collision meshes. Hex inspection tells a different story: .mcl is a custom Black Rock container (magic SDRI, chunk tags INSS/INSE) whose payload is a list of animated material-parameter overrides — colour fades, UV scrolls, fresnel/specular tweaks and texture swaps applied to props during powerplay sequences. It is not a Havok file and contains no geometry.

Naming caveat. The sidebar lists this under "Formats · Physics (Havok)" as "Mesh Collision (.mcl)", but every sample examined is a material container, not a collision mesh. No 57 E0 E0 57 Havok magic, no hkp*Shape class names, no vertex/index buffers — instead, shader-parameter strings and float/vector values. Real powerplay/prop collision is handled by the Havok .hkColl world meshes and by the powerplay physics packs (_pps, see .hkPPs). The expansion of the abbreviation is best read as Material Clip / Material Combo List, not "mesh collision".

Evidence: it is a material file

The smallest sample, AA_HelicopterShockwave.mcl (426 bytes), fits in one screen and is unambiguous — it is a flat list of named material parameters with values:

Every parameter name follows the pattern <effect>_<index>_<property>: coloralpha_3_fadeout, uvscroll_4_scroll, Fresnel_Alpha_5_fresnel_power, alphablend_6_texture, Diffuse_IPR_2_specular_amount, Diffuse_IPR_2_tint, Diffuse_IPR_2_diffuse, Diffuse_IPR_2_ipr. These are shader/material slot names — the numeric token is the material-layer index and the trailing word is the tunable. None of this is collision geometry. The hex string "0x622715a2" stored as ASCII is the target material/shader hash the clip retints; the preceding 10 c0 2f 34 is a binary 32-bit id for the instance.

This is corroborated by the EBOOT (eboot_strings.txt), which contains the material-combo factory classes cMaterialComboManagers and cMaterialComboFactory, and by the sibling load path powerplays(noassert)|Powerplays/Animations/%s/%s_pps — powerplay animations carry their own physics (_pps) separately from these material clips.

Container layout

The outer framing is a tiny tagged-chunk wrapper. All integers are big-endian. Confirmed constant across every sample (small 426 B, mid 2–3 KB, several Pre/Post variants):

OffsetLengthTypeNameDescription
0x000x4char[4]magic"SDRI" (53 44 52 49). Constant. Shared Black Rock container magic — the .shaderinst files also start with SDRI, which further ties .mcl to the material/shader pipeline rather than physics.
0x040x4uint32_theaderSize0x08 in every sample — bytes from 0x00 to the first body chunk.
0x080x4char[4]chunkTag"INSS" — "instance set start". Opens the material-instance list.
0x0C0x4uint32_tinstanceCountNumber of material instances in the body. Ranges 0–41 across the 1,636 files (most common is 2; 16 files have 0). Big-endian.
0x100x4uint32_tsetId / hashPer-file 32-bit id (e.g. 0x10c02f34, 0xd50c2fdf, 0xe0e77c01). Differs every file.
…varinstance[]instancesinstanceCount material-instance records (below).
EOF−40x4char[4]endTag"INSE" — "instance set end". Confirmed as the last 4 bytes of every file.

Material instance record

Value type tags (observed)

Theory — derived from value layouts in AA_tram_01.mcl, not from a symbol table. Only two top-level type tags were actually observed:

NameValueNotes
FLOAT0x02One or more inline float32 values. Used for both scalars (e.g. fresnel_power = bd 4c cc cd = −0.05, one float) and vectors (e.g. tint = 3f80 0000 3f80 0000 3f80 0000 = 1,1,1, three floats). The component count is not the immediately following word — that reads 1 for both — so it is implied by the parameter, not encoded here.
TEXTURE / HASH-REF0x03A material/texture reference: a 32-bit ASCII-free hash word (e.g. 27 c7 fe a5), then a 0x09 marker, then a short list of small int pairs. Used by diffuse, ipr and cubemap.
Correction. An earlier draft listed 0x03 as "VECTOR3 (three floats)" and 0x09 as a top-level "texture" type. Hex inspection of AA_tram_01.mcl shows the opposite: the white RGB tint is stored under type 0x02 (three floats), while type 0x03 is the texture/hash-reference form, and 0x09 is a marker inside a type-0x03 record, never a top-level type.
Worked example (from AA_tram_01.mcl). Each record is uint32 type; uint32 slot; uint32 count; value(s) after the NUL-terminated name:
  • Diffuse_IPR_2_fresnel_power → type 0x02, slot 0x4C, count 1, then one float bd4c cccd = −0.05.
  • Diffuse_IPR_2_tint → type 0x02, slot 0x13, count 1, then 3f80 0000 3f80 0000 3f80 0000 = three floats all 1.0 (white RGB tint).
  • Diffuse_IPR_2_diffuse → type 0x03, slot 2, then the hash 27 c7 fe a5 and a 0x09 index block — a texture reference, not a float.
These are exactly the kinds of values a material would expose, never collision data.

How it is used

The directory structure tells the rest of the story. .mcl files cluster under powerplay-animation groups (PG05_MultiBus, PG07_Cars, PA01_collapsing_building) and split into Pre/, Post/ and base sets — i.e. material states before, during and after a powerplay detonates a prop (e.g. a bus glowing, scorching, then fading out). Each clip retints/scrolls/fades the props' materials in sync with the Catnip timeline that drives the powerplay. See the Powerplay System for the sequence machinery these feed.

Confidence & open questions

Partial. The outer SDRI/INSS/INSE framing, the material-not-collision nature, and the readable parameter strings are Confirmed from multiple samples and EBOOT corroboration. The per-parameter type/slot/valueCount field split is reconstructed from value shapes and is the Theory part.

  • Exact meaning of the per-instance flags word and the per-param count? word (reads 1 even for 3-float values).
  • Full value-type enumeration (only top-level types 0x02 and 0x03 were observed; 0x09 is a sub-marker inside 0x03 records).
  • Relationship between the ASCII material hash here and the shader ids in .shaders/.shaderinst / .fxc.
  • Confirmed: SDRI is a shared Black Rock container — .shaderinst files use it too. Still open: whether yet other extensions reuse it, and how the inner chunk dialect differs between .mcl and .shaderinst.
