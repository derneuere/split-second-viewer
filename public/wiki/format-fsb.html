FMOD Sound Banks (.fsb)

An .fsb file is an FMOD Sample Bank — the container that holds the actual compressed audio waveforms for Split/Second. The game ships banks in the FSB4 revision produced by FMOD Ex 4.28 (contrib/FmodEx/42800), wrapped by the in-house LLAudio2 layer. Each .fsb is paired with a .fev event project that references its samples by index.

Where banks live

Banks are loose on disk under Deferred/Audio/, grouped by purpose. Most folders contain a non-streamed bank, an optional _Streamed bank for long sounds, and a DEBUG_ variant used by the QA build:

  • Vehicles/CPU/ — per-car engine/exhaust samples, e.g. Hypercar_03.fsb, Muscle_08_Corvette_C6_v3.fsb. These pair with the Tonic granular engine model.
  • Music/Music.fsb — the largest bank in the game (~158 MB), holding the dynamic race-music layers.
  • Speech/Speech.fsb — co-driver / announcer voice clips (the eboot references localised Speech/<Lang>/Speech.fsb variants).
  • Locales/<level>/ — per-track ambience (airport, docks, downtown, graveyard, powerplant, nem).
  • Collision, Scrapes, Surface, Swooshes, Particles, GenericPowerPlays, Menu, Global — gameplay SFX categories.
A bank can hold one sample (most vehicle banks) or many (Speech.fsb has 13, Music.fsb has 133). Examining a one-sample bank next to a 133-sample bank is how the constant container header was separated from the repeating per-sample record below.

Container header (FSB4)

The first 0x30 bytes are a fixed FSB4 header (a 0x20-byte field block followed by a 16-byte bank GUID). All multi-byte header fields are little-endian even though the PS3 and the rest of the engine are big-endian — this is an FMOD design choice and was verified by reading the sample count and header-size fields (see "How we know" below). The first sample record then begins at 0x30, and in the single-sample Hypercar_03.fsb the MP3 payload begins at 0x30 + 0x50 = 0x80, exactly as the hex excerpt shows.

OffsetLengthTypeNameDescription
0x000x4char[4]magicAlways "FSB4" (46 53 42 34).
0x040x4uint32_t (LE)numSamplesNumber of sample-header records that follow. 1 for vehicle banks, 13 for Speech, 0x85=133 for Music.
0x080x4uint32_t (LE)sampleHeadersSizeTotal bytes of all sample headers. = 0x50 for one mono sample; 0x410=1040 for Speech (13 × 0x50). Confirms an 80-byte base record.
0x0C0x4uint32_t (LE)sampleDataSizeTotal bytes of packed waveform data after the header block.
0x100x4uint32_t (LE)versionFSB version. Constant 0x00040000 = FSB 4.0 across every bank.
0x140x4uint32_t (LE)modeBank-wide mode flags. Constant 0x00000020 in all Split/Second banks (FMOD FSB_SOURCE_BASICHEADERS-style flag set).
0x180x8uint8_t[8]paddingZero in every sample seen.
0x200x10uint8_t[16]hash / GUID16-byte bank identity hash (differs per bank, e.g. 90 47 95 dd …). Used to match the bank to its FEV.

Per-sample header

Immediately after the container header comes numSamples sample records. Each begins with its own 16-bit size, so the array is walkable even when records vary in length (Music adds an extra SYNC sub-chunk — see below). The base record is 0x50 bytes:

OffsetLengthTypeNameDescription
+0x000x2uint16_t (LE)sizeLength of this record. 0x50 for a plain sample; 0x5C for Music samples (extra chunk).
+0x020x1Echar[30]nameNull-terminated sample name, e.g. Hypercar_03_Skyline_v3.wav, when_you_see.wav, intensity_3_b.wav.
+0x200x4uint32_t (LE)lengthSamplesDecoded length in PCM samples.
+0x240x4uint32_t (LE)lengthCompressedBytesSize of this sample's compressed data block. Matches sampleDataSize for single-sample banks.
+0x280x4uint32_t (LE)loopStartLoop start in samples (0 when unused).
+0x2C0x4uint32_t (LE)loopEndLoop end in samples.
+0x300x4uint32_t (LE)modeFMOD sample mode flags (codec, loop, channel layout).
+0x340x4uint32_t (LE)defaultFreqSample rate in Hz. 0xBB80=48000 (vehicles), 0xAC44=44100 (music/streamed).
+0x380x2uint16_t (LE)defVolDefault volume (0x00FF = full).
+0x3A0x2int16_t (LE)defPanDefault pan (0x0080 = centre).
+0x3C0x2uint16_t (LE)defPriDefault priority (0x0080).
+0x3E0x2uint16_t (LE)numChannelsChannel count: 0x0001 mono (speech, vehicle), 0x0002 stereo (music, streamed ambience).
+0x400x4float32_t (LE)minDistance3D rolloff min distance. Constant 1.0 (00 00 80 3f).
+0x440x4float32_t (LE)maxDistance3D rolloff max distance. Constant 10000.0 (00 40 1c 46).
+0x480x8uint8_t[8]varFreq/Vol/Pan + size echoPer-sample variation fields; the trailing 0x50 echoes the record size.
The two distance constants 1.0 and 10000.0 and the centred pan/priority bytes appear identically in every record of every bank, which is exactly what you expect for a default 3D rolloff — strong evidence the field interpretation above is correct.

The SYNC sub-chunk (Music only)

In Music.fsb the per-sample size is 0x5C, not 0x50, because each music sample carries a trailing SYNC chunk used to place beat/section sync points for the dynamic music director (the eboot drives Music/<track>/RaceSection0<n> events). The chunk literally starts with the ASCII tag SYNC.

The ff fb at 0x80 is an MPEG-1 Layer III frame header, confirming the codec. The lengthCompressedBytes field (0x138c0) equals the container's sampleDataSize — exactly what a single-sample bank should report — which is how the per-sample length field was pinned down.

Multi-sample banks

Speech.fsb demonstrates the walkable array: after the 0x410-byte header block (13 × 0x50), the records sit back-to-back. Names read out plainly as when_you_see.wav, if_the_target.wav, ok_kid.wav, come_on.wav — co-driver call-outs. Music.fsb's first samples are intensity_3_b.wav, intensity_2_b.wav (dynamic music intensity layers).

FMOD reports an error string in the eboot — "FSB mismatches the FEV it was compiled with, the stream/sample mode it was meant to be created with was different, or the FEV was built for a different platform" — so a bank and its .fev are a tightly-coupled pair; the 16-byte GUID and per-platform build keep them in sync.

How we know

  • Magic / version constant: FSB4 and version 0x00040000 are byte-identical across Menu, Music, Speech, Hypercar_03 and the streamed banks.
  • Header-size arithmetic: a 1-sample bank reports sampleHeadersSize=0x50; the 13-sample Speech bank reports 0x410=13×0x50 — proving both the 80-byte record size and that the count field is little-endian.
  • Names & codec: ASCII .wav names and 0xFFFB MP3 sync bytes are directly readable in the hexdump.
  • Endianness: reading the count/size fields as big-endian gives absurd values (e.g. 0x01000000); little-endian gives the correct small counts that match the file layout.
  • Middleware: the eboot strings EventSystem::unloadFSB, FMOD_OPENMEMORY_POINT, and PluginMiddlewareFMOD.inl confirm FMOD Ex.
