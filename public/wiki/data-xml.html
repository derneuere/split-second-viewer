XML Conventions

Alongside the .params tuning files, Split/Second uses XML for the data that is naturally hierarchical or list-shaped: vehicle and season databases, the camera graph, per-locale audio triggers, collision/breakable timelines, and the Havok *Xml tagfile twins. Two distinct dialects exist. The bulk are Black Rock's in-house XML — a tiny, attribute-heavy format always rooted in <Root FileVersion="…">. The rest are Havok 5.5 tagfiles, which are the middleware's own XML serialization and document the binary class layout. This page covers where each is used and the element patterns you will see.

In-house XML: the <Root> pattern

Every in-house file starts with a single <Root> element carrying a FileVersion attribute. The version is a per-schema number, not a global one — databases use "1.0" while the newer camera/collision schemas use a float like "0.1000000" or "0.0000000". Data is carried almost entirely on attributes; element nesting expresses ownership. Three structural styles recur:

1. Flat record list (databases)

Vehicles/VehicleDatabase.xml is one <Car> element per vehicle, all attributes, no children. This drives Vehicles and links each car to its .phys handling files, audio bank and cameras: 

<Root FileVersion="1.0">
 <Car ID="Musclecar_01" HumanPhysicsFile="Musclecar_01" AIPhysicsFile="AIMusclecar_01"
      ChaseCamFile="MuscleChase" BumperCamFile="DefaultBumper" BonnetCamFile="DefaultBonnet"
      Lives="3" ActionScriptimage="11" Locked="false" DCBModifier="0" PlayerSkillLevel="0.2"
      AudioFile="Truck_03" Speed="2" Drift="4" Strength="4"/>
 ...
 <Car ID="NemesisTruck01" HumanPhysicsFile="NemesisTruck01" AIPhysicsFile="NemesisTruck01"
      ChaseCamFile="TruckChase" Lives="3" Player="false" .../>
 <!-- This is how you comment out an entry. -->
</Root>

Note the real conventions on show: the Speed/Drift/Strength 1–10 star ratings, the PlayerSkillLevel 0–1 difficulty gate, the AIPhysicsFile pointing at a separate AI handling variant (e.g. AISupercar_04), and the standing <!-- comment out --> idiom designers used to disable rows.

2. Nested containers (season / scoring trees)

Season/Season.xml nests <Event> and <UnlockableCar> inside each <Episode>, building the single-player progression consumed by Scoring & Progression: 

<Episode Number="1" ScoreToUnlockFinale="16" VehicleSkillLimit="0.2">
  <Event GameMode="QuickRace" Track="Downtown"        SubTrack="A" Variation="2" DCBModifier="0.5"/>
  <Event GameMode="TimeTrial" Track="Downtown"        SubTrack="C" Variation="1" DCBModifier="0.5"/>
  <Event GameMode="QuickRace" Track="docks"           SubTrack="B" Variation="2" DCBModifier="0.5"/>
  <UnlockableCar Name = "musclecar_03" />
</Episode>

The file even ships with a long comment documenting its own schema (the three ways to declare allowed cars: <Car Name="…"/>, <Car Type="…"/>, or omit for "all unlocked"). The GameMode, Track and SubTrack values are the same identifiers used across Game Modes and Tracks.

3. Manifest that points at .params (cameras)

The camera system is the clearest example of XML and .params working together. A locale ships a trio — Cameras.xml, CameraGroups.xml, CameraSplines.xml (~65 of each in the build). Cameras.xml is a manifest: each <Camera> declares a type and a Data="…params" leaf file holding its tunable values, and nests <Behavior> children that do the same: 

<Root FileVersion="0.1000000" BatchedLoad="true">
   <Camera Type="FixedToSubjectCamera" Name="1229362827.9" Data="1229362827.9.params" FriendlyName="">
      <Behavior Type="TrackHeightBehavior" Name="1238751163.0" Data="1238751163.0.params"/>
      <Behavior Type="LookAtBehavior"      Name="1232986167.2" Data="1232986167.2.params"/>
   </Camera>
</Root>

CameraGroups.xml then references cameras by Name to build named selection sets (e.g. <Group Name="Gyro Cams (Subject Velocity) Ahead"> → <Camera Name="1229362827.9"/>), and CameraSplines.xml stores the actual rail geometry as nested <Spline> → <Control_Point> → <Transform> rows.

Those numeric Name/ID values (1229362827.9, 2639553593) are tool-generated GUIDs — a timestamp-plus-counter or hash. They are stable handles for cross-referencing between the three camera files, not anything the player ever sees.

Transform rows

Wherever XML carries a 3D placement it uses a <Transform> (or bare transform="…" attribute) of four rows. Row0..Row2 are the 3×3 orientation/scale basis and Row3 is the translation — a standard 4×3 affine matrix: 

<Transform>
   <Row0 X="1.0000000" Y="0.0000000" Z="0.0000000"/>
   <Row1 X="0.0000000" Y="0.9865719" Z="0.1633275"/>
   <Row2 X="0.0000000" Y="-0.1633275" Z="0.9865719"/>
   <Row3 X="-0.0075565" Y="1.3200014" Z="-5.6494946"/>  <!-- translation -->
</Transform>

Audio & Tonic XML

Per-locale audio data lives under Audio/Locales/<level>/ as the quartet *_Audio.xml, *_Audio_Ambient.xml, *_Audio_Reverb.xml, *_Audio_Swooshes.xml (24 files total). The reverb files are lists of <AudioTrigger> boxes, each a placement plus a dense set of mix attributes that drive the LLAudio2 reverb/dynamic-mixer system: 

<AudioTrigger id="712004398" label="Garage 1" type="Box"
   transform="43.318,0.000,4.954, 0.000,25.231,0.000, -12.146,0.000,130.751, -757.406,-5.208,107.989"
   reverb_type="HALL" reverb_wet_mix="0.700000"
   dynamic_mixer_preset="LESS_AMBIENCE" dynamic_mixer_gain="1.0000000"
   falloff_x="0.100000" falloff_y="0.000000" falloff_z="0.290000" falloff_response="Linear"
   activate_at_startup="true" sound_name=""/>

Here reverb_type is an enum drawn from a fixed set seen across the docks file — HALL, OVERHEAD, LARGE_TANK, SMALL_TANK — and dynamic_mixer_preset selects a named mixer snapshot (LESS_AMBIENCE). The transform packs the 4×3 matrix into one comma-separated attribute rather than nested rows. The separate Audio/Tonic/ tree (55 files: _CarSpecific, _CarTypeSpecific, _StateEngine) configures the per-car engine-sound state machine — see Tonic.

Collision timelines (*_collision_tl.xml)

Each level ships a <level>_collision_tl.xml (10 in the build, one per locale incl. the nem_* variants), loaded via the eboot path %s_collision_tl.xml. It maps breakable props to their Catnip collision timelines — <object> → <timeline> → <transform> — with flags such as singleUse, hideComponentOnBreakOff and collisionMagnitude that decide how the object reacts when hit: 

<object name="propane_cylinder_custom">
   <timeline name="docks_propane_cylinder_T" activateOnInitialCollision="false"
             hideComponentOnBreakOff="true" singleUse="true" collisionMagnitude="50"
             useContactTransform="false">
      <transform> <Row0 .../> <Row1 .../> <Row2 .../> <Row3 .../> </transform>
   </timeline>
</object>
<object name="crate_custom"/>            <!-- object with no timeline -->

Havok tagfile twins

The remaining XML are Havok's own serializer output. The level physics files .hkColl, .hkPPs and .hkRBs each have an XML twin (.hkCollXml, .hkPPsXml, .hkRBsXml — 32 / 19 / 19 files). These are not in-house XML; they begin with a real prolog and the Havok packfile root, and version-stamp Havok-5.5.0-r1: 

<?xml version="1.0" encoding="ascii"?>
<hkpackfile classversion="5" contentsversion="Havok-5.5.0-r1">
	<!-- Skipped hkpShape at address 0000000012740100 -->
	...

They serialize the same object graph as the binary Havok packfile (classes such as hkpRigidBody, hkpShape, hkpStorageExtendedMeshShape), so the twin effectively documents the binary class layout. The schema is Havok's, not Black Rock's — treat its internals as Opaque middleware and consult the Havok Packfiles page.

Do not parse a .hkPPsXml with the in-house <Root> assumptions. The two dialects share the .xml-family extension but nothing else: different root, different prolog, different authoring tool.
