Powerplays (.powerplays)

The per-subtrack catalogue of powerplays — the scripted track-altering events (the "Level 1/2/3" destruction set-pieces) that define Split/Second's gameplay. Despite the binary-looking extension this is a plain UTF-8 XML file. Each entry binds a powerplay to a world placement matrix, one or more timelines, an animated-prop instance tree and a table of per-event parameter overrides. The .triggers file in the same folder references these powerplays by id; the Powerplay System page covers the runtime side.

At a glance

Every file opens with <Root FileVersion="1.0000000"> and contains a flat list of <Powerplay> elements (one large file, Downtown/A, has 111 of them). The loader path is confirmed in the EBOOT: powerplays(noassert)|UI/Frontend/Powerplays/frontend.powerplays, and the runtime owner is cPowerplayManager / cPowerplayController. File size scales with how much a subtrack does — the racing city tracks (Downtown, docks, Powerplant, Graveyard, airport_test_03) run ~230–353 KB with 100–130 powerplays each, while the smaller Nemesis subtracks run ~124–161 KB with ~80–86 powerplays. The lone outlier is the frontend menu file at ~11 KB / 7 powerplays.

Read this together with .triggers. A .powerplays entry defines a powerplay and its content; a .triggers entry places the activation volume that fires it, referencing the same id/idCRC.

Document structure

The nesting is regular. A simplified skeleton of one powerplay:

<Root FileVersion="1.0000000">
  <Powerplay name="…\Powerplays\PG07_Taxis_Long_P" NiceName="PG07_Taxis_Long_Freeway_1"
             Repeatable="false" AICanTrigger="true" RouteChanging="false" OpenShortcut="false"
             TotallyUndriveable="false" MasterMaterial="7" PowerLevel="1"
             id="1249478979.1976132398" idCRC="0xCF520610" contextual="0">
    <worldmtx> <Row0/><Row1/><Row2/><Row3/> </worldmtx>          <!-- placement matrix -->
    <EventParams> <Event …> <Param …/> </Event> … </EventParams>   <!-- per-event overrides -->
    <Timeline name="Powerplays\Timelines\PG07_Cars_1_30m_T.xml" id="…" idCRC="…">
       <transform> <Row0/>…<Row3/> </transform>
    </Timeline>                                                    <!-- one or more -->
    <Ambient> <NODE_FACTORY …> … </NODE_FACTORY> </Ambient>        <!-- animated-prop tree -->
  </Powerplay>
  …
</Root>

<worldmtx> and <transform>

Both use the same four-row form: Row0/Row1/Row2 are the 3×3 basis (rotation+scale) and Row3 is the translation, each with X/Y/Z attributes. worldmtx places the whole powerplay in world space; each Timeline's transform offsets that timeline relative to the powerplay origin (e.g. Row3 Z="6.8707972" nudges one car group forward).

<Timeline>

One <Powerplay> can reference several timelines (Downtown/A has 191 across the file). Each names a timeline asset and carries its own placement.

AttributeTypeExampleDescription
namepathPowerplays\Timelines\PG07_Cars_1_30m_T.xmlTimeline asset path (the _T suffix = timeline).
id / idCRCuid / hex321262717283.-987787820 / 0xA5256093Timeline instance id and its CRC.

<EventParams> → <Event> → <Param>

Per-instance overrides applied to events on the referenced timelines. An <Event>'s idString is a comma-joined chain of three Catnip ids — powerplay-id, timeline-id, event-id — which scopes the override to a specific event inside a specific timeline of this powerplay; id is the event's own CRC. Each holds one or more typed <Param>:

AttributeTypeExampleDescription
NamestringAnimationParameter slot name. Observed: Animation, Route, Trigger, Timeline, Clip, PowerLevel.
Typeint6Value's data type (see enum below).
Valuetyped0x2B15FDF5The value — a CRC/asset hash for type 6, a small integer for type 1, etc.

Across all 20 files the (Name, Type) pairs are tightly correlated, which lets us pin the type codes:

Type codes 5 and 6 are inferred from how their values are written (integer vs. hex CRC) and from the parameter names, not from the EBOOT — treat the exact numeric meanings as Partial. Names 1=int, 5=route, 6=CRC/asset are consistent across all samples but unverified against the type enum in code.

<Ambient> — the animated-prop tree

A recursive <NODE_FACTORY> hierarchy that places and animates the actual scenery (cranes, rotors, etc.). Each node has a <STATIC_POSITION><node_transform> (same 4-row matrix form) and may carry motion modifiers and/or children:

Element / attributeDescription
NODE_FACTORY TypeNode kind. 0 = leaf with an <INSTANCE> (a real prop), 1 = leaf variant, 2 = internal/grouping node. (Distribution in Downtown/A: 115×0, 4×1, 10×2.)
NumberOfInstances / NumberOfChildrenCounts of direct <INSTANCE> leaves and child <NODE_FACTORY> nodes — redundant with the actual element counts, used by the loader to pre-size arrays.
ROTATION_POSITIONER Speed, AxisConstant spin: Speed in radians/tick about Axis (observed 120/121/122 = X/Y/Z axis codes).
OSCILLATION_POSITIONER MaxArcAngle, FrequencyPendulum-style sway, amplitude (radians) and frequency.
INSTANCE Name, Path, IdThe leaf prop, e.g. Name="AA_skycrane_01" Path="Powerplays\Animations\AA\" Id="0xE131565F". Id is the instance CRC.

How we know

The file is literally readable XML (head shows ASCII tags from byte 0). Element and attribute names below are taken verbatim from Downtown/Subtracks/A/Downtown.powerplays and cross-checked against frontend.powerplays. Class names (cPowerplayManager, cPowerplayMaterialManager, cPowerplayController) and the load path appear in eboot_strings.txt. PowerLevel maps to the EBOOT's SetPowerLevel1/2/3 and the "Powerplays/Highlight/Level 1-3" debug colour menus.

Open questions

  • The full Param Type enum (only 1, 5, 6 observed) and the exact axis-code base (120 = ASCII 'x'? the values 120/121/122 are suggestive) are unconfirmed.
  • MasterMaterial indexes the cPowerplayMaterialManager set but the material table itself lives elsewhere (see Powerplay System).
  • The two halves of the dotted id (1249478979.1976132398) look like a creation timestamp plus a serial; this is the same id scheme used throughout Catnip but its bit layout is unverified.
