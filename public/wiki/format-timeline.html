Timelines (.timelineInfo)

.timelineInfo is a tiny, fixed-layout binary index that accompanies every timeline-particle data set. It is an array of (64-bit name-hash, slot-index) records that maps the events on a particle timeline onto entries in the sibling .emitterControllers file. It carries no geometry or tuning of its own — it is purely the lookup/sequencing table that the Catnip timeline system walks at runtime.

Where it lives

Every TimelineParticles folder holds one particle class — Standard, Flare, Light, Geometry or Screen — and each class folder contains a matched set of files that are always loaded together: 

Environments/Levels/Downtown/Subtracks/A/Particles/TimelineParticles/Standard/
    Standard.data                 113 132 B   particle/system definitions
    Standard.emitterControllers 1 006 768 B   the controllers this timeline drives
    Standard.emitterSurfaces       18 008 B   emitter surface geometry
    Standard.timelineInfo          23 240 B   ← this file: timeline → controller map
    Standard.fxc / .shaders / .shaderinst / .textures   render assets

The EBOOT loads them in sequence: Loading '%s' particle data file… → Loading '%s' particle emitter controller file… → reads %s.emitterControllers then %s.timelineInfo. The per-class log banners (Standard Timeline Particles, Flare Timeline Particles, Light Timeline Particles, Geometry Timeline Particles, Screen Timeline Particles) and the internal timeline names (standard_timeline, flare_timeline, light_timeline, geometry_timeline, screen_timeline) all appear in eboot_strings.txt immediately adjacent to the %s.timelineInfo path token, fixing the format's owner classes (cStandardGameParticleDataGroup, cFlareGameParticleDataGroup, cLightGameParticleDataGroup, cGeometryGameParticleDataGroup + cParticleClass).

File layout

The format is trivial and 100% accounted for: an 8-byte header followed by count records of 12 bytes each. Across all 88 .timelineInfo files in the build, file size == 8 + count × 12 holds exactly, version is always 1, and the per-record index is always the exact sequence 0…count−1 — which together prove the record size and confirm there is no footer or string pool.

Record (12 bytes × count)

OffsetLengthTypeNameDescription
+0x00x8uint64_tcontrollerHash64-bit hash of the emitter-controller / particle-data name. Repeats across records (the same controller is fired many times on a timeline). Distinct from the controller's textual id — see note below.

The link to .emitterControllers

The companion .emitterControllers file opens with its own uint32 version (=5) then a uint32 count — and that count equals the .timelineInfo count in the simple frontend cases:

ClasstimelineInfo countemitterControllers count (offset 0x4)
UI/Frontend Light300 00 00 03 = 3 ✓
UI/Frontend Standard1100 00 00 0b = 11 ✓

Each controller entry in .emitterControllers begins with an ASCII name such as 1263313715.-811256644 (a two-part 32-bit id, the same notation the highlight-tags use) followed by an artist label like LIT_OrangeFire. The 64-bit value in .timelineInfo is the hashed key the runtime uses to resolve a timeline event to one of these controllers; the index field then orders the events on the timeline.

The EBOOT assert Wrong index for Emitter Controller found sits in the same loader block and is the runtime check that a .timelineInfo index is in range for the controller array — direct confirmation that the second field is an index into .emitterControllers.
The 8-byte field is a hash, not the literal controller id: decoding the first Standard-frontend controller id (1263313715.-811256644 → 0x4B4CA333 0xCFA534BC) does not equal the first record's hash (0xD24012E2A4B1A2DF). The exact hash function (CRC-64 / FNV-style over the name string) is unverified; only its role as a stable name key is confirmed.

Notes & open questions

  • Some classes have far more timeline records than unique controllers (Downtown Standard = 1 936 records). This is expected: a timeline fires the same emitter many times, so the hash repeats while the index keeps climbing.
  • Whether index is a flat array slot or a packed timeline-frame ordinal is not distinguishable from the data alone; both interpretations are consistent with the strictly sequential values observed.
  • No endianness ambiguity: the count and indices only read sanely as big-endian.
