HUD & Frontend

Split/Second's on-screen UI is split between a data-driven in-race HUD — laid out per game mode and per camera view by GameModeHUDs/*.params and rendered through Scaleform GFx movies — and a frontend built from dozens of .gfx Flash assets driven by an XML state map. Both sit on the same GFx 2.x runtime and the same shared font library.

The .gfx magic is verifiable in the data: UI/HUD/Hud.gfx starts 47 46 58 08 ("GFX" + version byte 08), and the eboot's loader emits incompatible GFX file, version 2.x expected — confirming a Scaleform GFx 2.x pipeline (assets are exported from .swf; the eboot still references gfxfontlib.swf).

In-race HUD: GameModeHUDs

The HUD layout is decoupled from the HUD art. Twelve files in GameModeHUDs/ describe, per game mode, exactly which HUD elements are visible and where they sit on screen. The eboot loads them on demand (Resident(noassert)|GameModeHUDs/QuickRace.params) and the layouts target the RaceSetupHUD controller.

Per-camera-view layouts

Each file contains the same three sections — /RaceSetupHUD/Bonnet, /RaceSetupHUD/Bumper and /RaceSetupHUD/Chase — one for each camera view, because element positions differ between bonnet, bumper and chase cams. Within a section, every HUD element has an Enable … boolean and an X/Y screen position (0–1 normalized, some allowing negatives to anchor off the right edge), plus a per-view Power Bar Turned On flag.

/RaceSetupHUD/Bonnet:
	'Action Element X' = 0.25 (0.0, 1.0);
	'Action Element Y' = 0.6109369 (0.0, 1.0);
	'Enable Action Element' = true;
	'Enable Lap Element' = true;
	'Enable Position Element' = true;
	'Enable Score Element' = false;
	'Enable Timer Element' = false;
	'Lap Element X' = 0.9617187 (0.0, 1.0);
	'Position Element X' = -0.0859375 (-1.0, 1.0);
	'Power Bar Turned On' = true;

How the layout differs per mode

The enable-flags are what make each mode feel different. Comparing the Bonnet view across modes (all values read from the data):

So Time Trial swaps the action/lap furniture for a target-time readout and a 2D race clock, Demolition turns off the power bar and skill pop-ups in favour of score, and Nemesis adds Score + Opponent Score + Multiplier for the truck-overtake economy. The Demolition/TimeTrial files also add Distance Driven and Target Distance elements that the QuickRace file lacks.

2D vs 3D HUD components

The HUD is rendered in two layers. The flat overlay items (timer, position, messages) are 2D components; the speedo/power dials are 3D components floated in world space near the car. The 3D rig is tuned in UI/HUD/Params/3D_HUD.params, again split Bonnet/Bumper/Chase, with an Infobar per view that supports three placement modes:

Position modeValueBehaviour
Fixed1Anchored relative to the car (uses vehicle look pos/dir)
Aim for cam2Lags toward the camera with limiting angles & max position/angular lag
2D3Screen-space fixed position

The movies expose Flash callbacks the engine drives directly: InfoBarShow/InfoBarHide, BumperCamPowerBarShow/Hide, ChaseCamPowerBarShow/Hide, and the action ticker (ActionShow, SetActionText, SetActionScore, SetActionMultiplier, ActionRePulse). The big skill banners are string-table keys — JUMP PASS!, BEST JUMP!, BEST DRIFT! — driven by the scoring system.

Powerplay icons

The diegetic markers floating above cars (and the Nemesis indicator) are the Powerplay icons, art in GameLogic/PowerplayIcons.textures (~598 KB, a TEXS pack) rendered by UI/HUD/PowerPlayIcons.gfx. The eboot exposes a debug switch UI/HUD/PowerplayIcons/Render the icons above cars in 3D or 2D. plus per-icon offsets (Car icon x/y offset, Nemesis Car icon x/y offset, On Nemesis icon x/y offset), and the movie pools "3D PP Icons", "2D PP Icons" and "Chopper Target Icons" instances. The chopper-battle icon glow uses SetAsGlow on ChopperBattleIcons.gfx.

The standalone fonts/ directory is a non-GFx bitmap font: a .bed glyph/metrics file, a .ker kerning table and a .textures atlas. This is the renderer's own debug text path (the %.2fms, Verts %.2f mill FPS overlays), separate from the Scaleform fonts used for player-facing UI.

Frontend (menus)

The menu system is a large set of Scaleform movies under UI/Frontend/Flash/, organised by the naming convention FE_<screen>_Billboard / _Poster / _Widget / _Ticker — the billboard is the big 3D panel, the poster/widget the interactive parts. The screen graph itself is data — UI/Frontend/FrontendMapV2.xml — and the frontend renders against 3D models in UI/Frontend/Models/ (e.g. FE_Asset_Billboard.model) with its own colour cubes at UI/Frontend/ColorCubes/.

Common chrome lives in UI/CommonUI/ (Message.gfx, Prompt_Bar.gfx, FE_LoadingScreen.gfx, SafeFrame.gfx, plus the debug-only QA_Menu.gfx / QA_Info.gfx / ResourceStats.gfx), and the in-game overlays (pause, results, objectives) in UI/InGame/ (FE_PauseMenu_NEW.gfx, RaceResults.gfx, RaceEnd.gfx, EpisodeAwards.gfx, EventObjectives.gfx, DebugMenu.gfx). The boot sequence uses its own UI/Bootup/Bootup.gfx.

The presence of QA_Menu.gfx, DebugHud.gfx and DebugMenu.gfx is a fingerprint of this devkit/QA build (qa_menus=true); a retail disc would strip these. The Scaleform movie set is otherwise representative of the shipped UI.
