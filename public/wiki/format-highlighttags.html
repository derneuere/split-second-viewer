Highlight Tags (.highlighttags)

.highlighttags is a plaintext XML file — one per level — that lists the on-screen highlight markers the HUD draws over powerplay locations. Each <HighlightTag> binds a 3-D placement (a 4×3 transform) to a powerplay, so the game can show "Highlight Geometry" (the boxed/pulsing indicator) where a powerplay is about to trigger or where a replay/highlight should be framed.

Where it lives

Exactly one file sits in each level root, named after the level: 

Environments/Levels/Downtown/Downtown.highlighttags        12 108 B
Environments/Levels/docks/docks.highlighttags              20 793 B  (largest)
Environments/Levels/Graveyard/Graveyard.highlighttags       7 042 B
Environments/Levels/airport_test_03/airport_test_03.…      18 204 B
Environments/Levels/nem_storm/nem_storm.highlighttags          32 B  (empty)

The loader token environment(noassert)|%s.highlighttags appears in eboot_strings.txt immediately beside the literal tag/attribute names this file uses — HighlightTag, highlightTag, relative_transform, transformation, Powerplay — and beside the UI strings Highlight Geometry, UI/HUD/Highlight Geometry/Cube Fade Down, Highlight pulse intensity and Tag Map. That ties this file to the HUD's highlight-geometry overlay rather than to the physics or particle systems.

Format

Because it is text, the whole format is directly readable; no hex interpretation is needed. The file is a single <Root> with a FileVersion attribute, containing zero or more <HighlightTag> elements. An empty level still ships the one-line self-closing root (the 32-byte nem_storm file).

Annotated excerpt

A complete tag from Graveyard.highlighttags — an identity-rotation marker offset on the Y/Z plane, pointing at one powerplay: 

<HighlightTag id="1263809359.-611452811" idCRC="0x492CC192" icon="0" relative_transform="true">
   <transformation>
      <Row0 X="1.0000000" Y="0.0000000" Z="0.0000000"/>   basis = identity
      <Row1 X="0.0000000" Y="1.0000000" Z="0.0000000"/>
      <Row2 X="0.0000000" Y="0.0000000" Z="1.0000000"/>
      <Row3 X="0.0000000" Y="229.37067" Z="-178.84406"/> translation (relative)
   </transformation>
   <Powerplay id="1239805449.2" idCRC="0x13887213"/>     linked powerplay
</HighlightTag>

And a rotated marker, also from Graveyard.highlighttags (note the non-identity Row0/Row2 = a yaw about Y; Row0/Row2 are still unit-length here, so it is a pure rotation, not a scale): 

<HighlightTag id="1262886888.-508312769" idCRC="0xD0A88A65" icon="0" relative_transform="true">
   <transformation>
      <Row0 X="0.8370311" Y="0.0000000" Z="0.5471553"/>   yaw rotation in XZ
      <Row1 X="0.0000000" Y="1.0000000" Z="0.0000000"/>
      <Row2 X="-0.5471553" Y="0.0000000" Z="0.8370311"/> (acos 0.8370311 ⇒ ~33° about Y)
      <Row3 X="-10.038172" Y="15.508035" Z="-6.7290363"/>
   </transformation>
   <Powerplay id="1256919348.-669000736" idCRC="0x6211B6B"/>
</HighlightTag>

Observations across samples

  • The same HighlightTag id recurs for several entries (e.g. 1249383404.… in Graveyard) each pointing at a different <Powerplay> — so one tag template can mark multiple powerplay instances that share a geometry footprint. The transformation block is byte-identical when the tag id repeats, confirming the id keys a reusable marker shape.
  • icon="0" is invariant across all 141 tags in all ten files (the icon set is unused in this build). relative_transform, however, is not fixed: of the 141 tags, 109 are "true" and 32 are "false" — and all 32 "false" tags live in the airport_test_03 test level (8 true / 32 false there), while every shipped race level uses "true" exclusively. So it is a real per-tag flag, not a constant.
  • For relative_transform="true" tags the translation magnitudes are level-scale offsets (tens to hundreds of units) from the linked powerplay's transform; the "false" tags in airport_test_03 carry larger absolute world coordinates (e.g. Row3 ≈ 141 / 14 / 77), consistent with an absolute placement.
  • File size scales with tag count: docks (busiest level) is largest at ~20 KB; nemesis-mode boss levels such as nem_storm ship an empty root.
The two-part id notation (<int>.<int>) and the idCRC hex attribute are shared with the Catnip entity, powerplay and emitter-controller data — strong evidence these are global Catnip object ids, and that .highlighttags is a cross-reference layer authored on top of the powerplay graph rather than a standalone asset.
Unlike most level data this file is human-editable text, which makes it the easiest place to confirm powerplay ids/CRCs and to verify the CRC algorithm against the binary CRC tables.

Open questions

  • The icon field is always 0 here; the icon index space (and where the icon art lives in the Scaleform GFx HUD) is not yet mapped.
  • Whether the HUD draws these only during the powerplay "danger zone" window or also in replay/ highlight playback (the format name suggests replay tagging; the runtime strings point at live HUD "Highlight Geometry"). Both uses may share the data.
  • The exact CRC-32 polynomial behind idCRC — verifiable since both the source id string and its CRC are present in the same file.
