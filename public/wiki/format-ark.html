Archives (.ark)

The .ark format is Split/Second's packed asset archive. Each file is a flat header, a hash-sorted table of contents, and a region of packed asset blobs. The engine's ArkLoader opens one archive at a time and looks members up by a 32-bit name hash via binary search. In the loose devkit/QA build documented here the only archives on disk are the per-level sector archives (<Level>.Static.ark and <Level>.Stream.ark) — 20 files totalling roughly 2.2 GB — but the EBOOT also references the retail global archives (Resident.ark, Frontend.ark, InGame1/2/3.ark) and per-rider animation archives.

Where they live

The EBOOT builds archive paths from a handful of templates (see eboot_strings.txt): 

Environments/Levels/%s.ark
Characters/RiderAnimations/animationNetwork/rider_%02d.ark
Resident.ark
Frontend.ark
InGame1.ark   InGame2.ark   InGame3.ark

On disk every level folder under Environments/Levels/<Level>/Sectors/ carries a pair of archives:

  • <Level>.Static.ark — resident geometry/data for a level that stays loaded (2.9 MB – 91 MB across the 10 levels).
  • <Level>.Stream.ark — the large streaming pool fetched per sector as the player drives (38 MB – 564 MB). This is what the streaming system pulls sectors from at runtime; see also Sectors (.sectorInfo).

The two members of a pair are addressed through paired logical archive names the EBOOT builds from CityStatic / CityStream (and the CityStaticOverlay / CityStreamOverlay variants), using the same <Archive>|<member> syntax as the global archives — e.g. CityStatic|%s_rig%d and CityStream|%s_rig%d. The Static side carries the resident serialized objects; the Stream side carries the bulky geometry and textures (%sMD.model.stream, %s|%s.streamtex), which the engine logs about when one is oversized (Stream model resource too big to stream: %s, Stream texture resource too big to stream: %s). Each archive is self-contained: every offset in a TOC indexes its own file, so the Static and Stream archives are opened and read independently rather than one indexing into the other.

The global archives (Resident.ark, Frontend.ark, InGame*.ark) and the rider_%02d.ark animation archives are not present in this build — the data they would contain is loose on disk instead. References such as Resident|UI/Bootup/Bootup.gfx in the EBOOT show the <Archive>|<path> addressing the loader normally uses; in this build the part before the | selects a logical archive that falls through to the filesystem.

File layout

An archive is three regions, in order:

  1. Header (16 bytes) — points at the data region and counts the entries.
  2. Table of contents — count × 16-byte entries, sorted ascending by 32-bit name hash.
  3. Data region — packed asset blobs, addressed by absolute offset from each TOC entry.

Header

OffsetLengthTypeNameDescription
0x00x4uint32_tversionFormat/flags word. 0 in every sample examined.
0x40x4uint32_tdataStartByte offset where the data region begins. Equals 0x10 + count·0x10 exactly (header + TOC size). Verified on all samples.
0x80x4uint32_tentryCountNumber of TOC entries. e.g. 142 (nem_downtown.Static), 768 (Downtown.Static), 330 (nem_downtown.Stream).
0xC0x4uint32_tentrySizeSize of one TOC entry in bytes. Always 0x10 (16) — confirmed by the dataStart arithmetic above.

TOC entry (16 bytes)

OffsetLengthTypeNameDescription
0x00x4uint32_tsizeUncompressed / in-memory size of the member. Summing this field across all entries equals the data-region size to within one alignment pad (e.g. sum 0x2d921a vs payload region 0x2da710).
0x40x4uint32_tnameHash32-bit hash of the member path. Strictly increasing across the TOC in every sample — the table is kept sorted so the loader can binary-search by hash. Stable across levels: a given resource hashes to the same value in every archive it appears in (e.g. 0x000ECB73 and 0x01C17470 are the 2nd and 3rd entries in airport_test_03.Static, nem_downtown.Static and docks.Static; 141 of nem_downtown's 142 hashes are shared with both other levels). This rules out the early "cumulative offset" reading of this field — it is a content-/name-keyed hash.
0x80x4uint32_treserved0 in every entry. Likely a flags slot or the high dword of a 64-bit field that never exceeds 32 bits here.
0xC0x4uint32_toffsetAbsolute byte offset of the member's stored data. Always in [dataStart, fileSize); values verified to point at real blob starts.
The members are not stored in TOC (hash) order, and size is not the on-disk stored length. Sorting the entries by offset and checking offset + size against the next member's start shows it almost never lands on a boundary, and — decisively — the largest offset + size in nem_downtown.Static reaches 0x318FDE, well past the 0x2DB000-byte file. size is therefore the decompressed / in-memory size (an "allocate this much" hint), and the on-disk blobs are smaller because they are packed. The EBOOT confirms the codec: it links zlib (inflate 1.2.3 Copyright 1995-2005 Mark Adler), so members are deflate-compressed rather than stored raw. There is no per-entry stored length in the 16-byte TOC, so a faithful extractor must derive each blob's on-disk extent from the next-higher offset (the approach ark_extract.py takes) and treat size as the inflate target size.

Hex walk-through

nem_downtown.Static.ark (2,994,176 bytes = 0x2DB000). First 32 bytes = header + the first TOC entry: 

000000  00 00 00 00  00 00 08 f0  00 00 00 8e  00 00 00 10
        └ version=0  └ dataStart  └ count=142  └ entrySize=16
                       =0x8F0

000010  00 00 00 00  00 00 08 f0  00 00 00 00  00 18 47 2c
        └ size=0     └ hash       └ reserved   └ offset
                       =0x8F0                    =0x18472C

count·0x10 + 0x10 = 142·16 + 16 = 0x8F0 = dataStart — the data region begins exactly where the TOC ends. The first member has size = 0 (an empty/placeholder entry whose hash, 0x8F0, happens to be the smallest). The next-to-last entry illustrates the size field: 

Entry with hash 0x9AB5946B:  size=0x0E644  offset=0x1B26
At 0x1B26 in the file:
001b26  02 00 00 08  00 00 00 21  00 00 00 18  00 00 00 00
        └ Static-archive members begin with this 02 00 00 08 record header
          (a serialized-object / chunk tag, not part of the .ark container)

In the .Static.ark archives, every stored blob begins with the four-byte tag 02 00 00 08 followed by a small typed header. This was verified exhaustively by ark_extract.py across all six levels checked — every non-empty member matches: 244/245 in airport_test_03.Static, 141/142 in nem_downtown.Static, 887/888 in docks.Static, 767/768 in Downtown.Static, 851/852 in Graveyard.Static and 1311/1312 in Powerplant.Static. In every case the only miss is the single size = 0 placeholder entry, so the real recognition rate is 100 %. Immediately after the tag comes 00 00 00 NN (the item type — the value behind Unknown archive item type %d) and a length/header word; this is the serialized-object framing the loader hands to the per-format reader.

Stream members are different. Members of the .Stream.ark archives do not share this tag — ark_extract.py found 0 of the 02 00 00 08 tag across all six Stream archives (621, 330, 1278, 1237, 1089 and 1221 members respectively). Instead each Stream member opens with its own little sub-resource framing of the shape 00 00 00 00 | innerSize | 00 00 00 00 | … (e.g. 00000000 00035ACA 00000000 …, where 0x35ACA is smaller than the entry's size of 0x9E55C) — an inner table of the packed *.model.stream / *.streamtex resources the streaming system pulls per sector. The leading word is not a zlib header and the member does not deflate-decode at offset 0, so the compression boundary is inside this sub-framing rather than at the member boundary. The 02 00 00 08 serialized-object framing is therefore specific to the Static archives.

Extraction & verification

A standalone parser, _tools/ark_extract.py, reads the 16-byte header, walks the count × 16-byte TOC, and dumps each member. Because the TOC has no per-entry stored length, the extractor derives each blob's on-disk extent as the distance to the next-higher offset (trailing member runs to EOF) and writes the slice; size is kept only as the decompressed-size hint. It also sniffs the first bytes of every member against a table of known sub-format magics. 

# analyze only (header checks + magic histogram)
python _tools/ark_extract.py "/Sectors/.Static.ark"
# analyze + dump blobs to _tools/ark_out//
python _tools/ark_extract.py --dump --limit 60 "/Sectors/.Static.ark"
Test result (all six levels, 12 archives). For every archive the header arithmetic held (dataStart == 0x10 + count·0x10), the TOC was strictly sorted by nameHash, and the reserved word was 0 in every entry. Magic sniffing then confirmed the container reading:
  • Static — 100 % of non-empty members begin with 02 00 00 08: 244/245, 141/142, 887/888, 767/768, 851/852 and 1311/1312 across airport_test_03, nem_downtown, docks, Downtown, Graveyard and Powerplant (the lone miss in each is the size = 0 placeholder). Dumped .sobj blobs open exactly on the tag.
  • Stream — 0 members carry that tag in any of the six Stream archives; they open with the inner 00 00 00 00 | innerSize | … sub-resource framing instead.
That a single offset/length interpretation makes every Static member land on the same serialized-object tag is strong evidence the TOC layout (size · nameHash · reserved · offset) is correct.
  1. Hash the requested member path to a 32-bit key.
  2. Binary-search the sorted TOC on nameHash.
  3. Seek to offset, allocate size bytes, and stream the member out.
The EBOOT's archive diagnostics line up with sequential streaming rather than random reads: Warning: Skipping backwards in archive by %d bytes and Skipping forwards in archive by %d bytes are logged when reads jump around, and can't have more than one archive open at a time! shows ArkLoader keeps a single archive resident (mInstanceCount).

Open questions

  • Resolved: members are stored raw, not deflate-compressed — see Full extraction. The EBOOT's inflate 1.2.3 belongs to Scaleform GFx / libpng, not the .ark pipeline. What remains is whether a stored-length lives in the 02 00 00 08 payload header (the TOC size is rotated, so the authoritative length is the offset delta).
  • Resolved: the nameHash function is a reflected table-driven CRC (seed 0xDEAFD00D, poly 0xDB710641, step H=(H>>8)^T[(H^c)&0xFF], no final XOR — see Naming). What remains is purely enumeration: build the list of resource-ID strings (from the .sectorInfo sector ids + the <archiveTag>|<relpath> templates) and hash them to recover every member's name. The only friction is that the CRC table is built into runtime BSS, so the constants were pinned from the Rosetta corpus rather than read from the binary.

  • Decode the Stream sub-resource framing (the 00 00 00 00 | innerSize | … table) and its *.model.stream / *.streamtex payloads.

Full extraction

A second tool, _tools/ark_extract_full.py, produces a complete, named, decompressed dump of a level's .Static.ark + .Stream.ark pair into a tidy tree (level_extract/<Level>/{Static,Stream}/ plus a manifest.json). It folds together the two findings below.

"Decompression": members are stored raw

Correction to the zlib note above. The EBOOT does link inflate 1.2.3, but that zlib is used only by Scaleform GFx (compressed CFX/SWF) and libpng (PNG IDAT) — the only inflate call sites are GFx_InflateWrapper/GFxLoader and the Invalid zlib … in IDAT path. The .ark stream pipeline does not deflate-compress members. Evidence: a whole-file scan for 78 01 / 78 9C / 78 DA headers found hundreds of byte-pairs but none inflate to a sensible payload; raw-deflate (wbits=-15) at every member / sub-frame offset yields zero hits; and the sum of member on-disk lengths equals the payload region to within one alignment pad (ratio 0.99–1.00), i.e. there is no room for any compression gain. Members are stored 1:1.

The only transform needed is therefore stripping the 12-byte Stream sub-resource frame (00 00 00 00 | innerSize | 00 00 00 00 | …innerSize bytes…); Static serialized objects and unframed GPU textures are copied verbatim. The extractor still attempts a genuine zlib/raw-deflate inflate first (so it stays correct if a member ever is compressed), then falls back to frame-strip, then to raw store. On airport_test_03 the breakdown is 121 frame-strip · 743 raw · 0 inflated, confirming the raw-store finding. The authoritative member length is the offset delta (distance to the next distinct offset, EOF for the last) — the TOC size column is rotated by one slot in hash order and must not be trusted per entry.

Placeholder offsets are not boundaries. When computing the offset delta, only offsets of live members (TOC size > 0) bound a member. A size = 0 placeholder entry can sit at an offset that falls inside a live member's byte span; counting it as a boundary truncates that member. On airport_test_03.Static the lone placeholder (hash 0xF60, offset 0x2BC056) lands inside member 0x4EF7A1CF (offset 0x29E714): excluding it gives the correct length 0x22AAA rather than the truncated 0x1D942. Both the Python extractor and the Steward port filter the boundary set to size > 0 offsets for exactly this reason — with that fix the port's 864 written members are byte-for-byte identical to ark_extract_full.py.

Naming: the nameHash is a reflected table-driven CRC (solved)

The 32-bit nameHash is solved. It is a reflected (right-shifting) table-driven CRC over the resource-ID string (<archiveTag>|<relpath>), with the canonical byte loop: 

H = 0xDEAFD00D                       ; seed (initial register value)
for each byte c of resourceId:
    H = (H >> 8) ^ T[(H ^ c) & 0xFF]  ; reflected CRC step, table T (256 × u32)
nameHash = H                         ; NO final XOR / NO final inversion

The table T is the standard reflected-CRC table generated from polynomial 0xDB710641 (the same poly the earlier GF(2)-affine analysis had already isolated as the within-byte bit recurrence — that analysis was correct as far as it went; the missing piece was that the inter-byte propagation is a plain reflected-CRC (H>>8) ^ T[…] rather than an LFSR shift). Concretely T[1] = 0xDB710641 and each entry is the usual for(i=0;i<8;i++) crc = (crc&1) ? (crc>>1)^poly : crc>>1 reflected build. The seed 0xDEAFD00D (a deliberate "dead food" constant) and the absence of a final XOR were both pinned by fitting the Rosetta corpus: with this seed, poly and no final xor, the closed form reproduces every harvested hash→name pair exactly. This rules in a specific CRC and supersedes the older "GF(2)-affine but not closed-form" reading — the affineness was just the linearity every CRC has; the constants are now nailed down. We call the recovered constants the Rosetta naming key.

Offline blocker: the table lives in runtime BSS. The algorithm is solved, but computing a hash from a name still needs the 256-entry table T — and in this build T is not a static data section. It is generated at runtime into the EBOOT's BSS (zero -initialized at load, filled by the CRC-table init routine on first use), so it cannot be read straight out of the binary the way a const table could. A faithful offline implementation therefore either (a) regenerates T from poly 0xDB710641 as above — which is exactly what the formula does — or (b) recovers T from a memory dump of a running instance. Because the table generation is fully specified by the polynomial, option (a) is sufficient and the formula above is self-contained; the "blocker" is only that we could not confirm the constants against the BSS table directly and instead pinned them from the corpus.

Today the extractor still names a member from the hash→name dictionary — the TEXS Rosetta corpus (texs_rosetta_corpus.json, 309 pairs) plus any extra --names JSON. Those corpus hashes are UI/texture-pack hashes and do not overlap the level Sector member hashes, so for a Sector .ark pair members still fall back to <hash8>.<ext>. With the closed form recovered, the next step is mechanical: enumerate the candidate resource-ID strings (the .sectorInfo tables list this level's sector ids q000…q122 for airport_test_03, the inputs to CityStatic|q<n>_rig… / CityStream|q<n>_rig…), hash each with the formula above, and match against the TOC — no further cryptanalysis is required. See _tools/ark_namehash.py.

Format detection & output

Each member's type is detected by magic where one exists (02 00 00 08 serialized object → .sobj; Havok, FSB, GFX/CFX, SWF, DDS, PNG, BIK), else by structural shape: a framed Stream member → .geo (vertex/index geometry stream), and an unframed non-serialized Stream blob → .gputex (PS3 swizzled/DXT texture). The detected extension drives both the filename and the manifest's type histogram. 

python _tools/ark_extract_full.py \
    "<Level>/Sectors/<Level>.Static.ark" \
    "<Level>/Sectors/<Level>.Stream.ark" --level <Level>
Result — airport_test_03 (full run). 866 TOC members (245 Static + 621 Stream), 864 written (2 are size = 0 placeholders), 152 MB extracted — matching the Stream payload region exactly.
  • Type histogram: 365 model (244 Static .sobj + 121 Stream .geo) · 499 texture (.gputex) · 0 other.
  • Decompression: 121 frame-strip · 743 raw · 0 zlib/deflate.
  • Naming: 0 named / 864 hash-only (no corpus overlap for Sector hashes).
  • Spot-checks (byte-exact vs raw archive): a .sobj opens on 02 00 00 08; a .geo has its 12-byte frame stripped and decodes as a big-endian float vertex stream (first vertex ≈ 11.0, −346.2, 0.0); a .gputex of 0x55580 bytes is 16-byte (DXT-block) aligned. All three match the source bytes exactly, and 119/121 .geo satisfy payload == on_disk − 12 (the other 2 carry trailing alignment padding handled via the inner innerSize).
