Tonic (.tonic / .tonic2)

Tonic is Black Rock Studio's in-house granular engine-sound synthesizer — the system that builds each car's continuously-varying engine note in real time instead of crossfading pre-baked loops. The .tonic / .tonic2 files are the compiled grain tables: a pool of raw audio samples plus an index that a per-vehicle "state engine" walks according to RPM, rate-of-change and engine load. There is one pair per car under Deferred/Audio/Tonic/_OutputFiles/.

What Tonic is

Conventional racing games loop a handful of recorded engine RPM bands and crossfade between them. Tonic instead does granular synthesis: it stitches together tiny snippets ("grains") of recorded engine audio, choosing and pitch-shifting them on the fly. The eboot is unusually explicit about this. The debug-capture log emits, per playback:

Num Sweeps [%u]
 - Sweep [%u] type[%s] smallestGrain[%u] largestGrain[%u]
rpm [%.2f] rpm rate-of-change [%.2f] load [%.2f] targetNumSamples [%u]
   - GrainJob: fadeStart[%.2f] fadeEnd[%.2f] isReversed[%s]
      - SampleJob: gain [%.2f] numSamples[%u] numBytes[%u]
>> Grain [%u] <<
grainIndex [%d] startSample[%d] endSample[%d] size[%d] startValue[%.2f] endValue[%.2f]

So the runtime selects a Sweep (a set of grains of a given size range), picks a Grain by grainIndex, slices startSample…endSample out of the sample pool, optionally reverses it, applies a fade and gain, and mixes it. The driving inputs are rpm, rpm rate-of-change and load — i.e. it is a true physically-driven engine model, not a loop player. There is a dedicated JobTonic SPU job and a TonicHeap allocator.

Tonic feeds the PlayerEngine and CPUEngine mixer buses you can see in every .fev. The small Tonic/Tonic.fsb + Tonic.fev next to _OutputFiles/ hold the FMOD-side wrapper event (Tonic/Tonic/Engine); the heavy lifting is the custom grain data documented here.

State-engine source (the XML twin)

Each compiled .tonic is generated from a small bundle XML in _StateEngine/. Hypercar_03.xml reads:

<Root FileVersion="3.00">
  <Bundle>
    <File1 Filename="../_CarTypeSpecific/Engine_States.xml"/>
    <File2 Filename="../_CarTypeSpecific/Hypercar_States.xml"/>
    <File3 Filename="../_CarSpecific/Hypercar_03_Specific.xml"/>
  </Bundle>
</Root>

The model is layered: a global Engine_States.xml, a per-class file (Hypercar_States.xml, Muscle_States.xml, Supercar_States.xml, Truck_States.xml), and a per-car override. The states themselves are enumerated in the eboot as engine behaviours, each driving a different grain selection:

StateMeaning
SteadyConstant-throttle cruise.
OverRunThrottle lifted, engine braking.
GearChangeMid-shift transient.
UpGearEngage / UpGearExitUpshift engage / settle.
DownGearEngage / DownGearExitDownshift engage / settle.
BackfireExhaust pop on lift/downshift.
EnterBoost / ExitBoostTurbo/boost spool transitions.
ThumperLow-frequency idle/lope layer.

Compiled file header

The .tonic header is a fixed block of big-endian 32-bit words (no ASCII magic). Comparing Hypercar_03, Muscle_22 and Truck_09 separates the constant fields from the per-car ones:

OffsetLengthTypeNameDescription
0x000x4uint32_tversionConstant 7 — Tonic compiled-format version.
0x040x4uint32_tfield_04Constant 1 (likely stream/channel count).
0x080x4uint32_tnumSweepsConstant 4 across the cars sampled — number of grain "sweeps".
0x0C0x4uint32_tcountAPer-car: 0x12 (Hyper_03), 0x17 (Muscle_22), 0x13 (Truck_09) — a state/table count.
0x100x4uint32_tsampleRate44100 for .tonic; 21000 for .tonic2. This is the one field that distinguishes the two variants.
0x140x4uint32_tfield_14Zero in all samples.
0x180x4uint32_tnumGrainsPer-car: 0x34d=845, 0x314=788, 0x2c7=711 — total grain count (largest per-car field).
0x1C0x4uint32_tfield_1CConstant 2 (sample stride / 16-bit depth indicator).
0x200x4uint32_tfield_20Zero.
0x240x4uint32_tfield_24Constant 2.
0x280x4uint32_tsizeAPer-car size word (0x0d9c, 0x0785, 0x0a92).
0x2C0x4uint32_tsizeBPer-car size word (0x07e0, 0x0460, 0x0620). Grain-index / sample-pool sub-sizes.
The .tonic2 variant is byte-for-byte the same header except the sample-rate word (21000 vs 44100) and the per-car size/count words shrink roughly in half, and the file itself is about half the size (e.g. Hypercar_03: 2.76 MB .tonic vs 1.37 MB .tonic2). So .tonic2 is a lower-rate LOD of the same grain set — used for distant/opponent cars or memory-constrained mixes, matching the separate CPUEngine bus.

The grain payload

After the header the file is a dense block of packed audio. The hexdump shows long runs of high-nibble-biased bytes (aa bb cc cb ba …) interleaved with small chunk lengths (e.g. 00 2e 1e 90) — the signature of a delta/ADPCM-style nibble-packed sample pool sliced into grains, preceded by a grain index table. This payload is opaque without the Tonic decoder: the grain index (startSample/endSample/size per grain) and the raw pool are both inside, but their exact packing is not yet decoded.

What is still open

Confirmed: this is a custom (non-FMOD) big-endian granular format; header field roles (version, sweeps, sample rate, grain count, per-car sizes) are pinned by cross-car diffing; .tonic2 is a half-rate LOD; the system is RPM/load-driven granular synthesis per the eboot. Unconfirmed: the exact grain-index record layout, the sample-pool codec (looks ADPCM-like), the float "sweep parameters" at 0x30+, and how the four sweeps partition the grains. A decoder would need the Tonic class internals — only the debug-capture WAV/RAW path is documented in the eboot.

How we know

  • It's granular synthesis: eboot strings Num Sweeps, smallestGrain/largestGrain, GrainJob fadeStart/fadeEnd isReversed, grainIndex startSample endSample size, driven by rpm / rate-of-change / load.
  • It's per-vehicle & custom: files are named per car, live under Audio/Tonic/_OutputFiles/, compiled from _StateEngine/<car>.xml; the format is big-endian (unlike FMOD's little-endian FSB/FEV) and has its own TonicHeap / JobTonic.
  • Header roles: diffing three cars isolates the constant fields (7,1,4,44100,2,2) from per-car counts/sizes; the grain-count field is the largest per-car value as expected.
  • .tonic2 = LOD: identical header except the rate word (21000 vs 44100) and ~half file size with halved size words.
